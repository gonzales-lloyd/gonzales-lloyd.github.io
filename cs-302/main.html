Main

Three major concepts of OOP: Encapsulation, inheritance, and polymorphism.

Encapsulation is the idea that attributes and methods can be placed into one object or package, as in structs or classes. It also allows us to define boundaries and interfacing (protection via public, protected, and private keyword.)

The goal of encapsulation is a fully functional package. A programmer doesn't need to know how it does what it does, they just need to know what it does. You know that if you put bread in a toaster, you get toast out. Mess with it, and you may get burnt or electrocuted instead.

In sum, encapsulation is exposing a solution without requiring the user to understand the problem domain. We hide implementation and choose which variables and methods the user can access. A class's code should be complete (i.e. even if you don't use a part of your class, you must implement the whole class)

Inheritance is a relationship between classes. In inheritance, the child/derived class has access to the parent's public and protected member functions and variables. You may use identical getters and setters in a child class, but you do need to create new constructors (if you need to add new behavior). Your goal is to make a class extensible, but not necessarily modifiable.

You can create new functions in the child class, but you can also use overrides in the child class or overloads (in either class) where applicable.

Polymorphism is the condition of occurring in different forms. The types (and examples are):

  • Subtype: runtime polymorphism (example: overriding)
  • Parametric: compile time (example: templates)
  • Ad-hoc: overloading
  • Coercion: casting
    (See slides when available.)

Compile-time (Templates)
Note that templates are all created at compile-time. Since C++ is a strongly-typed language that depends on knowing all of the types at compile-time, template classes are not "normal" - depending on your compiler, it may or may not remember templated classes created with like data types. It also means that templated classes don't exist anywhere until instantiated as necessary.

This also means you don't include templated classes in the makefile, because it doesn't make sense to compile a standalone object file when there aren't strictly defined data types. Also note that changing a templated class may not cause the necessary recompiles to happen; make sure to run make clean as necessary.

In this class, separate templates into a source and header file. In practice, templated classes typically have declarations at the top and implementation at the bottom in one .h file.

Ad-hoc (Overloading)
This is a principle where functions with the same name may behave differently for each type. Put simply, the addition operator needs to do different things if you're handling different data types, or classes where addition is expected to work in a specific way.

For example, you might need to override an operator to have it behave as expected. Most objects usually want their comparison operators to compare two class attributes/member variables, rather than the actual addresses themselves.

The friend keyword gives access to private and protected members of the class to a non-member function. It goes solely in the header file. It is used in the source file. You also exclude the scope resolution.

Overloading the extraction and insertion operators

You're calling the function on the left-hand side, which is the stream object. (std::cout) Even though you're overloading the insertion operator in your own class, you are actually overriding the stream object (recall that the left-hand side is implicit in overloaded operators). As a result, this is where you would use the friend keyword when overloading these opeartors.

Header:
friend ostream& operator<<(ostream&, Object)
Source:

ostream& operator<<(ostream& out, Object rhs){
	out << rhs.value << " " << rhs.value2;
	return out;
}

Observe that unlike our usual operators, we actually need both sides of the operator in the parameters, rather than just the right-hand side; again, this is because it's not a member function of std::cout (or whatever iostream).

Think about how this works carefully - when you have multiple <<, the evaluations are successively done from left to right. In the code std::cout << a << b << c;, a is inserted into the stream, which then yields its own stream, and so on.

  • (std::cout << a) << b << c;
  • (ostream) << b << c;
  • (ostream) << c;

Likewise, you can also overload the extraction operator.

Interfaces
...

Inheritance is primarily a tool to share implmentation across a set of classes. Is one object a subset of another object?

Interfaces are a mechanism to share behavior across a set of classes. They force behavior upon certain child classes. For example, if we set a standard that we expect all Objects to be movable, then all children of the Object class must be movable. They are a list of behaviors that an object must have.

Any class is abstract if it has at lesat one pure virtual function, which is done by assigning it to 0 in declaration.

A pure interface class which has exclusively pure virtual functions should not have member variables. An abstract base class may have attributes. It may not make to include variables, if there is no reason.

You may want to split the interface and abstract base class, i.e. the ABC inherits from the interface, if you want some child classes to be constrained by behavior and some by both behavior and attributes.

Abstract data types
...

In an abstract data type (ADT), the user of the type should not need to worry about how its values are actually stored, in the same way that a programmer can ignore how the compiler actually stores integers. All that matters is the operation, or the behavior of abstract data types.

Again, an ADT is a logical description, while a data structure is a language-specific implementation. For example, a "stack" and a "queue" are logical representations with specific computer science meanings. But their actual implementations in each language are going to differ. A Java queue is a data structure you can physically use in code; the concept of queing is more abstract.

Background vocabulary
...

Abstraction is omitting or hiding low level details with a simpler, higher-level idea.

Now, remember that ADTs are just a way to collect objects. A collection is an object that groups other objects together. A collection...

There is also representation independence. Here, using the type is independent of its representation, which is how it's actually implemented. No matter what implementation you use of an ADT (which is to say, the language-specific data structure), the behavior should be the same, but the implementation may be different. For example, a std::string and a C-style string are generally equivalent, but their implementations are quite different.

Now, there are both built-in types (primitives), each with their own operations. These operations can be divided into four categories:

  • Creators create new objects of the type from nothing - which is to say, nothing existed before, and now something exists. A creator may take an object as an argument, but not an object of the type being constructed.
  • Producers create new objects from old objects of the type. The concatenation function of a string in most languages is a producer. Another is substring() in available implementations.
  • Observers take objects of the abstract type and return objects of a different type, usually taking a given object and reporting back information on it. The length() method of most list implementations are observers.
  • Mutators change objects. List appending is (typically) a mutation of the list, as opposed to immutable objects.
    Each function should be restricted to one category. Something that generates a substring should not also report on the length of the string.

Bags
...

Moving forward, we're going to consider storing a collection of items in terms of the container's behaviors. One of the easiest way to carry things from one place to another is a bag. The behavior of a bag is that it's simply an unordered collection of (usually, any type of) objects. It isn't generally used in practice, but it is a very good place to start thinking about ADTs in general.

For example, we need to consider the quantity of items in the bag.
You might also want to consider some other behaviors, like:

  • The number of items getCurrentSize() -> Observer
  • Is it empty? isEmpty() -> Observer
  • Adding and removing items add() -> Mutator
  • Emptying the bag empty() -> Mutator
  • Getting the number of specific items getFrequencyOf() -> Observer
  • Checking for presence of a specific item contains() -> Observer
  • See each item in bag (as a vector) toVector() -> Producer or observer, approximately

So how do we implement an unordered collection of objects? In C++, we translate these behaviors using a templated interface class. The net result is a set of pure virtual functions, as well as an empty destructor. (See slide set 4, slide 19.)

But you might also note that some of the mutators actually return something, like the add() and remove() function return bool(). This can be used to get some information about the add or remove operation, which is best practice with respect to (part of) exception handling.

In addition, also consider the fact that observers should not do anything to the object, so they should have the const keyword added.

We've now defined the interface, which constrains the behavior of the bag. But we haven't decided how to physically store objects. We want something raw - vectors and lists have already had their behaviors defined. In this case, we have two approaches - pure arrays and linked lists.

A linked list is dynamically sized, but typically requires traversal of the collection to access its elements. In addition, the dynamic allocation of each node can be confusing, given the extensive use of pointers.

An array has a fixed size, which means it may be too small or too large at any given moment. It might also result in having to shuffle elements around. However, element access is significantly easier.

Note that because C++ is fairly low-level, you do need to approach this relatively directly. You will be given a templated interface, and now you must determine how to actually implement the storage method for the templated class.

Implementation using arrays
...

We start with an array of a fixed size (i.e., stack-allocated). Of course, there are some questions that we should ask ourselves:

  • What is the size of the array?
    • How does that affect the bag?
  • How many items are stored in it? (Do we need to know how many items are in the bag, and would that be useful?)
  • What is the state of the array?
    • What is the state of the bag?

We also include following attributes to our array:

  • The array itself
  • A default capacity (it won't compile without it).
  • The maximum number of items
  • The current number of items

When deciding the default capacity, you typically want to use an internal private static const int that results in a standardized capacity. You may also have an additional attribute called maxItems that can be changed later, but it's a good idea to enforce a standard at-intialization capacity.

Regarding the destructor: Since this is a stack-allocated array, the destructor can just be empty in the bag's header file.

So let's begin implementing the bag. We begin at the bag's constructor, which specifies the starting state of the bag. In this bag, we have three distinct attributes to the bag's class - the current number of items, the maximum number of items, and the array holding the items itself.

Therefore, in our constructor, we assume that the bag starts empty. The current number of items, then, is set equal to zero. The maximum number of items is also set equal to the size of the array, or the default capacity. Finally, given that the array has been intialized as part of the header file, there is no need to make the array in the constructor. (Note that as a result, it calls the default constructor for the elements as needed.)

Let's talk about our toVector() function. Here, we need to put all of the contents from the bag and stick it into a vector. The simplest way is to simply create a vector, allocate it, and call push_back on the vector with each of our elements.

As for getCurrentSize(), we just return itemCount. isEmpty() can be done by simply checking if itemCount==0.

What about adding elements to the array? First, we need to check if there's enough room. If there is, we simply assign the new thing at the next free spot in the array (the smallest unused index), where the index is simply equal to itemCount and then itemCount is incremented.

But what if there isn't enough room? One option would be to extend the array. But that isn't part of the specification, so we simply return false.

Regarding counting the number of times an element shows up in the array, we simply iterate over the array, check for equality, and increment if the element is equal to the element being searched. Then, return the counter.

To see if our bag contains a specific item, iterate over the elements - one option is to simply use the getFrequencyOf() function. Another is to simply iterate over the items and return true if it's found (slightly faster, since it's not traversing the whole list every time.)

Finally, there's the question of deleting an element. There are a few approaches.

  • Shuffle: Remove the item, then move down all of the items after it.
  • Swap: Find the item to remove, swap it with the last item, and decrement itemCount.

Both are valid approaches, because the behavior constraints say that the object collection is unordered. Swapping is more efficient and simpler to implement.

Clearing the bag is simply as setting itemCount equal to 0, which makes the bag effectively considered "empty" for the purposes of behavior.

Linked structures
...

Let's talk about linked structures in general. It's much like the trains on a car - each car is connected to the next, and there is a locomotive and a caboose (beginning and end). Each car holds cargo, and you can hook cars to other cars.

In linked structures, the individual cars are nodes. Each node contains one data object (as based on the template) and a pointer to the next node.

An important thing is that it's impossible to predict the location of the next node ahead of time. You physically cannot know where a given node is or where a new node will be located (at least, not without extra work).

Furthermore, you cannot directly iterate through linked structures using built-in C++ features.

To keep track of the entire linked structure, you store the "head" pointer which represents the start of the list. There is also a tali, which is the last node containing a null pointer.

Note: the implementation in class does not have the head and the node as one element. The head points to the first node, rather than the first node struct itself being the head. The head does not contain its own data, at least not in this case.

The reasoning for this is that the "head" always needs to be a separate, existant variable in the linked implementation. If the "head" were a node, then this would require that the node contain some data, which is certainly not true once the bag is initially created. Like the array implementation, the array is initially empty and its itemcount is 0; we represent the same empty initial state by doing something like pointing the head to null.

In other words, our class might look like this:

template<typename ItemType>
class LinkedBag: public BagInterface<ItemType>{
private:
	Node<ItemType>* head;
	int itemCount;
}
// Node header, somewhere else
template<typename ItemType>
class Node{
private:
	ItemType item;
	Node<ItemType>* next;
}

As it turns out, the node is an object. In this case, we'll add additional methods with getters and setters (as opposed to a strict struct).

Implementation using linked structure
...

The easy parts
...

The easiest parts are the getCurrentSize() and isEmpty() functions, which have identical logic to the array-based implementation. Simply use the itemCount attribute of the LinkedBag.

When creating the vector representation, the main difference between the linked implementation and the array is the use of a cursor. Rather than iterating using an inherent forloop that iterates over the elements of the array, we use a cursor that will keep iterating until a null pointer has been reached. (Of course, if you just used the LinkedBag's head attribute, and then you changed it, you'd lose knowledge of where previous nodes were forever.)

Note: In class, rather than using a while loop that keeps going until the cursor hits nullptr, a for-loop that increments on i that is restricted by both the cursor hitting nullptr and the size as dictated by itemCount. Both are fine.

Also, remember that dereferencing a nullptr is a segfault.

Adding items
...

Now, the time has come to add items to the linked structure. Our function prototype looks like bool add(ItemType&);.

In order, if we're appending to the end of the linked structure:

  • Create the new node, and get the address associated with the node.
  • From the head, find the tail. (Note - you could jump to the tail and then create the new node as well, but it's preferable to separate the cursor-related stuff from the node creation itself)
  • Replace the nullptr on the old tail.

Note that since order doesn't matter in a bag, you can also simply have the new node point to the old head (which was the old first node), then replace the attribute head pointer with the pointer to the new node. Thus, you can insert to the start of the linked structure, which is considerably better since traversing the entire linked structure is no longer necessary. (This is not the case with ordered structures, like a linked list.)

Note that there's a very special benefit to a linked structure - you're not going to run out of room (if done correctly, and if you have any reasonable amount of memory.)

Removing items
...

If you want to remove an item, you need to find it. That, in turn, means you'll need to traverse the list until you find it. We iterate using the cursor as we've done in previous functions.

Now, we get the next pointer stored in the current item, and set that as the next pointer in the previous item. The result is that the item that was after the deleted item is now pointed to by the item before the deleted item.

Finally, deallocate memory (and, under good practice, set the deleted pointer to null.)

Clearing the bag
...

It's necessary to free every single node in the list. Free the memory as needed, moving the cursor ahead to the next element, and then deleting to the previous element. This requires iteration.

Lists
...

Note that position in a list is one-indexed, not zero-indexed. Therefore, calling getItem(1) yields the first element in the list.

When implementing it with an array, one option is to simply ignore the first index in the internal array always, such that the first position actually corresponds to the first index in the array, which holds the first item (not the second, as one might expect).

Linked lists
...

Use the Node implementation and List interface provided in class, which are all copy-pastable from slide set 7. You may also use the linked bag implementation.

You probably want the following attributes in your linked list:

  • head pointer
  • item count (optional - makes certain actions easier, like determining preconditions for insertion and deletion)

When thinking about linked lists, we can ask - is position concrete, or abstract? Do we hard-code things about position, or do we keep position in mind while coding?

You may include private helper functions. A getNodeAt() function is allowed, but a user of the class should not know that the underlying implementation uses a node.'

clear() may simply use the remove() function repeatedly.

See slides for preferred naming conventions.

When building a really big templated project, start with just stub/function prototypes with the entire structure defined, returning default values. Then, start adding the functionality into each function.

Stacks
...

One major thing about a stack is that it's last-in, first-out. In other words, the defining feature and behavior of the stack ADT is that it's a collection of items where elements are added to the top, last-in first-out.

(In addition, all the things in a stack are the same data type, aren't necessarily unique, and has finite object count.)

But there's also the fact that items are referenced by the top of the stack. Only the top of the stack is accessible, and anything in the middle is not accessible without removing the items on top of it.

The methods:

  • isEmpty: self explanatory
  • push(item): add item to top of stack
  • pop: remove item from stack (and does not return item popped, in this case)
  • peek: return item from top of stack

Array-based method
...

You need to hold an array, its default capacity, and some numerical representation of the position of the "top" element. You may choose 0, which represents the first available index in a new array, or -1, which may be useful when demonstrating that the array is empty (since an index of -1 doesn't exist.) A value of -1 is used in the slides for clarity.

To push, simply check for capacity, increment the counter for the top position, and then set that top position to the new entry.

To pop, check if it is possible to pop, then decrement the top counter; a push will overwrite this element and a peek will look at the new top counter.

To peek, check if the top is accessible (i.e. the stack is not empty), and then use the top counter to access the relevant information.

Linked stack
...

Rmember that the feature and behavior of a stack is that we add things to the top and access things first-in, last-out.

But a notable part of a linked stack is that the "head" pointer is really the "top" of the stack. Compared to a list, we have no concept of position; we can't access an item at a specific position. (There is order in a loose sense, in that you put things in and they will stay the same order over time, but it's not functional.)

And compared to the array implementation, a linked stack keeps its "top" at the front of the linked chain; the array implementation keeps its top at the end of the array.

With respect to pushing, there's no need to consider edge cases; we're always adding to the first position. Simply use the "insert at first node" code from the LinkedList implementation.

For popping, check if there exists a top node. Then, simply remove the top node and replace the head pointer (which is really the top pointer). Unlike the remove function of a list, we are actually removing; we're not returning the item we popped.

For peeking, simply get the top pointer.

There also needs to be a destructor! Pop repeatedly until the top pointer is null.

Use in assignment 3
...

When your program takes in the input, two things have to happen in parsing the input in infix form:

  • deciding if a given character is an operator or an operand
  • the conversion of an operand to an actual numerical value, possibly via comparison or by ASCII subtraction
  • turning the infix string into a postfix string
  • turning the postfix string into an actual numerical output, as a double

Now, take the assumption that we take in an infix expression as the pure input of the program. Computers are not necessarily efficient in jumping between the various parentheses and orders of the expression (you would have to traverse the expression left to right and parse it many, many times potentailly) - as a result, one option is to use postfix expressions.

Infix: (4+2) (5+3)
Postfix: 4 2 + 3 +

When we put things into postfix, we take in the two operands and then place the operator after the two operands. So how do we actually parse and build a postfix expression?

Infix to postfix
...

Evaluating each character in the infix expression:

  • If it is an operator:
    • If it is a parenthesis:
      • An open parenthesis goes on the stack.
      • A closed parenthesis means we pop the operators off the stack and add them to the postfix expression until we find the open parenthesis.
    • If it is not a parenthesis:
      • If the precedence of the top operator is greater than or equal ot the current operator, pop the top operator and add it to the postfix expression.
      • Otherwise, push the operator onto the stack.
  • If it is an operand:
    • Add it to the postfix expression.

For example, take the expression (4+5) * (4+7).

  • The open parenthesis goes into the stack.
  • The 4 goes into the postfix.
  • The plus sign goes into the operator stack. Since the precedence of the addition is greater than that of the parenthesis (which is a special case), it goes into the stack rather than the postfix expression.
  • The closed parenthesis signifies that it is necessary to pop operators off the stack and add them to the postfix expression until the open parenthesis is popped off.
  • The multiplication goes into the operator stack.
  • The open parenthesis goes into the operator stack.
  • The 4 goes into postfix.
  • The addition is not higher precedence than the open parenthesis, which is on top (it is lower precedence than the multiplication, but that isn't known.)
  • The 7 goes into postfix.
  • The closed parenthesis pops off all operators.
  • The multiplication is popped off.

Then, the resulting postfix expression is 4 5 + 4 7 + *.

Postfix to numerical result
...

To evaluate the postfix expression, we use what's known as an operand stack. The operand stack is like the operator stack, but instead we put numbers in.

  • If it is an operator, pop two operands from the stack.
  • Evaluate the expression.
  • Push the resulting value onto the stack.

After the postfix expression is fully evaluated, there should be one value at the top of the stack that can be peeked for the resulting value.

For example, with the postfix expression from the previous expression
4 5 + 4 7 + *:

  • Push 4 onto the stack.
  • Push 5 onto the stack.
  • Upon reading the +, pop 5 (a) and 4 (b) and evaluate b+a, or 4+5. Push 9 onto the stack.
  • Push 4 onto the stack.
  • Push 7 onto the stack.
  • Read the +. Pop 7 and 4 off, and evaluate 4+7 = 11.
  • Read the multiplication. Pop 9 and 11 off, which results in the answer, 99.

Another example
...

Convert 50+10/2+7%3.

  • 50 goes to the postfix expression: 50
  • + is an operator thrown onto the stack. [+]
  • 10 goes to the postfix expression: 50 10
  • / has a higher precedence than addition, so push it onto the stack. [+ /]
  • 2 goes to the postfix expression: 50 10 2
  • + has a lower precedence than the current operator, so the top operator is popped off and added to postfix, while the current operator is pushed onto the stack. 50 10 2 / [+ +]
  • 7 goes to the postfix expression: 50 10 2 / 7
  • % has a higher precedence than addition, which is currently on the stack. Push it onto the stack. [% + +]
  • Push 3: 50 10 2 / 7 3
  • The infix is complete; peek and pop repeatedly until the remainder of the stack is gone. 50 10 2 / 7 3 % + +.

for(int i = 0; i<n; i++){
	std::cout << "a" << std::endl;

  • Compare i
  • Print to screen

But since we're referring to families of functions, we take only the term with the largest growth rate. Therefore, we eliminate 2 and n.

We also eliminate constant scalars/coefficients, so the big O notation simply becomes O(n).

Sorting - basic
...

Internal sorts is a sort where the collection of data fits in memory - that is, in-place sorting.

For external sorts, a collection of data does not all fit in memory, and requires an additional data structure for storage.

Bubble sort
...

The largest value in the collection moves to the end in each iteration. Each iteration starts at the beginning and ends right before the item placed in the previous iteration. The largest item "bubbles" up to the top in each iteration.

This is achieved by comparing pairs of elements, swapping elements if it should keep going towards the "top", or end of the collection.

Selection sort
...

Conceptually, think of there being two sections to the list - the unsorted section and the sorted section.

Find the smallest item in the "unsorted" section. Swap it with the leftmost item in the unsorted section. Then, it becomes the largest item in the sorted section. Repeat, starting from the start of the unsorted section each time.

Insertion sort
...

Similar to selection sort, except that instead of picking which element to put in the sorted section, we grab the first item from the unsorted section and put it where it belongs in the sorted section. This means that shuffling is necessary, as we iterate backwards over the "sorted" section to find the correct spot to insert the item.

Recursion
...

When you loop, you have three parts - the conditional expression, iterations, and some way to update the conditional expression (to avoid infinite loops).

You can solve all recursive problems with loops, but not all loop-based problems can be solved with recursion (generally).

In recursion, we also have approximately three parts:

  • a base case, which detemrines when we stop creating smaller problems
  • way to make the problem smaller, to reach the base case
  • a recursive function call

Functions are recursive when they call themselves.

A classical example is the evaluation of a factorial.

You could also evaluate exponents. In each recursive call, we would reduce the exponent called, and multiply the result of that call by the base:


int exponent(int base, int pow){
	if(pow == 0){
		return 1;
	}

	return base*exponent(base, pow-1);
}

To count the digits in a given nonzero number:

int digits(int number){
	if(number < 10){
		return 1;
	}
	return 1+digits(number/10);
}

(another valid solution is to check for number==0, and return 0 if that's the case.)

Trees
...

Previously, we've stored data in a linear fashion. However, a tree stores data in a hierarchial order. The remainder of data structures in this course are nonlinear.

A notable thing about this structure is that items in trees can lead to more than one thing. Before, in a linked list, you had a (singly) linked set of nodes. Now, in trees, you have many links to other nodes.

  • Trees begin with a root. This is the single starting point, where there is nothing above this node (i.e. it has no parents).
  • Each point in a tree is a node, which holds data. Each node should have exactly one parent and zero to many children.
  • The connections between nodes are called edges.
  • If a node has no children, it is a leaf.

We can also talk about relationships - any child or subchild of a node is a descendant of that node, any parent is an ancestor of that node.

A subtree of a tree T is a tree S consisting of a node in T and all of its descendants in T. That isolated node becomes the new root.

Now, a path is a collection of edges. So path length is the number of edges from node to node (so the path connecting three nodes linearly has a path length of 2). (Note that paths do not necessarily need to work from the root.)

You can also talk about the depth of a node, which is the longest path length measured from the root to the node. This is from the top downwards.

The level of a node is the longest path length from a leaf to the node in question. This works from the very bottom upwards (that is, from the lowest-level or bottommost leaf to that node).

The height of a tree is the number of nodes on the longest path from the root to the lowest-level leaf. Observe that this is equal to the number of levels (but it is equal to the maximum level value plus one), or the longest possible path length from the root to a leaf plus one.

Now, there are a few additional qualities of trees that may exist.

  • A full tree is where every node other than the leaves have n children. These would apply to binary, trinary, quarternary trees, where n = 2, 3, 4 respectively.
  • A complete tree is a tree where every level, except possibly the last, is completely filled and all nodes are as far left as possible. Full trees are complete.

Binary trees
...

A binary tree has between zero to two children for each node. We can extend this to the idea of a binary search tree, which has a very specific construction.

In a binary search tree, the left subtree of a node contains only nodes with keys lesser than the node's key. The right subtree of a node contains only nodes with keys greater than the node's key.

The left and right subtree of any given node must each also be a binary search tree.

There are three ways to traverse a binary search tree:

In preorder, you visit a node (usually, act on the data), then traverse the left subtree and then the right subtree. Traversing the left subtree kicks off another preorder traversal (so you visit the left subnode, visit its left subnode, and then visit its right subnode).

In inorder, you go to the left subtree, visit the node, and then traverse the right subtree. That is to say, you go as far left and as far down as you can, print its data, and then go back up to its parent, "visit" it, and then go to its right subtree if one is present.

In postorder, you go to the left subtree, go to the right subtree, and then visit the node. In this case, you again go as far down left as you can, and then go to that node's parent's right subtree, and then go as far down left as you can again, and so on.

It turns out that if you've set up an alphabetical binary search tree and you want to print everything out in alphabetical order, you would use inorder.

Implementation
...

It is possible to use both an array and a linked-based implementation. A linked-based implementation probably looks exactly the way you think it does; you store a root node, with each node storing its data, a pointer to its left child, and a pointer to its right child.

But it's also possible to use an array-based implementation, where the pointers to children instead are index locations.

In general, for these nonlinear data structures, we'll use linked implementations.

Binary search trees
...

A binary tree simply has the condition of having (up to) two children on each node; a binary search tree has the condition where the nodes in the left subtree are less than the value of the node itself, and the nodes in the right subtree are greater than the value of that same node.

Although it is true that binary search trees are a binary tree, and therefore a binary search tree should inherit from a binary tree, it is not required to implement both binary trees and binary search trees for this assignment. You must still inherit from the Tree interface, if you opt not to use a binary tree.

A binary search tree only has one attribute, the rootPtr.

The user should not need to know about the variety of helper functions we use underneath to work with recursion and the pointers that form the tree. Those functions are protected, so child classes can still use them but class users cannot see them.

Adding things to a binary search tree
...

When inserting an item, you make a decision at each subtree, ultimately "placing" or adding the item once you reach a subtree where the next subtree would be null. This means that when a value is greater than or less than to a given node, you recurse; when a value needs to compare to null, do nothing.

A naive way:

bool placeNode(LinkedBTreeNode<ItemType>*rootPtr, LinkedBTreeNode<ItemType>* newNodePtr){
	//assert(rootPtr != nullptr && newNodePtr != nullptr);

	if(newNodePtr->getValue() > rootPtr->getValue()){
		LinkedBTreeNode<ItemType>* rightPtr = rootPtr->getRightPtr();
		if(rightPtr == nullptr){
			rootPtr->setRightPtr(newNodePtr);
			return true;
		}else{
			placeNode(rootPtr->getRightPtr(), newNodePtr);
		}
	}else if(newNodePtr->getValue() < rootPtr->getValue()){
		LinkedBTreeNode<ItemType>* leftPtr = rootPtr->getLeftPtr();
		if(leftPtr == nullptr){
			rootPtr->setLeftPtr(newNodePtr);
			return true;
		}else{
			placeNode(rootPtr->getLeftPtr(), newNodePtr);
		}
	}else{
		return false; // ignore duplicates
	}
}

Another way, where we return the newNodePtr to the last calling function, and where the check is done at a possibly null pointer rather than at the leaf with node pointers:

template<typename ItemType>
LinkedBTreeNode<ItemType>* placeNode(LinkedBTreeNode<ItemType>*curPtr, LinkedBTreeNode<ItemType>* newPtr){
	if(curPtr == nullPtr){
		return newPtr;
	}else if(curPtr->value() < newPtr->value()){
		tempPtr = placeNode(curPtr->rightChild, newPtr);
		curPtr->setRightChild(newPtr);
	}else if(curPtr->value() > newPtr->value()){
		tempPtr = placeNode(curPtr->leftChild, newPtr);
		curPtr->setLeftChild(newPtr);
	}
	return curPtr;
}

This works by returning the new pointer to be set as a leaf's child on the base case, and then simply returning the existing child to any non-base case calls.

Deleting
...

You have four cases while traversing the tree to find the target value to delete:

  • The node is not in the tree. (current node is nullptr)
  • The current node is the target.
  • The current node is greater than the target.
  • The current node is less than the target.

It's easy to delete leafs. But what if you have to delete a subtree with children? Once you've found the target value to delete, you also need to take some other things into consideration:

  • The node is a leaf.
  • The node has only one child.
  • The node has two children.

It turns out that if you only have one child, you can simply take out that node and let its child be the new left/right node of the old node's parent.

But if the node has two children, you need to find its inorder successor. This also has two cases:

  • The node has a left child.
  • The node has no left child.

If the node has a left child, then you get the leftmost node, set that as the new left child pointer, and then delete the node.

If that node has no left child, then you simply remove this node ...? resolve

Graphs
...

Graphs represent relationships and store vertices, where relationships are represented by edges. It's identical to the discrete mathematics definition, but our implementation will obviously have some extra things attached to it depending on what we need.

You can have both connected and disconnected graphs, where nodes can be isolated or subgraphs can be disjoint. Complete graphs are where every vertex in the graph is connected to all other vertices.

A weighted graph has a "weight" or "cost" for each edge.

There are directed and undirected graphs. Directed edges have a start and finish, represented as an ordered pair (tuple).

Paths and cycles have identical definitions to their discrete mathematics form.

For the purposes of this class, we'll make the assumption that all graphs are undirected and weighted.

When adding elements, nodes do not already have to exist; they may be created adn then added, or simply create an edge between them.

Brute forcers for TSP often have an underlying data structure containing a set of named points, and then a permutation generator to get all permutations of all named points. Then, the "cost" of each permutation is calculated, and then stored and compared. (For the purposes of TSP, we are trying to find a path in a complete graph.)

Graph storage is variable. One way is through the adjacency matrix, which allows us to translate vertex adjacencies into a graph.

You may also want to use a 1D adjacency list, which is effectively a 1D array of pointers where each element points ot a list of vertices connected to each (index) of a named point. In other words, it's a collection of lists (and is effectively a 2D array, but with variable sizes on column- or row-major.) Said list can be linked, since the variable size makes linked implementations useful and linear traversal is necessary.

Of course, an adjacency matrix makes it very easy to check if there exists an edge from one vertex to another. You can also check for which vertices are adjacent to given vertices linearly, but an adjacency list is more efficient for that, where the only elements in an adjacency list are those that are actually adjacent to given vertices.

Also, a list often requires less space than an adjacency matrix.

A depth first search goes as far as possible from a vertex before backing up; that is, the last visited nodes are the first nodes explored. You might use a stack to do this, storing all the ones you've visited previously, and then popping things off (and "exploring" it) once you reach a pendant node.

A breadth first search visits all vertices adjacent to a vertex before going forward; that is, the first nodes visisted are the first nodes explored. You can use a queue for this, storing the ones that come first and also exploring those first.

You can also do this recursively.

It is important that you either restrict the traversals (like pulling available nodes to visit from a bag) or store which ones you've already seen while traversing.

So when you do searches and you have two edges, what do you do? You can choose based on the adjcacency list for that element and store that onto the stack, skipping elements in the adjacency list that have already been seen.

For depth first search, you might end up in a spiral that terminates in the center, but hasn't actually explored all other points. Then, as a result, you start popping things off the stack , until you end up at a point where you go to nodes you haven't seen yet.

For breadth first, you queue up all the elements in your starting node's adjacency list and then dequeue the starting node off. Since it's been seen, you can add it to your path.

Then, dequeue the second element in your starting node's adjacency list, adding it to the path once it's been seen and finishing off its own adjacency list.

Assume that for TSP, all nodes are accessible from any other node. Even though technically (for our specific version of the problem) the cities may not all be connected to each other, we can always just calculate the path as "infinity" or perhaps throw an exception if we find an invalid set of cities for a potential path.

You may want to make a path class since it makes logical sense to group path properties together - is it valid? what's it's length?

Dictionaries
...

In a dictionary, your goal is to map some key into a specific value. You can do this a variety of ways - binary search trees, unordered arrays, etc; but the most common way is through hash maps.

Here, we shove our key through a hashing function, yielding a number. Take the modulo of that using the size of the underlying container (if applicable), and you end up getting a specific position in theoretically O(1) time. This works for lookup, insertion, and removal.

But what if you end up with a hash collision after the modulo step? That's where we can do a few things:

  • Probing is where you simply start walking down past the original index and look for either a free space (in insertion) or the same element (in lookup/removal). Linear and quadratic are often used.
  • Chaining is where you use a linked list at each index, and add collisions to that chain.

Heaps
...

Heaps are a special type of complete binary tree, such that:

  • There exists a parent-child relationship where each parent has two children
  • It is complete; that is, a complete tree is a tree where every level, except possibly the last, is completely filled and all nodes are as far left as possible.
  • Any given parent (including the root) contains a value that is >= or (<=) its children. Thus, the root contains the largest (or smallest) value of the entire heap; the leaves contain the smallest values.

A maxheap is where the root contains the largest value; a minheap is where the root contains the smallest value.

Here, we will want to use an underlying array to store it, because the completeness of the tree is easier to fulfill using array arithmetic than to rebuild a linked version of the tree each time.

In contrast, because a binary search tree has an arbitrary shape (it's not complete), a node makes more sense, since you would need additional bookkeeping or wasted array space to store the BST in most cases.

Since it's complete, we know that each level is filled left-to-right. So in an array:

  • index 0 is the root
  • index 1 is the left child, 2 is the right child
  • 3 and 4 are the children of the root's left child, 5 and 6 are the children of the root's right child
  • 7-10 and 11-14 describe the children of those children
  • and so on

So it holds that if a left child exists, we would get [2*i+1], where i is the index of a given element. To get a right child, we get [2*i+2]. For a parent, we would get [(i-1)/2]. You can see that these work because all parents for a given level must exist at specific places; all left children exist on odd nodes, and all right children exist on even nodes (excluding the root).

Adding things
...

We have two considerations in mind for heap adds:

  • we need to preserve completeness
  • we need to preserve the "ordering" of items (that is, the relationship that all parents are bigger or smaller than other things.)

Completeness is easy; just figure out the next available array element.

Let's pretend this is a maxheap. What if the new number is greater than the immediate parent of the free array space? What if the new number is greater than the root itself? There are generally three addition cases:

  • the new item is greater in value than the immediate parent, but less in value than the next parent
  • the new item is greater in value than the root or some higher-level parent
  • the new item is less than the parent (so direct insertion is valid)

For the first case, it's as simple as swapping the parent and the new child.

For the second case, it's actually the same as the first case, but with a "recursive" aspect; we swap parents and the new element until the heap is in the desired form, either because the new element has become the root or because the new element is no longer greater than its parent and needs to be swapped.

So, in general:

  • insert at the end
  • while the parent is less than the new value
    • figure out the index of the parent relative to the new value (implicit in the evaluation of the while loop)
    • swap the parent and the new value
    • note the current index of the new value
  • return

There are, of course, some additional details here; you might use a boolean variable instead for the while loop condition, checking that and updating it inside the loop instead of directly evaluating the parent.

Removal
...

You can't remove elements at random, since you'll end up with broken heaps if the element is an internal node. You also still have to preserve completeness.

If you add elements from left to right and then fix the tree, removal is very similar; we should remove them right to left so that our condition for completeness is trivial to fulfill, and then work from there.

Since we know that our rightmost leaf is simply located at the last position in the array, that's the element we'll want to remove. But unless we're deleting that last element specifically, we still need to move that value somewhere, like if we're deleting the root.

So we'll swap the item being deleted with the last element, so that the last element is preserved. Then, we "delete" or ignore the last element.

So now we might have the smallest element as the root, or at least we've got a parent that is smaller than its child. To fix this, we "trickle down" the old smallest element, such that we swap the element with the larger of its two children until the tree is fixed. We stop when both children are smaller (or the only child, or it becomes a leaf.)

We typically implement this "tree-fixing" function as a separate helper function, since fixing it recursively may make more sense.

Heap sort
...

Heap sort is a sort in which the root is always the largest item, so that we can pluck the largest value off the heap and then rebuild the heap.

So for our heapified array, we have the largest value at index 0; we place that into a "sorted" region (either an external array or the same underlying array), move everything down, and then reheapify the whole array.

This might feel much like a bubble sort, which operates in time. But heap sort is time, because reheaping only happens for each half of the heap's subtrees.

Compared to the other sorts, this is nonrecursive and is in-place, unlike the external sorts.

Priority queues
...

You can implement a queue using a heap, where you simply take the top thing from the maxheap and remove it. New items are added such that priority is maintained, while also giving you the "back" and "front" of the queue using simple array accesses.

Other uses
...

Heaps are also used in things like Dijkstra's algorithm and other scenarios.
