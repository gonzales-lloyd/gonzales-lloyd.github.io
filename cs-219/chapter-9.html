We'll ignore some parts of an accurate model to make it less difficult to understand. We'll call it the single cycle model, since we assume that each step runs in one clock cycle.

Building blocks of a CPU
...

Remember from earlier chapters that we looked at the register file and the ALU, or arithmetic logic unit.

In reality, the 16 registers of a CPU are located in a register file. This is a 16x32 register file, which is to say that it has 16 registers that are all 32 bits wide each. The register file has one write input and two read outputs. The CPU can write to one register and read from two registers on every clock cycle, and it looks like this:

And as for the ALU, it's a 32-bit component that outputs the result of its two inputs under some function, determined by the ALU_OP control input. We'll treat the values for ALU_OP as an enumerated data type (which means that each name for ALU_OP in the table below corresponds to some constant integer, which are what the CPU actually uses.)

How a CPU runs instructions
...

Basic R-type instructions
...

So how do we actually run instructions? We start by connecting the register file to the ALU. This is a simple circuit that can actually run part of the CPU's instruction set. Then, all we have to do is to set the control signals to the correct values to run the particular instruction we want, which would be ALU_OP and RdA_addr, RdB_addr, Wr, and Wr_addr on the register file.

We pick any two registers, do an ALu operation on their data, and write the result back to a register. As a result, we can use this circuit to run R-type instructions from the instruction set, where the ALU_OP is derived from the operand, Wr is 1 (since it's writing), and Wr_addr to the target register. RdA_addr and RdB_addr are set to the target registers.

The compare (CMP) instruction
...

Now, we know that CMP is special in that it doesn't store the result anywhere. It's probably obvious that ALU_OP should be SUB and RdA_addr and RdB_addr should just be the input registers of an R-type CMP, but what about Wr and Wr_addr?

Since we don't want to store the answer, we let Wr be 0. Then, as a result, Wr_addr can be anything at all, since it's only used to set the destination address when Wr is 1 (that is, the target address only matters when we're writing.)

For digital outputs, "XX" means that we don't know what the output is, usually because it hasn't been initialized. But for digital inputs, "XX" means that we don't care what the input is, and so it allows us to say that whenever Wr = 0, Wr_addr = XX.

As was the case in CPE 201, knowing when an input is XX is helpful; this often allows us to generate more efficient circuits that use less power.

Flags?
...

For this model, we assume that the ALU takes care of the flags.

I-type instructions
...

Remember that when we give the CPU an immediate operand, the operand must fit within 8 bits after an number of rotations.

Before dealing with I-type instructions, let's talk about where CPU instructions come from. Instructions are stored in memory, and in Von Neumann architectures, instructions and data are in a shared memory. However, for our model, we'll assume that instructions and data are stored in two separate memories.

Remember that one of the registers in the CPU is the program counter, which holds the address of the next instruction to be executed. As a result, we ask the memory to give us the data at that address, and then increment the program counter by 4. (The only exception is when branching, where the PC is set to the target address of the branch instead of incrementing.)

Where does the requested data at that memory address go? We send it to an instruction register, which holds the machine language instruction that we are running. The instruction register causes an "immediate" signal to be output to the control logic, which indicates that the immediate operand should be selected instead of the register operands (used in R-types).

The control logic is a combinational logic function that generates the control signals from the machine language instruction, which then is sent over to the ALU. Here's what everything looks like as we combine things, where the control signals from the control logic are sent to the register file, ALU, and a newly-added mux to perform some calculation:

So for the circuit above, calculating ORR r5, r5 #0xF would cause the following control signals to occur:

  • Clearly, ALU_OP should be OR.
  • Imm_nReg is set to be HIGH, which causes the mux to select the immediate operand rather than data stored in the second register.
  • RdA_addr is the first operand, or register 5.
  • Since the multiplexer is selecting the immediate operand (labeled Imm in the figure above), it doesn't matter what RdB_addr is.
  • Wr = 1 since we're trying to write to a destination register.
  • The destination register is r5, so Wr_addr is 5.
  • PC increments by four, to the start of the next instruction.

Loading and storing
...

How do we implement LDR and STR? Since these instructions access data memory (and not instruction memory), we need to add data memory access to our CPU. A store instruction will write a register value to data memory, while a load instruction writes a register value to a register from data memory.

Again, we assume data memory is separate from instruction memory.

The data memory is located after the ALU, so our complete model will look like this:

As you can see, this adds some additional control signals for any given instruction, on top of the data memory and another mux.

Data memory takes in some address and data, where DM_RD and DM_WR determine whether data is written or read. When writing (and so DM_WR is HIGH), this allows data memory to be written. When reading, DOUT is equal to the contents of the specified address as dictated by the ALU (which is simply the address contained in register input A).

MEM_nALU dictates whether the data memory or the result of the ALU should be written back to the registers. Assumig that Wr is HIGH, then this is how the contents of memory are loaded into a register while also providing support for the storing of basic operations directly into a destination register.

Control signals
...

In short, here are the control signals:

So for LDR r2, [r3], here's the value of all the control signals:

  • ALU_OP is "A", which simply means to return the value stored in register A.
  • It doesn't matter what Imm_nReg is, since the B input on the ALU isn't used - there's no third operand.
  • RdA_addr is 3, since we need the contents of register 3 to determine what address we need to load from data memory.
  • Again, since the B input isn't used, it doesn't matter what RdB_addr is.
  • Wr is 1, since we are trying to write to a destination register (r2).
  • The destination register is 2, so Wr_addr is 2.
  • DM_RD = 1, since we want to read from data memory.
  • DM_WR = 0, since we're not writing to memory.
  • MEM_nALU = 1, since we do want the results of the data memory read to go into the register file.
  • The program counter is incremented by 4.

You can find more details in the slides for chapter 9.

Shifts and rotates
...

For simplicity, we'll assume that we only shift 1 bit at a time, since this is the most common case. That said, ARM does support anywhere from 1 to 31 bits per operation.

Recall from the table containing the ALU operations that shifts and rotates are done only on the ALU's A input. The B input is completely ignored for these operations.

Here's what an LSL r3, r3 looks like:

  • ALU_OP is LSL.
  • It doesn't matter what Imm_nReg is, since the output of that multiplexer is hooked to ALU input B, which is ignored.
  • RdA_addr is r3, since the contents of r3 are used in the ALU for the shift.
  • RdB_addr doesn't matter, as it's not used anywhere, and isn't used for data memory.
  • Wr = 1 since we're trying to write back to the register file - specifically, register 3. As a result, Wr_addr = 3 as well.
  • DM_RD and DM_WR are both equal to 0, since we're not doing anything with data memory. You may wonder why these aren't XX, or don't cares - this is because we might accidentally write to data memory, or we might accidentally read from data memory at some arbitrary address (which gets output to DOUT).
  • MEM_nALU is 0, so we read from the result of the ALU, which has shifted the contents of r3 by some amount.
  • PC gets incremented by 4.

Branches
...

Branching is actually straightforward - all we have to do is let the program counter be equal to the address specified by the branch instruction.

You may have noticed that the program counter is just being incremented arbitrarily by nothing, even though we're doing other things to the register file - the control logic does this implicitly in the model.

The same holds for branches, where the PC's value is just handled outside of the register file logic that we have at the bottom of this figure.

Data paths and control
...

With this, we can better define two things we talked about in chapter 4, regarding the CPU model.

A CPU is made up of two things:

  • the data path, which is the components and connects that form the path where data flows inside the CPU
  • The control, which is everything else. This is the logic that tells everything in the data path what to do and when to do it.
