Recall that the first 13 registers, R0 to R12, are all general-purpose registers. We've talked about R15, which is the program counter, which holds the location of the next instruction to be processed. Now, we'll talk about R13 and R14, the stack pointer and link register, and how those work.

The stack and the stack pointer, r13
...

For many purposes, CPUs have an area in memory called the stack. The stack is a set of memory locations that are only accessed using the stack pointer, R13, and can only be accessed in last-in, first-out order. This is accessed using load and store instructions, given that the stack (and the heap) are both located in memory.

When the CPU starts, it initializes the stack pointer to point to the top of the stack. The stack pointer always points to the next available memory location in the stack.

The stack often grows downwards; that is to say, lower addresses are at the bottom of memory, and the stack occupies the highest address values first. So if the stack ends at 0x0400, then the stack pointer would point to 0x03FC, which is four bytes down.

As with the data structure, storing a register on the stack is commonlly called pushing it onto the stack. After the value from a register has been stored on the stack, we then decrement the stack pointer to point to the next open location.

Taking data off the stack is commonly called popping, and is popped in the reverse order that it was pushed. Unlike the data structure definition used in CS 302, popping implies a peek and a pop - that is, the data is removed and is given back to you. This would appear in an instruction like LDR r7, [r13, #4]!, which takes the the thing currently pointed to by (r13+4) and also increments r13 by 4, which means that the popped data would be overwritten on the next store instruction.

Note that not all CPUs have their stack growing downwards.

If the data pushed to the stack exceeds the space allocated for the stack, then a stack overflow results. This often happens when you make too many nested function calls, which successively grows the size of the stack until it may run out of space. A system designer must ensure that the stack is large enough to handle its uses, although a poorly designed function can still cause stack overflow.

Functions
...

Functions, also called subroutines or procedures, are blocks of code that are called from a larger block of code. One definition is that a function "is a sequence of instructions (code) that is packaged into a unit that can be reused."

Functions can have parameters as inputs, return a result, and have local variables that are not within the scope of the reminder of the program.

Functions increase code density, saving code space if a function's code is called more than once. The caller is the code that calls the subroutine; the callee is the subroutine itself.

The caller must provide parameters, transfer control to the procedure, and branch to that procedure while also providing a "link" back to the main program to continue the program flow.

The callee must acquire or create resources needed to perform the function, such as local variables and registers. It must also perform its function, and if applicable, place results in a place where the caller can find them. It must then return control back to the caller through the supplied link.

The link register, r14
...

There are a few questions we have ask about functions:

  • How does the caller jump to the callee?
  • How does the callee jump back to the right place in the caller function?
  • How are arguments passed into the callee?
  • How are local variables in the callee stored?
  • How do we return values from the callee to the caller, and how does the caller access that value?

We'll address the first two questions.

One option to jump to the callee is to have a branch that points to the label or destination address from the caller. Then, at the end of the callee, we also have a branch that points to the instruction immediately after the branch in the caller. The problem with this solution is that there could be multiple places where the callee is called; then, a hard-coded branch instruction doesn't work.

So instead, the caller can store the return address in a register. r14, the link register, can be used to store the caller's return address. At the end of the callee, r14 is moved into r15, and so execution continues from the instruction after the branch.

Interestingly, ARM has a "branch and link" instruction that allows you to combine the assignment of the return address to r14 and also branch off to the subroutine in question. Therefore, the two pieces of code are equivalent:

BL max3

LDR r14, =return_addr
B max

BL works by copying the value of r15 into r14, since r15 right before the branch is actually taken is the instruction after the branch.

The problem with this solution is that it's not possible to make more than one level of calls; if the callee makes a call to another function, r14 is already occupied, so you need to put it somewhere else. In other words, we need a way to store more than one return address. Thus, the branch and link approach only works for calling "leaf functions", where there only exists one level of function calls.

The "real" solution for calling nested functions is to push the current link register onto the stack, such that previous return addresses are saved on the stack before a nested call is made. Then, once the last callee finishes executing, it puts the lowest-level return address in the program counter. If a callee has not called any functions, then the return address is in the link register. Otherwise, the callee pops the return address from the stack.

Arguments, local variables, and returns
...

  • How are arguments passed into the callee?
  • How are local variables in the callee stored?
  • How do we return values from the callee to the caller, and how does the caller access that value?

We now turn to these last questions.

These questions are solved with what is known as a calling convention, which is a set of rules about which registers may or may not be used for certain purposes.

So far, we've used everything from r0 to r12 as a general purpose register. However, the most common calling convention for ARMv7 is that r0-r3 are used for passing parameters into a function and returning results, while r4-r12 are reserved as strictly register variables. However, a function can use these if it stores the existing register values on the stack and then pops it back off the stack before returning to the caller.

Therefore, the caller puts up to four arguments to the callee in r0 to r3. If there are more than four parameters, the caller typically stores those parameters on the stack.

To return a value, the callee also uses r0 to r3. If there are more than four return values, the callee typically stores the extra values on the stack before returning.

Finally, the callee can also use r0 to r3 as local variables. If it needs more local variables, it can push r4 through r11 to the stack before using them, and then pop the values back to their original state before it returns to the caller.

Exceptions
...

In addition to running functions, a CPU also needs a way to handle exceptions. These are conditions that unpredictably interrupts the current program flow and causes the CPU to run an exception handler routine, also known as an interrupt service routine.

Hardware interrupts are one example of exceptions. A hardware interrupt is a digital input to the CPU, which can be toggled by an external system to signal an important event. Such interrupts are generally more efficient than polling, which is where you check if an exception condition has been fulfilled.

Exceptions play a key role in multi-tasking operating systems. The OS allocates a slice of time to each process, and it needs a way to guarantee that each process will not use more than it alloted time. This is how the OS makes it appear that many processes are running at the same time.

Since the OS cannot trust user processes (they may be buggy or malicious), it must use exceptions to ensure that a user process crash does not affect the OS. Furthermore, the OS needs to guarantee fairness to all user processes, so that one process cannot hog CPU time. These are achieved through timer interrupts.

Another reason for using exceptions is to handle runtime errors. For example, the CPU might read some illegal instruction. This would occur when the CPU is trying to read instructions in a place that is actually just data. Another runtime error might also be a divide-by-zero operation.

When an exception occurs, the CPU leaves the current instruction sequence and starts running an exception handler, switching context to the exception handler. The CPU knows the address of each of these exception handlers by storing them in a table, which is initialized right after the CPU starts up. When possible, after the interrupt service routine is complete, the context switches back to the main instruction flow.

What does it mean to switch context? It means that the state of the CPU at the moment an exception occurs is stored on the stack, so that it can be recovered after the context switches back to the original context. More accurately, a context switch is the process of storing the state of a process, so that it can be restored and resume execution at a later time. This allows multiple processes to share a single central processing unit, and is an essential feature of a multitasking operating system.
