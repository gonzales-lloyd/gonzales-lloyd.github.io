Pipelining
...

The problem
...

Our single-cycle CPU that we talked about last class does run all of the instructions in our ISA. But this is actually very slow, so let's talk about how the speed of a digital circuit is determined.

Many digital circuits can be simplified to some register output driving signals through some combinational logic into a register input. The clock frequency for a circuit like this is 1/D, where D is the delay required for the signals to go through the logic. In particular, we care about the critical path, or the longest possibly delay. This puts a cap on how fast we can make everything, since having the clocks run faster than the delay means things will quickly go out of sync. Digital circuit tools often compute this delay for us.

For our CPU, the critical path looks like this green dashed line:

And we can calculate the delay using some software, since all of this is just a bunch of combinational logic for any given situation. Another way to see this delay is by looking at a sample waveform, where the delay in each component cascades into successive components. But all of these are hooked up to the same clock, so if this delay bleeds into another clock cycle, this can be bad.

Now, you can tell that since everything is assumed to happen on one cycle, we run one instruction per clock cycle. The faster we run this clock, the more instructions we can run (up to the limit defined by our critical delay). We can't reduce the critical delay, but you might notice that a lot of the individual components in the CPU aren't doing anything while the other components are, for the same instruction.

A solution; laundry analogy
...

This is where pipelining comes in, where we can have multiple stages running simultaneously on different resources. It still takes the same amount of time to process a single instruction, but it does improve the throughput since queued instructions can be started during the evaluation of one or more instructions, rather than just at the end of each instruction.

The analogy given in class is that there's multiple parts to laundry - you put it in the washer, dry it, and then fold it. You can wait until you finish folding the laundry for one load before starting the next load (which is what our one-cycle CPU currently does), or you can start another load in the washer after putting the first load in the dryer. A full load of laundry still takes the same amount of time, but now you're utilizing your resources better. The potential speedup for an N-stage pipeline is that it's N times faster.

The pipe is most efficient when it is full. Time spent filling the pipe and training the pipe reduces its speedup. For an N-stage pipe, filling or draining the pipe each take N-1 stages.

In practice
...

What does this mean in practice? If we can split the logic into multiple stages, we can theoretically clock the circuit N times faster.

For the circuit here, we get an answer out every 5 nanoseconds (after the pipe has been filled/loaded) instead of every 10 nanoseconds, so we can double the clock speed.

However, if each stage in the pipe has different delays (4 and 6 ns, for example), then the maximum clock frequency is limited by the slowest stage in the pipeline.

Worth noting is that pipelining is not free - we must add an extra register for each stage, storing the result from the previous stage. This is usually a very small price for a significant speedup, however.

So in our CPU, we can divide our datapath into five stages: fetch, Reg, ALU, Mem, and Write Back.

We then place a register between each stage:

You can see that we can look horizontally to see when each subtask is done for each instruction, and we can look vertically to see which instruction each stage is handling at any given moment.

Regardless, you can tell that the overlap of stages is what gives this pipelined CPU its speed. Other processors have complex CPUs that can provide for even more pipeline stags, thus increasing their speedup. All high-end CPUs use some form of pipelining.

Pipeline hazards
...

The use of overlapping tasks or stages does increase speed, but it can also introduce problems that we call pipeline hazards.

Branch hazards
...

For example, let's say that a branch instruction executes in a pipeline and is taken. The CPU won't realize that it needs to change its program counter to reflect the branch it needs to take, but instead it will load the instruction immediately after the branch, since the pipeline assumes instrcctns always execute sequentially.

This is unavoidable; we don't know an instruction is a branch to somewhere else until we get to the register stage, at which point the next instruction has already been fetched. Thus, we need to flush the incorrectly-loaded instruction so that it does not execute. This is where we dynamically disable the instruction, which is achieved with extra CPU logic.

Depending on how long the pipeline is, we may flush more instructions. Our CPU loses 1 cycle when a branch is taken. This problem is called a control or branch hazard, which is an error that occurs when an instruction that should not be executed is incorrectly loaded into the pipeline because an incorrect assumption is made about whether or not a branch will be taken.

Sophisticated CPUs employ a technique called branch prediction to minimize the performance impact of control hazards, which predict in advance whether a branch will be taken. Based on the production, it chooses which code to laod into the pipeline after the branch instruction (the code after the branch, or the code pointed to by the branch). If the production is correct, the penalty is avoided; if it's wrong, we do end up needing to flush.

Branch prediction logic is very complicated, but it significantly increases performance by avoiding regular instruction flushes.

Recall that we can place conditionals after normal statements instead of using branch statements everywhere, like MOVLE and MOVGT. (This is known as conditional execution.) This is very useful in pipelined CPUs, since the decrease in code density allows us to avoid wasting additional cycles flushing instead of just having it not execute. Conditional execution does not cause pipeline stalls like branches do, making them more efficient on average.

Data hazards
...

Another problem is a data hazard, which occurs when a result is needed as an operand for an instruction before that result is available. For example, if our addition writes to r1 but r1 is also an operand in the next expression, we would normally need the entire instruction to complete (thus storing the result of the addition into r1) before executing the next instruction.

Here, the value for the AND is read from the register into the ALU before its operands have been readied from the SUB, which means that the values the AND instruction gets are invalid.

The are are three potential solutions to this:

  • Wait to run the second instruction until the data is available, known as stalling the pipeline. This always works, but obviously defeats the purpose of pipelining.
  • Add circuitry to the CPU to detect a data hazard and send the required data directly to the second instruction, called operand forwarding or data forwarding.
  • Re-order instructions so that the data hazard doesn't occur.

(Note that the register write phase and the register read phase are the same, since the register takes in all of its inputs on that cycle, but it doesn't actually spit it out to the ALU until the next cycle. The result of the SUB is written at the start of the cycle, and we don't need to read from the register file until the end of the cycle.)

Operand/data forwarding
...

Another way to resolve a data hazard is to use the result when it is computed from the ALU, rather than waiting for the result to be stored in a data register. This requires extra connections in the datapath, which would involve the use of a multiplexer to use the ALU output instead of the register file output.

Using a more formal definition, it's a method of resolving a data hazard by retrieving the missing data form internal buffers rather than waiting for it to become available in programmer-visible registers.

Load-use data hazard
...

It's not always possible to avoid stalls by data forwarding; if we're performing a load to a register followed by the use of that register as an operand, then the result actually needs to come out of the data memory instead of the ALU. However, by the time the ALU needs it for the second instruction, the data hasn't been loaded from data memory yet, making it impossible to forward.

The solution to this is to do operand forwarding from data memory for the previous instruction, but also performing a one-cycle stall so that the ALU evaluates after the data memory loads the relevant information.

Detecting data hazards
...

On a CPU that does not have operand forwarding, then it is necessary to perform the one-cycle stall for arithmetic/logical/move instructions or the two-cycle stalls for loads as mentioned above. In other words, the result of arithmetic instructions cannot be used in the next instruction; the result of an LDR cannot be used in the next two instructions. If this occurs, stalling or reordering is needed.

For CPUs that have operand forwarding, this still means that a one-cycle stall for loads is necessary. Then, the result of an LDR cannot be used by the next instruction; else, a stall or reorder is needed.

Where possible, instructions can be reordered by a compiler or human programmer to reduce or eliminate data hazards, where instructions are dependent on others.

Out-of-order executions
...

On some sophisticated CPUs, a feature exists that analyzes a stream of instructions as they are being fetched. If there are data hazards, it tries to reorder the instructions to eliminate or minimize the number of stall cycles needed to resolve the hazards. This is known as out-of-order execution.

It effectively does the same thing as a compiler or a human programmer would ahead of time; however, this feature occurs in real-time, as the instructions are going to the pipeline.

Superscalar CPUs
...

Another way to improve throughput is superscalar CPUs, which use multiple parallel execution units to process a single instruction sequence. Using the laundry analogy, this would be equivalent to having two washers, two driers, and two folding stations (or more on each, although it's hard to go past three). RISC and CISC
...

One way to classify computers into Reduced Instruction Set Computers (RISCs) and Complex Instruction Set Computers (CISCs). RISC was introduced in 1980, although most CPUs prior to that time were CISC. The CPU we have been studying is basically RISC.

Recall our variety of addressing modes:

Making a chart of addressing modes for different instruction types in a RISC CPU, you'll notice that most addressing modes are only available for load/store instructions, which are the only instructions able to access external memory. This is known as a load-store architecture, which is common in RISC CPUs.

A CISC processor can do things like this, where some CISC instructions are able to do more than one RISC instruction.


These are not available on ARM, but were available on very old CPUs. One such analogy is the difference between a four-character Chinese phrase and a 40-character English phrase. Each Chinese character contains a lot more information (and therefore there are more of them) than an English character.

Worth noting is that even though a RISC CPU requires more instructions to do the same task as some CISC instructions, this doesn't necessarily make CISC better. Complex CISC instructions are infrequently used and so identical RISC program are not much longer, not to maintain that the advantages of RISC are often much greater than the disadvantages when compared to CISC.

The advantages of RISC:

  • Because a RISC ISA is simpler, the control logic is easier to implement, making it faster. "Make the common case fast."
  • RISC instructions tend to be more uniform in size and execution time, making pipelining easier.
  • RISC instruction sets are generally easier for a compiler to target, since people don't write machine language; compilers do.

The advantages of CISC:

  • CISC CPUs often have higher code density, which was more important when memory devices were smaller and more expensive. However, memory has become much cheaper for more memory, so this is not as big of a deal now.
  • Some CISC instructions are equivalent to two or three RISC instructions. However, many CISC instructions are equivalent to one RISC instruction, with complex instructions rarely used.

Another advantage of RISC ISAs is that they tend to be orthogonal, which means that there are no data registers with special functions. That is to say, any register can be used for any purpose in an instruction, rather than having a specific register for memory fetching, another for math, and so on. (In our model, we have 3 registers r13-r15 that are special-function, while the other 13 are orthogonal.)

Most modern processors are hybrids that include elements of both RISC and CISC.

Multiplication in CPUs
...

When we multiply two decimal numbers by hand, we add partial products. If we multiply two numbers that are both N digits, we need (at most) 2N digits to hold the answer. The number of digits across all of our partial products is on the order of N^2.

To multiply two unsigned binary numbers by hand, we can actually do it the same way:

As you can see, for our 4-bit multiplication, we have a 7-bit result and 17 total bits in the partial products (including carries), which shows the 2N bit answer principal and the N^2 partial product bit count.

To make the process more suitable for assembly, we add one partial product at a time, known as "shift and add."


This is known as a software multiply, where we make the partial product for each binary place by looping over. To compute the answer of two 16-bit unsign numbers, this requires 84 cycles.

An alternative is a hardware multiplier, which adds the partial products all at once using half adders and full adders. Here's what a 4-bit by 4-bit multiplier looks like:


You can actually see how this could be pipelined for many multiplies in succession, though the one above is known as a single-cycle multiplier.
The amount of circuitry required for an NxN hardware multiplier, measured in gates, is on the order of N^2.

If a CPU has a hardware multiplier, it has at least one assembly language instruction for doing a multiply. Some variants of ARM processors have the 16x16 instruction MUL Rd, Rn, Rm, while other variants have the 32x32 instruction MUL Rd1, Rd2, Rn, Rm where the result is stored in two separate 32-bit registers.

A hardware multiplier is sometimes used in CPUs depending on the application; software multiplications allow the CPUs to be smaller or lower power, but take longer to get an answer (up to 100x longer). Hardware multipliers are required for high speed applications, but consume more power.

Division?
...

Divisions often require multiple cycles, and is inherently iterative which means that an actual hardware circuit is not often made. Integer divisions provide two results, a quotient and a remainder, which makes it more likely that software division is used instead of hardware division.

Overclocking
...

Overclocking is not an inherent feature like branch prediction or operand forwarding, but it exploits the fact that circuits are designed to handle worst-case critical delays, but are often capable of running faster due to variations in manufacturing. Overclocking can be risky, since the CPU may eventually crash or cause thermal runaway (power dissipated is proportional to clock rate).

So if you guess that your CPU game from a really good part of the performance distribution, you could clock the CPU much faster.
