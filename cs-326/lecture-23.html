Lecture 23

Resolve: prior to slide "Ordering"

In general propositional logic, order does not matter; a AND b is the same as b AND a, while a OR b is the same as b OR a.

However, in Prolog, order does matter. In the case of a typical AND statement, p :- a, b is evaluated left-to-right. Because earlier statements may initialize variables - that is, statements have side effects - this is not necessarily the same as p :- b, a.

We've emulated OR statements through

p :- a.
p :- b.

but we can also use

p: a; b.

Again, order still matters; because clauses are tried from top to bottom in Prolog's database, OR expressions can be different for the same reasons.

sidenote

Consider the statement "No other products have been proven to be more effective than this one." The intent here is to convince you that the product is the best. But the statement is still true if:

  • their sample consisted of just their product, so of course there weren't any other products that were better
  • the other products in their sample failed, in addition to their own, so they were all equally bad

For example, suppose we have this database:

child(martha, charlotte).
child(charlotte, caroline).
child(caroline, laura).
child(laura, rose).
descend(X, Y) :- child(X, Y).
descend(X, Y) :- child(X, Z), descend(Z, Y).

and we query

?- descend(martha, rose).
yes

But what if instead, we do

descend(X, Y) :- descend(Z, Y), child(X, Z).
descend(X, Y) :- child(X, Y).

What happens instead is that an infinite loop happens, and Prolog will return an error that it's out of stack space.

Imperative control flow
...

This is a feature of Prolog that gives you control over the way Prolog performs backtracking.

The ! predicate, also known as a "cut", is like a fence where once the predicates before it succeed, can never be "jumped" over again to backtrack. ! always suceeds. Additionally, the "main" clause being used is the only clause that can be used once ! is hit; no other candidate clauses can be used.

There are several common uses for cut:

  • Telling Prolog that is has found the right rule for a particular goal, i.e. "If you've gotten this far, you have picked the correct goal"
  • resolve: slide 11?

We can use cut to achieve the effect of if a then b else c.

% This doesn't work because if `b` fails, it would try 
% `c`. But in a "real" if-else, you would never even touch
% `c`.
p :- a, b.
p :- c.

% This is the solution.
p :- a, !, b.
p :- c.

not()
...

The expression not(p) succeeds if p fails. If we wanted to write if a then b else c, we can also do

p :- a, b.
p :- not(a), c.

Towers of Hanoi
...

The Towers of Hanoi is a problem in which your goal is to move N disks from the left peg to the right peg using the center peg as an "auxillary" holding peg. Only one disk can be moved at a time, and a larger disk can never be placed upon a smaller disk.

resolve: an image

How do we solve this? It turns out that we actually have a rather simple recursive solution.

First, notice that when we move the top disks to the center peg (not all at once), we are left with just the biggest disk on the left peg. Then, we can place the sole disk on the right peg. We are left with the same problem of size on the center peg, and we can use the left peg as the auxillary peg instead. This can be repeated an arbitrary number of times.

resolve: slide 15?

Here's an implementation of this in Prolog:

move(1, X, Y, _) :- 
	write('Move top disk from '),
	write(X),
	write(' to '),
	write(Y), nl.

move(N, X, Y, Z) :-
	N > 1,
	M is N-1,
	move(M, X, Z, Y),
	move(1, X, Y, _),
	move(M, Z, Y, X).

This concludes our discussion of Prolog, though more examples exist in the slides.

Java
...

We now move onto Java. Before doing so, we begin with dynamic method binding.

Suppose that we have a class called 3DObj with method draw(), which is subclassed by Cube, Sphere, and Pyramid, each of which have their own draw() methods.

Now, suppose we do

3DObj* A[10];
A[0] = new Sphere();
A[1] = new Cube();
// ...

A[0]->draw();

What draw() function is called?

  • The array was originally declared as an array of 3DObj, so we could use its draw() method.
  • But the array element actually contains a Sphere instance (which is a 3DObj), so we could use Sphere's draw() method.

Under static method binding, we make these decisions based on what is known at compile time. It is a fact that we know (since the array was declared as 3DObj) the existence of a 3DObj array, so we can use the draw() in 3DObj.

Under dynamic method binding, we know at runtime that the array actually contains an instance of a Sphere, so we use its draw() method instead.

In C++, we use static method binding by default. But if we want dynamic method binding, we use the keyword virtual in 3DObj, like so:

class 3DObj{
	virtual void draw();
}

If you do not provide an implementation (making it a pure virtual method, also known as an abstract method):

class 3DObj{
	virtual void draw() = 0;
}

then 3DObj becomes what is known as an abstract class, and you can no longer instantiate objects of that type. However, you can still instantiate pointers of that type. Then, you must actually assign an address to a concrete subclass of 3DObj, for which a concrete implementation of draw() is defined. This is indeed, relevant in Java.

Java was originally intended for embedded systems, like microwaves and TVs. Its main characteristics are that:

  • it's portable and platform independent
  • loaded dynamically from a netowrk
  • "robust" - strongly typed, no pointers, garbage collection
  • closer to object-oriented paradigm

Java is portable through the Java Virtual Machine. The Java compiler, javac, translates the source code into machine-independent source code. That Java interpreter, java, interprets and runs byte code.

The Java interpreter is implemented for a particular architecture, which means it's machine-dependent. The combination of the physical machine and the interpretor is the Java Virtual Machine, or JVM.

Since it uses an interpreter, it's slower than a compiled program. There is just-in-time compilation, which does permit code improvement.

Java class definitions
...

Java class definitions look like so:
RESOLVE: slide 22

Java has primitive and constructed types. Primitive types are the "basic" types - resolve: slide 23

Constructed types are any class name, like Point, or any array type (including those of primitives), like Point[] or int[].

Notably, Java has a hybrid variable model. For primitive types, it uses the value model; for constructed types, it uses the reference model. Some people claim that "Java is like C++ without pointers" - there are pointers, but you just don't see the pointers (or gain access over manipulating them).

Strings are predefined in Java, though they are not primitives (unlike in C, where they are arrays of characters). It is actually an object, containing the given string of characters, where the + operator has various special overloading and coercion behavior for the class String. It also has various helpful methods.

In C++, to create a new object that is is an instance of a given class, you use new and assign the result to a pointer of the relevant type. But in Java, the * denoting that something is a pointer is implicit:

Point p;
p = new Point;

p = new Point(3, 8);

To deallocate objects, you don't need to do anything; the garbage collector will eventually deal with it.

Arrays are always allocated dyanmically:

int a[];
a = new int[4];

double m[][];
m = new double[5][100]

Arrays in Java are also objects, not just contiguous collections of elements. After allocation, you can obtain the number of elements by using the field length:

  • a.length gives 4
  • m.length gives 5
  • m[0].length gives 100

These examples show you that in the case of multidimensional arrays, they're implemented as row pointers.

Classes can be labeled as public or not (with no label). Fields and methods may be public, private, or protected. Additional labels in addition to scoping are:

  • final: a constant field or method that may not be redefined in derived classes
  • static: a field or method shared by all methods, belonging to the class and not to each instance

In general:

  • no more header files
  • each class is written in a separate file, where the class name is identical to the file name (plus the .java extension)
  • or there may be several classes in a single file, but only one can be public

Interfaces
...

As in C++, interfaces in Java are a collection of method prototypes - just a header with no body. They are defined explicitly with interface:

public interface Drawable(){
	void show(int xPos, int yPos);
	void hide();
}

A class can declare that it implements a particular interface, which must then provide pubilc method definitions that match those in the interface.

Then, you can have:

public class Icon implements Drawable{
	public void show(int x, int y){
		//...
	}
	public void hide(){
		//...
	}
	//...
}

public class Square implements Drawable, Icon{
	//RESOLVE: slide 42
}

An interface can be implemented by many classes. Interfaces can also be used as a polymorphic type, similar to abstract classes in C++:

Drawable d;
d = new Icon("i1.gif");
d.show(0, 0);
d = new Oval(20, 30);
d.show(0, 0);

You can't have an object of Drawable, but you can declare it and assign concrete classes to it. And notably, you can arbitrarily reassign it to other compatible types.

You can also have polymorphism with interfaces:

static void flashoff(Drawable d, int k){
	for (int i = 0; i < k; i++){
		d.show(0, 0);
		d.hide();
	}
}

The class of the object referred to by d may not be known at compile time, but since it must be a class that implments Drawable, it has show and hide methods that can be used.
