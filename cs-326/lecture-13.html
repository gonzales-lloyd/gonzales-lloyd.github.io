Lecture 13

Last time, we found that people in the 1970s and 80s decided gotos were bad. But we still might want gotos in things like loops, where you want to exit early; we have this in C as break and continue.

break is where you want to skip the remainder of the loop early. continue is where you abandon the current iteration early and go to the next iteration. Early returns are also similar to gotos. It's worth noting that some languages do not permit early returns.

Another is exception handling, which allows you to back out of deeply nested blocks.

Something to note is that if we use goto to back out of function calls, you also need to "repair" the stack of the current subroutine; that is, do what normally needs to be done when returning from a function. A variety of languages permit exception handling, such as through throw and catch.

Selection
...

Basic selection uses if...then...else notation, introduced in Algol 60. In Algol 60 and Pascal, only one statement is allowed (or a compound statement).

But some languages allow no else. There is some ambiguity in the following code:

if x then
	if y then
		z
	else 
		a

Which if does the else belong to? In Pascal, the disambguiating rule says that the else belongs to the most recent unmatched if - that is, if y.

In other languages, like Scheme, there exists no ambuguity through its parenthetical groupings.

How does the compiler generate the code for an if...then..else statement? Suppose we have

if ((A > B) and (C < D)) or (E != F) then
	then_clause
else
	else_clause

Without short-circuit evaluation, we evaluate the entire Boolean expression before ever jumping. But what if we have short-circuit evalauation?

Instead, the target code will end up generating gotos after each individual evaluation, inserting the "short circuit" after inequalities are checked.

Case/switch statements
...

Switch statements are interesting. They are an alternative syntax for a special case of selection from a set of discrete constants. Each arm must have a value that is discrete and disjoint (in addition to else or default), that is:
- a constant
- an enumeration of constants
- ranges of constants

The simplest way to implement switch statements is as if it were a series of if-then-else statements.

Another way to implement this is to implement a jump table - we directly compute the address of the clause we want to execute based solely on the expression to switch on.

resolve - see slide 11 for example
You can see how this implements the jump table as intended (though address computation could be placed above). Notice that each explicitly specified constant has an entry in the jump table, with non-specified constants in between the relevant range jumping to the same clause as else.

Furthermore, we have additional checks to jump to the else clause if the variable to switch on is out of range.

This is (always) efficient time-wise, but not necessarily space wise. If you have a switch statement with only 1 and 1000000 in it, you can see how an if-else would be far better over creating a huge jump table with unused values. When the overall range is small and dense, jump tables are good; in large and parse ranges, you should prefer sequential testing.

A notable feature in C is the ability to "fall through" arms, in which case break becomes required. If you forget to use break, then bad things will happen.

	case 1: 
		A
		break;
	case 2:
	case 7:
		B
		break;
	case 3:
	case 4:
		...

Iteration
...

Iteration is usually implemented as a loop, and are usually executed for side effects. There are two mechanisms to perform iteration:

  • Enumeration-controlled loops, like the for loop, is executed once for every value in a given finite set.
  • Logically-controlled loops, like the while loop, is executed until some Boolean experssion is fulfilled.

In enumeration-controlled loops, the number of iterations is know, and should not change during the loop.

There are some things you need to think about:

  • changing the loop index, step, or bounds
  • empty bounds (no iteration needed)
  • evaluating "steps"
  • jumping in and out of the loop

In most languages, the loop index, step, and bounds are often prohibited. The bounds and step are evaluated exactly once in such cases, before the first iteration occurs. Notably, in Pascal, nothing is allowed to "threaten" the index variable at all.

With respect to empty bounds, we need to test the termination condition before the first iteration; if the bounds are empty, we do not execute the loop.

r1 = first
r2 = step
r3 = last
// Check for termination
L1: if r1 > r3 goto L2
	...
	r1 += r2
	goto L1
L2:
	...

// Alternative
r1 = first
r2 = step
r3 = last
goto L2
L1:
	...
	r1 += r2
L2:
	// Check for continuation
	if r1 <= r3 goto L1

For the specific code above, notice that it assumes the step is positive. If we had a negative step, you can verify that this wouldn't work - if we wanted to go from 10 to 1 with a step of -1, we would exit immediately because r1 > r3, even though we still have 10 iterations to go.

What if first/step/last are not known at compile time? In other words - what if we don't know the direction? One solution is to not use the index directly, but to instead calculate the number of iterations needed from the first/step/last. Some others are:

  • Require the direction to be declared
  • Require the step to be a compile-time constant

If we want to use iteration count that works in any direction:

r1 = first
r2 = step
r3 = max(floor(last-first + step)/step), 0)
if r3 <= 0 goto L2


L1:
	r1 = r1 + r2
	r3 = r3 - 1
	if r3 > 0 goto L1
L2:
	...

Notice that we have two extra calculations now:

  • the initial calculation of r3
  • the repeated decrement of r3 within the loop

The second extra calculation means this has some extra overhead as a result.

The last thing we need to consider is jumping in and out of the loop. It's difficult to implement and understand.

Jumping from the outside into the loop is difficult, as you need to figure out where exactly in the loop you need to go, as well as the current iteration and the number of iterations left to go (more accurately, the index and bounds). This is prohibited in almost every language.

Jumping out of the loop is known to be more permissible, as shown by break .

In C
...

We now consider this in the context of C, specifically. C provides us for, while, and do loops. An interesting thing is that we might not expect what kind of lists these are in C.

Suppose we have a linked list traversed in terms of

for(p = q; p != NULL; p=p->NULL){
	...
}

This is purely logically controlled, unlike what we might expect of a for loop. The number of iterations is not known ahead of time.

A for loop is a more concise way of expression a while loop. It turns out that all three loops in C are logically controlled. In addition to not needing to know the exact number of iterations ahead of time, you can also change the bounds, step, and index as you wish.

Logically-controlled loops
...

Logically controlled loops are simpler than enumeration controlled loops. We have:

  • Test before each iteration (while in C, most common)
  • Test after each iteration (guarantees one iteration; do in C)
  • Mid-loop test and exit (present in Modula-1; tests a condition repeatedly in a loop and may exit early)

Recursion
...

One theorem is that all recursive problems can be solved iteratively - you might just not have a great time in certain cases.

Recursion is equally powerful to iteration. But it takes more time to manage the overhead of recursion than iteration, namely, because of the stack manipulation that's required for function calls.

In turn, "naive" implementations of recursion can be less efficient than iteration.

But we have something we'll call tail recursion.

Remember that each time we perform a recursive call, we need to invoke things onto the stack. We need new stack frames at each step because we need the parameters and local variables of each recursive calls, which vary between each recursive call.

Are there cases where we don't need to make new stack frames, but "overwrite" the previous stack frame instead? The only reason we keep stack frames around is because we need to do something with the information that's already there, possibly doing additional processing afterwards.

But if calling yourself recursively is the last thing you need to do in the function, such that the only reason you're keeping stack frames is solely to keep information, then you don't need the stack frames - you just need the information.

If it's possible to just move things around in the same stack frame and just reuse it while going back to the start of the function, this is much more efficient and doesn't require stack frames.

Suppose we have an implementation of calculating the GCD in Scheme:

(define (gcd a b)
	(cond 
		((= a b) 
			a
		)
		((< a b)
			(gcd a (- b a))
		)
		(else
			(gcd (- a b) b)
		)
	)
)

Notice that when we do make recursive calls (i.e. non-base case), the last (and only) thing we do is make a recursive call. The compiler might translate this to

gcd (a, b)
start:
	if a = b
		return a
	if a < b
		b = b - 1
		goto start
	else
		a = a - b
		goto start

Notice that the recursion is gone; this function has become purely iterative.

For a function to be tail-recursive, what do we need?

  • The function must have no additional computation after the recursive calls. Since there's nothing else to do after we return to the caller, there's no need to keep all the parameters around. We can just reuse existing space.
  • The function returns what the recursive call returns. That just means we have return function(), and amounts to not having any additional computation after the recursive calls. return ... + function() is not tail-recursive without some modification.

Let's see an example of making a recursive function tail-recursive.

Suppose we want to implement summation as a recursive function:
Then we have

sum(f, lo, hi):
	if lo = hi
		return f(lo)
	else
		return f(lo) + sum(f, lo+1, hi)

Notice that we have some extra computation to do, namely adding f(lo), after returning from the recursive sum() call.

What if we want to make it tail-recursive? What we need is some sort of accumulator so that we calculate partial sums, passing in the result of adding f(lo) instead of having it be a separate addition after the recursive call.

So instead, we'll have sum(f, lo, hi, s), where s is the partial sum up to the current value (initially 0).

So we have

sum(f, lo, hi, s):
	if lo = hi:
		return s + f(lo)
	else:
		return sum(f, lo + 1, hi, s + f(lo))

Now, this is tail recursive - after calculating the parameters and making the recursive call, there's nothing left to do after the recursive call returns. Therefore, this is tail-recursive and can be implemented directly in the compiler as recursive.
