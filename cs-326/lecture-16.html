Lecture 16

Records allow data of heterogenous types to be manipulated together.

  • In C, C++, Common Lisp, and Algol 68, they're called structures.
  • C++ structures are special forms of classes with all members public - they only exist for legacy purposes
  • Java doesn't have structures at all, only classes.

In general, you typically get access to fields of records in some way. Dot notation is common, but older languages have different ways of accessing them.

Memory arrangement
...

In memory, the fields are often stored next to each other. To access records, you will need to keep its offset from the beginning of the structure in the symbol table.

It's worth noting that fields of structures are usually aligned (often to every 4 bytes). For efficiency reasons, fields are typically aligned to addresses divisble by 4. For example, a 1-byte field (e.g. a 1-byte bool) actually takes up 4 bytes in the structure, with 3 bytes of padding for alignment.

You can pack records so that there is no padding, but you sacrifice speed for space. Since we can only access one word at a time, if the removal of padding means that a given field spans several words, you'll need to access each of those words to get the data in the field in its entirety.

Here, atomic_weight requires three address accesses instead of just two.

Assigment
...

Assignment of structures is often allowed in most languages; in general, this is a bit-by-bit copy.

But there's something worth noting. If you have a field in the structure that's a pointer to some other data in memory, performing assignment means that the new structure will also point to that same data. This means that if you want two truly independent copies, you will need to copy the external data separately and update the pointers in the record accordingly. (In other words - if some fields are pointers to dynamically allocated objects, only the pointers are copied, not the objects.)

Comparison is generally not allowed. If we can do bit-by-bit assignment in most languages, why don't most languages also permit bit-by-bit comparisons? The main issue is that the padding may contain random garbage. You can fill the holes with zeros at allocation time, but this is time consuming.

With statements
...

Consider the following code:

ruby.chemical_composition.elements[1].name := 'Al';
ruby.chemical_composition.elements[1].atomic_number := 13;

Obviously this is not very easy to write, so some languages like Pascal have expressions like:

with ruby.chemical_composition.elements[1] do
begin
	name := 'Al';
	atomic_number := 13;
	atomic_weight := 26.98154
	metallic := true;
end;

In C, which doesn't have with, we can use pointers instead:

p = &ruby.chemical_composition.elements[1];

p->name = "Al";
p->atomic_number = 13;

But there are some problems with the with block of Pascal:

  • You get a special kind of local scope, where the available variables are those accessible in the record/structure. Global variables (and any other outer variables) of the same name are always shadowed.
  • You cannot "open" two records of the same type simultaneously
  • It's possible to use nested with statements, but this is annoying.

In Modula-3 and other languages, you can assign an alias:

WITH e = ruby.chemical_composition.elements[1] DO
	e.name := 'Al';
	e.atomic_number := 13;
	...
END;

You can also access several records and make aliases on arbitrary expressions:

WITH e = whatever, f = whatever DO
	...
END

WITH d = complicated_expression DO
	...
END

This is very similar to performing a #define in C.

Variant records/unions
...

Variant records are records that provide severfal alternative fields or collections fo fields, where only one is valid at any time.

How are these laid out in memory?

  • The size of the structure is always the same, with the overall size based on the largest "case" of the structure
  • Variants can share the same space.

In C, we have unions; they're slightly awkward to use, however.

Notice how you must access the "shared" region first before accessing the variant fields.

Historically, you could use equivalence, in Fortran, so that you told Fortran two fields would ever be used in the same time.

But there are several safety issues with variant records.

  • The lack of a tag, or discriminant; you don't necessarily know what the "alternative field" is until you actually check (and even then, you still might not know)
  • If a tag is provided, you may have the ability to change the tag and the fields independently, in which case it's possible the tag does not reflect the underlying data.

Arrays
...

Arrays are a sequence of elements with the same type; mathematically, indexing into an array can be thought of as a function that maps the index type to the element type.

In most languages, the index type must be discrete, though some languages like Awk and Perl permit non-discrete index types. These are associative arrays, which are implemented with hash tables.

Generally, languages use either paranthetical or bracket notation for array accessing.

Slices are rectangular portions of an array. Fortran 90 permits extensive facilities for slicing. Such slices can be extracted as if they were smaller arrays on their own.

Selection of an element and asisgnment exist in all languages. There are several operations that can be performed on entire arrays (or slices of the array), including:

  • equality tests
  • comparisons for lexicographic ordering
  • bitwise operations
  • arithmetic operations
  • mathematical functions

The shape of an array is the number of dimensions and bounds for each dimension. It may not be known at compile time, which has some impact on allocation.

We have a few cases:

  • Global lifetime, static shape.
    • Global variables in C, Pascal
    • Allocated statically
  • Local lifetime, static shape.
    • Local variables in C, Pascal
    • Allocated on stack
  • Local lifetime, shape bound at elaboration time (when a function is entered).
    • Local variables in Ada
    • Allocated on stack, with an extra level of indirection
    • Stack frame divided into fixed-size part (to support instructions having fixed offsets) and variable-size part
  • Arbitrary lifetime, shape bound at elaboration time
    • Java arrays, allocated with new
    • once allocated, shape remains the same until deallocation
    • dynamically allocated on heap
  • Arbitrary lfietime, dynamic shape
    • Perl arrays
    • shape of the array can be changed during its lifetime
    • if size is increased, allocate a larger block, copy all the data over, and then deallocate the old block
    • dynamic allocation on the heap

With respect to the third case, consider the following:

procedure foo (size: integer)
	M: array(1..size, 1..size);
begin
end foo;

How do we stick arrays on the stack for which their size is not known? So consider the case in which we have more items after the array; then because we don't know the size of the array, we can't possibly know where the rest of the variables will be at compile time.

And the compiler needs to generate instructions that are at some fixed offset to access these variables. Suppose our stack frame looks like:

  • var1
  • var2
  • array of unknown length
  • var3

How are we going to generate the code to get to var3? (We're fine generating code with var1 and var2, since those aren't affected by our variable-length array.)

The general solution is to simply store the array elsewhere and put a pointer to that array in its place; since we know the size of pointers ahead of time, we'll also know the position of var3 ahead of time. We stick the variable-size array towards the top of the stack instead.

So to resolve "local lifetime, shape bound at elaboration time" arrays, they're allocated on teh stack, but with an extra level of indirection. In turn, the stack frame is divided into a fixed-size part and a variable-size part.

Memory layout of arrays and matrices
...

For one-dimensional arrays, they are always allocated contigously (or not at all).

But matrices, or mutli-dimensional arrays, can be allocated in two strategies. One is contiguous allocation, where you just put one row after the other. But the other option is to just have a one-dimensional array of pointers, with each pointer pointing to another one-dimensional array of a specific row. This means the rows themselves must be contiguous, but everything else need not be.

Continguous allocation
...

In contiguously allocated multi-dimensional arrays, they are often done in row-major order (that is, traverse rows before columns); some languages, like Fortran, use column-major order.

This is the difference between doing

for each row
	for each col

for each col
	for each row

The results are exactly the same in that you'll access everything once. But if you know your language operates in a specific order - either column-major or row-major; one will be more efficient.

If you have row-major order, the first snippet of code runs faster; the elements are accessed sequentially in memory. But more importantly, you'll hit the cache more often; the principle of locality states that if you access something in memory, then you will likely access things near it as well. So the next elements are likely to be loaded into the cache from prior accesses (which is less likely if you jump around many addresses by going "against" the memory arrangement).

Accessing the elements in each row - that is, sequentially - in row-major order, then you'll bring various the next elements into the cache as well. Accessing elements exactly as they are allocated in memory allows you to hit the cache more often.

While you can certainly still calculate the position of elements if you go "against" a specific order, you'll be missing the cache pretty often if the cache blocks aren't large enough. This is why you go row-by-row in C.

Row pointers
...

The main disadvantage of using row pointers is that you have some extra overhead; the pointers themselves are not "useful" in that they don't contain usable data; they just point to each row.

But it comes with several major advantages, including:

  • allowing rows to be allocated anywhere
  • it's not necessary to use multiplication to figure out where elements are
  • can accomodate rows of different lengths, like an array of strings

Array address calculation
...

So how do we calculate the address of an element in an array, one-dimensional or otherwise?

Contiguous allocation
...

On contiguous allocation (row-major), you simply take the size of the element type and multiply it by its "offset" - the size of a row times its row number plus the column in that row. Start from the address of the array and go as far as you need.

That is, if we have an array [L1..U1] of arrays [L2..U2] of elem_type, let:

  • S2 is the size of elem_type
  • S1 is (U2-L2+1)*S2, the size of a single row

Then the address of A[i, j] is &A + (i - L1) * S1 + (j - L2) * S2.

Alternatively, we can rewrite this as (i * S1) + (j * S1) + &A - ((L1 * S1) + (L2 * S2)). The advantage of this expression is that we can precompute the multiplications of L1/S1 and L2/S2 at compile-time even if we don't know the size of the array (if dynamically allocated), which saves time.

Row pointer allocation
...

In general, it's fairly simple; there's no multiplication, but more loads are required from memory. In low-level assembly:

; A[i][j]
; i is in r1, j is in r2
r4 := &A          ; Get location of start of array
r4 := *r4[r1]     ; Go to pointer of desired row 
r5 := *r4[r2]     ; Go to column in selected row, return as r5

This requires no multiplication, but does involve more loads from memory.
