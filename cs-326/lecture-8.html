Lecture 8

Consider a function filter which takes f, a predicate (a function returning true or false), and a list L. It returns a list containing only elements in L that satisfy f.

> (filter number? '(a 4 (1 2) "test +"))
(4) ; Note that this is a list!

> (filter (lambda (x) (>= x 0)) '(-2 3 -1 4))
(3 4)

How do we implement this?

(define (filter f L)
	(cond
		((null? L) 
			'()
		)
		((f (car L))
			(cons
				(car L)
				(filter f (cdr L))
			)
		)
		(else
			(filter f (cdr L))	
		)
	)  
)
  • In the base case, simply return a null list.
  • Here, we again take the approach of letting the head of L be the "current" element. If the head fulfills the predicate, then we construct a list consisting of the head and the filtered tail.
  • If the head does not fulfill the predicate, simply exclude it from our result and run filter against the remainder of the list.

Importantly, note that the syntax for passing around functions is simple. You don't need to pass around function pointers, as C would require you. This also demonstrates the ability to define functions in place and have them work.

Sequencing
...

We have not talked about sequencing. begin is a function with the following syntax:

(begin <exp1> <exp2> ... <expn>)

where <expn>, the last expression, is returned. This is useful, for instance, in a if statement. if takes exactly three arguments - the condition, the expression if true, and the expression is false.

But what if you want to do multiple things in a given if expression? This is where begin is useful:

(if (< x 0)
	(begin 
		(display "negative")
		(f x)
	)
	(begin
		(display "positive")
		(g x)
	)
	
)

You can see here that this allows us to not only display a message, but also return an actual value from the if statement.

begin is significant in the context of function calls with side effects. It's true that we only return the last value, but the other functions in the begin call might have several side effects.

Pretend we have the following:

> (define x 1)
> (+ x 1) ; Note the list is evaluated, but x remains the same.
2

> x
1

> (define y '(1 2 3))
> (reverse y)
(3 2 1)

> y
(1 2 3)

All of these functions do not exhibit side effects. They do not alter their inputs. That is to say, these functions only compute and return a result, not altering their input. Functions with side effects include define and display.

  • define has a side effect in that it does affect its input, the identifier passed in.
  • display modifies the environment by performing I/O.

Notice that we have never used imperative constructs with side effects. They do exist in Scheme, but they are generally discouraged.

Internal structure of expressions
...

All variables are pointer-bound to values. For example:

  • if you have an atom like (define x 1), then the result is that x is a pointer to the value 1.
  • if you have a list like (define y '(1 2 3), y is a pointer to the head of a linked list, where each element in the linked list contains a pointer to the next node, as well as pointing to the value that this node contains.

Note: when I say "head" of a linked list, I refer to the first fully-fledged node of the linked list.

In a list, each element is a cons cell with:

  • a pointer to a value
  • a pointer to the next cell

This is significant because it means that:

  • car simply returns the value pointed to by the head of the linked list
  • cdr simply goes to the next node in the linked list and returns that, because that node onward is a sublist itself (and represents the tail)

Say we have the following:

> (define x '(b c))
> (define y '(d))
> (define z (append x y))

What happens here?

Notice a few things:

  • We must make a deep copy of the first argument to the append.
  • The end of the newly-created list from the first argument to append points to the start of the original terminal end of y.
  • We cannot simply have the end of x's linked list point to the start of y's linked list, because that would modify x (even though it would have z hold the "correct" value).

Imperative programming
...

Let us now refer to something we'll call list surgery.

To alter the structure of a list, we have two functions:

  • set-car! which changes the pointer-to-value
  • set-cdr! which changes the pointer-to-next

For example, say we wanted to change the first element in a list:

> (define x '(a b c))
> x
(a b c)

> (set-car! x 'm)
> x
(m b c)

and if we wanted to change the second element, we would do

> (set-car! (cdr x) 'm)
> x
m m c

and you can see how this could be done continuously.

Say we now wanted to change the tail.

> (define x '(a b c))
> (set-cdr! x '(m n))
> x
(a m n)
On garbage collection in Scheme

There is something important to note about all of the cdr/car changes. Notice how when we change the cdr two examples up, we just have the "floating" elements for b and c. Ideally, we would like to free this memory up, since it becomes inaccessible. Indeed, this is deallocated/freed by the garbage collector, since it notes that this memory is now inaccessible.

The fundamental idea is that such elements are freed if there are no longer any active pointers that point to the element. For example, if we had a pointer y pointing to the tail of x (that is, the node pointing to b) and we were to perform this operation, both b and c would remain in memory.

But if we were to have a pointer y pointing to just the node containing c, eventually b would be garbage-collected.

Garbage collection occurs on the regular, but you can also force it using (gc).

Now, let's say we wanted to insert an element into the second position.

To do this:

> (set-cdr! x (cons 'm (cdr x)))
> x
(a m b c)

Here, we construct a new list where m and the remainder of the tail is appended together, and that becomes the new tail. This does what we want.

A discussion on programming languages
...

Consider the following issues in any given language:

  • syntax, semantics
  • naming, scopes, bindings
  • control flow (any capacity to determine what happens next - if/then/else, loops, jumps, etc.)
  • data types
  • subroutines

There are also issues specific to particular classes of languages:

  • data abstraction in object-oriented languages
  • non-imperative programming models in functional and logic languages
  • concurrency

Let us now consider names, bindings, and scopes.

  • A name is just some mnemonic character string you can use to refer to something.
  • A binding is an association between a name and the entity it names.
  • A scope refers to the textural part of the program (the literal source code) where a binding is active (not a name, since the same name can refer to different things in different scopes).

Naming
...

What names you can use are usually enforced by a language's specification. This includes things like name length, allowable characters, and so on. The name @#$%^& is legal in Scheme, but not in C. Names in C are case-sensitive, but not in Pascal.

There are also things like naming conventions (recommended, but not enforced), like imperatives in Scheme ending in a question mark (which is not required!). In the Windows API, you may see things like SZName, bBooleanVar, and fFloatVar that denote the type of the variable in the prefix itself. This is known as Hungarian notation.

Another naming convention is m_iIntVar, where i denotes an integer and m_ denotes that it is a member of a class.

Another is OnMouseClick(), which is related to functions responding to an event in event-driven programming. Such functions are often named On<Event>, where Event is whatever the semantic event is.

Binding
...

Binding can be more general than the definition given above.

Binding time is when we make associations. For example:

  • during the language design time, we decide what keywords are significant, how they relate to each other, and so on.
    • what the control flow constructs are: if, while, etc.
    • fundamental/primitive types: int, float, etc.
  • during the language implementation time, we decide how we handle runtime exception, datatype range and precision
    • how I/O operations are coupled to the OS's file implementation
    • how runtime exceptions like arithmetic overflow are handled
    • the precision/width/number of bits for primitive types
  • programming writing time
    • algorithms, choosing names
  • compile time
    • the mapping of high-level constructs to machine code
    • the layout of (static) data in the memory
  • link time
    • the layout of the whole program in memory
    • bindings between names and objects in different modules
  • load time
    • conversion from virtual to physical addresses
  • run time
