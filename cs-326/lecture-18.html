(more) pointer-array duality
...

We have the sizeof() operator, but what does it do?

Suppose we have:

int a[5]
int *p
p = (int*) malloc(5 * sizeof(int))
int **s;
s = (int **) malloc(...) // Allocate a 5*5 matrix

What will various calls of sizeof() return?

sizeof(int) == 4
sizeof(a) == 20 
sizeof(p) == 4 // for 4-byte addresses
sizeof(s) == 4 // for 4-byte addresses

This is because sizeof() does not actually generate any functional code; these are all evaluated at compile time.

Note

sizeof() against the array only returns 20 because this information is available on the stack. Once you start passing the array arounda s a pointer, it'll start returning the size of an integer pointer instead.

C declarations
...

The rule for C declaration is to start at the variable name, read right as far as you can (subject to the parentheses), then left, then go out a level of parentheses and repeat.

For example:

int *a[n]; // Array of n pointers to integers
int (*a)[n]; // Pointer to array of n integers
int (*a)(int*); // Pointer to a function taking a pointer to an integer as an argument, with the function returning an integer

References
...

How are objectes deallocated?

Generally, we have three storage classes:

  • static: never deallocated, has the same lifetime as the program
  • stack: deallocated automatically on subroutine return
  • heap: explicit or implicit deallocation:

Several examples of explicit deallocation:

  • dispose in Pascal
  • free in C
  • delete in C++ (which calls the destructor, then deallocates)

A dangling reference is a pointer that no longer points to a live object. Generally, it is produced in the context of heap allocation:

p = new int;
r = p;
delete r;
*p = 3; // crash

But you can also have it produced in the context of stack allocation (not in Pascal, which only has pointers to heap objects). In C, the "address-of" operator makes this possible:

int* f(){
	int x;
	return &x;
}

void main(){
	int *p;
	p = f(); // x deallocated after exit
	*p = 3; // crash
}

It is important to note that dangling references can only be produced in languages with explicit deallocation. This is because it allows the programmer to deallocate objects that are still referred to by live pointers. Garbage collection for implicit deallocation, however, checks if objects still have references to it before deleting it.

We need to detect dangling references to warn the programmer - to generate an error, rather than silently retrieving some garbage. There are two mechanisms to detect dangling references:

  • tombstones
  • locks and keys

Tombstones
...

resolve: see slide 7

The idea behind tombstones is to have an "intermediate" pointer, where variables point to a tombstone which in turn points to the actual object.

This constitutes an extra level of indirection, such that you place a special value in the tombstone when a delete call is made against any of the pointers pointing at the tombstone. Then, when any other pointer pointing at the tombstone tries to access the tombstone's pointer, it will discover the special value in the tombstone indicating it's already been deleted.

It is important in several aspects:

  • It is guaranteed to catch all dangling references
  • It can handle both heap and stack objects
  • It makes storage compaction easier, since it is much easier to update just one pointer (the tombstone's) rather than all of the various pointers to the tombstone/object

There is some overhead in managing tombstones. In a time sense:

  • it takes time to create tombstones when allocating objects
  • checking the validity on every access
  • double indirection

In a space sense:

  • you may elect to never deallocate them since they're relatively small
  • you may also elect to deallocate them by keeping track of the number of references to it, which requires some additional overhead

Locks and keys
...

slide 9

Upon allocation, we generate a number, storing it in the object as the "lock" and in the pointer as the "key". If the key is the same as the lock, then the access is considered valid.

Upon deallocation, we put a special value in the lock, such if the "key" doesn't match, the access is considered invalid.

It is necessary to keep a "key" because of the following scenario:

  • Suppose we just have some value in an allocated block of memory representing the "lock".
  • We have a pointer to this lock with no "key".
  • We deallocate the block, setting the lock number to the special value.
  • We ask for more memory, and if it happens that we end up using the same block of memory, the lock number is reset and we get a new valid pointer
  • But the old pointer still points to that block, and now that it sees the lock is "valid", it will consider the access valid, despite this not being its own data anymore

Locks and keys have a few notable properties:

  • They aren't guaranteed to catch every dangling reference: a reused block may get the same lock number, though this occurs very rarely
  • They only (really) work for objects on the heap. If we were to apply this to stack objects (to account for "address-of"), we would end up with a "lock" on every single local variable, which is overkill. This takes up even more space, since a lock would need to be on every local variable and argument, regardless of whether or not their address is taken.
    • We didn't have this object with tombstones, since 100 pointers to the same object only needed one tombstone; we'd need 100 locks if we used locks and keys.

resolve: slide 10
It takes time to compare locks and keys, as well as time to generate the locks and keys themselves.

It clearly takes space to store all of the locks and keys; every "object" will have some extra space for this system.

Garbage collection
...

When explicitly deallocating heap objects, this has implementation simplicity and speed. The issue is that this increases the burden on the programmer, such that mistakes can produce memory leaks and dangling references.

The automatic deallocation of heap objects, or garbage collection, is safe (no memory leaks or dangling references) and convenient for the programmer. But there exists complex implementation and run-time overhead behind it.

Garbage collection is essential for functional languages, where we have frequent construction and return of objects from functions. It is increasingly used in imperative languages, like Clue, Ada, and Java. In general, we have two mechanisms:

  • reference counts
  • mark-and-sweep

Reference counts
...

How do we know when a heap object is no longer useful? It's when there are no more pointers to it, and therefore the object is unreachable. The general solution is:

  • Keep a reference counter in each object, which is equal to the number of pointers referring to the object
  • When allocating the object, increment the reference counter by 1
    • p = new int
  • When assigning pointers, decrement the reference counter of the original pointer, but increment the reference counter of the pointed-to object:
    • p = r;
    • the object originally referred to by p has its reference count decremented
    • the object currently referred to by r and now referred to by p as well has its reference count incremented
  • When the reference count is zero, you can destroy it

Does this always work? Consider the fact where we have a (binary) tree with a bunch of pointers to each of its elements. Each object initially has a reference count to 1, including the root itself, which you keep around. Suppose that you move the pointer to the root node away, such that the root node now has a reference count of 0.

So you destroy the root; but the problem is that with our current rules, we don't do any "recursive" destruction or decrement the objects that are referred to be the soon-to-be-destroyed object.

So an additional rule:

  • If the object referred to by p can be deleted (such that its reference count is 0), recursively follow pointers within it to decrement refcnt for those objects, and delete them as well if their refcnt becomes zero as a result

There's one more case we need to consider. Suppose we have pointers declared as local variables:

void f (int* x){
	int i;
	int* p = x; // the refcnt of object referred to by x is incremented
	// all local variables die here, so we must find all 
	// the local variables that are pointers and decrement 
	// their associated reference counts
	return; 
}

The solution to this is to use type descriptors:

  • At the beginning of each record (like a stack), specify which fields are pointers
  • in each stack frame, specify which variables are pointers

One more problem is that of circular lists. Suppose you have a circular linked list, and you have some pointer into the linked list. Therefore, exactly one node has two references, and all the other nodes have one reference. But if you delete "your" node, all the other nodes will still have one reference, and so we would never delete these.

Mark-and-sweep
...

Whenever the free space becomes low:

  • Walk through the heap, and mark blocks (tentatively) as "useless"
  • Recursively explore all pointers in the program (to find recursive data structures), marking each encountered block as "useful"
  • Walk again through the heap, and delete every "useless" block that could not be reachedi

To find all pointers, we must use type descriptors (to actually know whether something is a pointer or not.)

To explore recursively, you do need a stack:

  • the maximum length of the stack is the longest chain of pointers found
  • but there might not be enough space for a stack, since the free space is already low
  • we can perform this exploration instead using pointer reversal

Pointer reversal is an exploration approach whereby we keep a pointer to the previous element and the current element. You can see how if we go two steps down, we would already lose access to the root of a recursive structure. So right before we go "downwards", we reverse the direction of the pointers in the structure to point upwards instead of downwards, such that we can make our way back by following the reversed pointers.

And to fix the data structure, as we come back, we put the pointers back to the way they were.

Storage compaction
...

How do we reduce external fragmentation?

An elegant solution is the stop-and-copy algorithm, which can also be used to achieve garbage collection.

  • Achieve compaction and garbage collection and simultaneously eliminate steps 1 and 3 from the mark-and-sweep algorithm
  • resolve: slide 17?

What is the advantage over standard mark-and-sweep? This is linear as well compared to mark-and-sweep, but the difference is that we only visit those that are still in use - not every single block in the entire heap. In turn, we end up making fewer accesses:

  • No more walks ("sweeps") through the entire heap
  • The overhead is proportional to the number of used blocks, instead of the total number of blocks

Comapring the garbage collection strategies
...

There's a significant difference between the reference count strategies and mark-and-sweep-strategies. The reference count approach is on an "immediate" basis; the moment a reference count hits zero, it goes away. When an object is no longer needed, it can be immediately reclaimed.

For mark-and-sweep, you have a "stop-the-world" effect, where the whole program must stop on an as-needed basis.
