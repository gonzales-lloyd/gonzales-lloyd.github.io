Lecture 2

There are various reasons we might want to use compiled vs. interpreted languages.

See lecture 2 slides.

There are a few set of tools you might use in your programming environment, like:

  • editors
  • pretty printers
  • style checkers
  • preprocessors
  • debuggers
  • linkers
  • module management
  • browsers (cross-references)
  • profilers
  • assemblers

In some cases, these are provided separately, as is often the case in Unix, or as an integrated environment, like Microsoft's Visual C++.

Semantics
...

There are two aspects of correctness - syntax and semantics. (We may also want to analyze it lexically, to break it up into its individual words).

Syntactically, "you should go there" is valid. Semantically, "you" is ambiguous.

Compilation occurs in a few stages:

  • The character stream is passed to the scanner, which performs lexical analysis and outputs tokens.
  • Those tokens are passed to the parser, which performs syntactic analysis and generates a parse tree.
  • The parse tree is analyzerd by a semantic analyzer and intermediate code generator, which creates an abstract syntax tree or some other form.
  • The AST can be passed to a machine-independent code-improvement component, which is optional.
  • The modified intermediate form is passed into a target code generator, which creates assembly or machine language.
  • If applicable, machine-specific code improvement may be made. This generates machine code in the target language.

We usually treat the first three steps as the "front-end" since it is machine-independent; the last three steps are machine-dependent. Often times, a compiler will physically separate these sections of code.

The scanner, or lexical analyzer, will extract tokens - the smallest meaningful units of code. Tokens include keywords, grouping characters, variable names, and operators.

This outputs a context-free grammar, dedscribing the language syntax. For example, a program may be described as
PROGRAM identifier ( identifier more_identifiers ); block

Blocks look like:
labels constants types variables subroutines BEGIN statement more_statements END

Identifiers look like
, identifier more_identifiers

and identifiers may have an indefinite amount of identifiers contained in them, up until is reached, which represents an empty string (which is the end of the recursive description of the identifier).

The parser generates a parse tree reflecting the structure of the language. This shows how the syntax can be expanded according to the definitions of the language.

Semantic analysis discovers meaning in a program.

There is static and dynamic analysis. Static analysis occurs at compile time, past the syntax errors. For example, the use of an undeclared variable is syntactically correct, but not semantically correct. Another is calling a function with incorrect arguments (maybe the wrong number or type of args). Such analysis checks if:

  • identifiers have been declared before use
  • subroutine calls provide the correct number and type of arguments

There are also dynamic semantics, which cannot be checked at compile time and must be checked at runtime. Examples include indexing an array out-of bounds, or dereferencing null.

The parse tree is simplified to a syntax tree. The symbol table is maintained, with attributes attached as applicable.

Usually, you only get indices into the symbol table, which the simplified syntax tree refers to.

Moving onto the back end, target code generation:

  • Generates assembly or machine language
  • Traverses the syntax tree to generate elementary operations, like loads and stores, arithmetic operations, tests and branches

There is also code improvement, which is optional. We often call this optimization, though "improvement" is the formally correct term due to optimization's mathematical definition. Usually, this involves using less memory or improving speed.

In sum:

  • Lexical analysis occurs through the scanner, which breaks up the program into primitive components called tokens (identifiers, numbers, keywords, etc.) and involves the formal models of regular grammers and finite automata.
  • Syntactic analysis occurs (see slides)
  • Semantic analysis understands the relationships between tokens in the program.
  • Code improvement, which is machine-independent, rewrites the syntax tree to create a more efficient program.
  • Code generation, which converts the parsed program into an executable form in machine code.
  • Machine code improvment, which is machine-dependent improvement.

Formal translation models
...

A formal language is a set of strings of symbols drawn from finite alphabet. Because it's a set, then it holds that there's a language which contains no words - the empty set.

The language might be English, the words "black" and "white", some random combination of binary, and so on. A language composed of the letters "a" and "b" in any order and length would contain an infinite number of strings. Languages have finite alphabets, but the set of strings may be infinite.

Let's talk about grammars. The set of nonterminals is a finite set of entities, like sentences, subjects, predicates, verbs, articles, and then noun. We define the various parts of speech (and a sentence) in this way.

The set of terminals is a finite set of concrete entities, like "the", "boy", "girl", "ran", "ate", "cake".

Finally, we need a start symbol, which is one of the nonterminal elements that indicates the start of a sentence.

Rules, or productions, are a finite set of replacement rules. For example:

  • sentences must contain subjects and predicates
  • subjects must contain articles and nouns
  • predicates must contain a verb, article, then noun
  • verbs consist of words like "ran" and "ate"
  • articles consist of words like "the"
  • nouns consist of words like "boy", "girl", and "cake"

In a derivation, we expand elements in order. Doing it from left-to-right is referred as leftmost derivation. For example, the derivation beginning from sentences yields:

  • <sentence>
  • <subject> <predicate>
  • <article> <noun> <predicate>
  • Here, we observe that we can replace <article> with the terminal "the" , based on our rules above. So we expand it like the <noun> <predicate>.
  • Expanding <noun> with something from the grammar gives us three choices, which we are free to expand as we see fit. This yields the boy <predicate>.
  • the boy <verb> <article> <noun>

Based on the rules, we are free to say things like the boy ate the cake or the cake ate the boy. Semantically, the second sentence is odd, but syntactically, it is fine.

Let us define another grammar as

  • B -> 0B | 1B | 0 | 1

Then we observe B is the only nonterminal, and therefore must be a start symbol. Then, if we want to generate 010, we expand B into its first rule, 0B. Then, we still have the nonterminal B - we expand it to 01B, then replace B with 0 to leave us with the terminals 010.

Notice how all nonterminals are internal nodes, where as all terminals are leaf nodes. By definition, the nonterminals are those that must be expanded, and therefore shall have children. And of course, terminals are leaves by definition, as they cannot have children.

Let us consider another grammar defined as
S -> SS | (S) | ()
and try to generate (())(). Note the quality that the grammar itself requires matching parentheses.

Then our derivation might look like:

  • S
  • SS
  • (S)S
  • (())S
  • (())()

Notice how if we read the leaves from left to right, we will yield our target string.

Generating the string ()()() requires us to perform the following expansion:

  • S
  • SS
  • SSS
  • ()()()
