A brief review of pointers:

> (define x '(a b))
> (define y x)
> (eq? x y) ; Pointer equality, not value equality!
#t

We mentioned that we would like to determine deep-membership; that is, a function checks to see if an element is present in a list or any of its sublists.

(define (member-d x L)
	(cond
		((null? L) ; Cond 1
			#f
		) 
		((list? (car L)) ; Cond 2
			(or
				(member-d x (car L)) 
				(member-d x (cdr L)) 
			)	
		)
		((equal? x (car L)) ; Cond 3
			#t
		)
		(else ; Cond 4
			(member-d x (cdr L))
		)
	)
)

A brief description of each condition:

  1. If the list is empty, clearly the element isn't there. This also serves as a base case.
  2. If the current element we're looking at is a list itself (the head), then perform a recursive call on both the sublist and the remainder of L. This is necessary because calling it on the sublist alone would cause us to skip over the rest of L. If the element is present in either the sublist or the remainder of the list, we want to return true, so we berform a boolean OR.
  3. If we find value-equality, then return true.
  4. If we haven't found the desired value in the head (and it's not a list), then recursively call member-d with the remainder of the list.

You can see how we use the head as the "current" element, like in a for loop.

Now, consider a function that returns the nth element in a list, assuming that L is not empty and :

(define (nth L n)
	(if (= n 0)
		(car L) ; return the head
		(nth (cdr L) (- n 1)) ; perform recursive call with smaller list
	)
)

Consider a case where we would like to reimplement append, which takes in two lists and returns the concatenation of them, having the elements of both.

Notice how the concatenated list's head and tail is built.

  • The head is simply the head of the first list.
  • The tail is the concatenation of the tail of the first list with the entirety of the second list.
(define (append L1 L2)
	(cond
		((null? L1) L2)
		(else
			(cons (car L1)
				(append (cdr L1) L2))
		)
	)
)

We can perform a recursive call, noting how our statement that we concatenate the tails together is the same problem, but smaller.

Consider a function where we want to return a copy of a list with the elements in reverse order. (That is, this function is not in-place.)

(define (reverse L)
	(cond 
		((null? L) '())
		(else
			(append 
				(reverse (cdr L))
				(list (car L)) ; N1
			)
		)
	)  
)

Our approach here is to take the head of the list and place it at the end of the target list, then reverse the tail and place it at the start of the target list.

You can see that we would perform an append of the reversed tail and the current head. This is where our recursive call occurs.

N1: append only works on lists, but (car L) will only return a single element. We need to create a list from this element for it to actually work.

At a high level, you might notice that the base case usually involves null. Most of the time, recursive problems tend to involve the head and a recursive call against the tail. The list's head is often used as the "current element", like in a for loop.

In general, you will probably not write the entirety of your functions at the interpreter each time. You can use load to run things asn needed:

> (load "file_name")
> (reverse '(1 2 5))

Note that like most languages, load only looks at the working folder. You may use absolute paths as desired. Additionally, load is not dynamic in the sense that if a loaded file changes, those changes are not immediately reflected. You must call load again.

Some other useful definitions:

  • let
    • has a list of bindings and a body
    • each binding associates a name to a value, somewhat like define
    • bindings are local, and are only visible inside the let expression
    • evaluates the containing expressions and returns the last expression in the body
    • makes its bindings in parallel
> (let 
	( ; The bindings.
		(a 2)
		(b 3)
	)
	(+ a b) ; The body.
)
5

> a ; Out of scope.
Error: variable a is not bound

> b
Error: variable b is not bound

With respect to let making its bindings in parallel, consider the following example:

> (let 
  (
	  (x 1)
	  (y (+ x 1))
  )  
  (list x y)
)
Error: variable x is not bound

Because the two bindings are made in parallel, then when y is attempted to be bound, x does not yet exist.

But what if we want to do it in that order, such that x is bound and then y is bound? We have let*, which is similar to let but makes its bindings sequentially.

> (let* 
  (
	  (x 1)
	  (y (+ x 1))
  )  
  (list x y)
)
(1 2)

You can, of course, force sequential bindings by performing nested lets (which is actually how let* is implemented):

> (let
	((x 1))
	(let 
		((y ( + x 1))) ; x is in scope because it's in the top-level let.
		(list x y)
	)  
)
(1 2)

(Such "shorthand" ways to do things is known as syntactic sugar. It doesn't add new functionality, but can make code and intent clearer.)

Why would we want to do things in parallel? Pretend we have

> (define x 'a)
> (define y 'b)
> (let
	(
		(x y)
		(y x)
	)
	(list x y)
)
(b a)

then we can swap the values of x and y.

A final note: indentation is important, because the cost of such simple syntax is that you have to keep track of your formatting. You may use any style you want, but make sure you are consistent.

In all languages, there exists some functionality to do I/O. In Scheme, this includes:

  • read
  • display
  • newline

Let us define a function that asks for a number but keeps asking if the input is not a number:

(define (ask-number)
	(display "Enter a number: ")
	(let ; We want a local variable to store the user's input.
		(
			(n (read))
		)
		(if (number? n)
			n
			(ask-number)
		)
	)
)

Notice/recall that all recursive problems can be rewritten iteratively, and vice versa. (Perhaps not as easily, but it is always possible.)

An important idea in functional languages is the idea of higher-order functions.

In Scheme, a function is a first-class object. This means that it can be:

  • passed as an argument to a function
  • returned as a result from another function
  • created dynamically, or at runtime

A function is a higher-order function if it does one of two things:

  • it takes another function as an argument
  • it returns another function as a result

note: in C (or at least, the standard), functions adhere to all of the above, except for dynamic creation. Functions must be defined ahead of time.

map is a function that does a few things. It takes a function as its first argument. Its successive arguments must be lists of equal length, with the number of lists equal to the number of arguments of the function. The function is then applied to all sets of corresponding elements, in order.

Some examples are:

> (define (square x) (* x x))
> (map square '(1 2 3 4 5))
(1 4 9 16 25)

You can see here that it has applied square to each element of the list.

> (map + '(1 2 3) '(4 5 6))
(5 7 9)

Here, map has applied the + function to each corresponding element of the two lists.

We can also define the function in-place (that is, as an anonymous function):

> (map 
	 (lambda (x) (* 2 x))
	 '(1 2 3)
)
(2 4 6)

This is signifcant in that it demonstrates that we can generate functions dynamically at runtime.

conj, or "conjuction", is the logical AND. (Disjunction is the logical OR).

It takes a function F and a list L and returns the conjunction of the results of applying F to elements in L.

This tells you that F must be a predicate, such that it returns true or false so that we can perform a logical AND across all the results.

For example:

> (conj number? '(3 #t 4 '(1 2))) ; Are all elements numbers?
#f

We may define it as shown:

(define (conj f L)
	(cond 
		((null? L) #t) ; N1
		(else
			(and
				(f (car L))
				(conj f (cdr L))
			)
		)
	)
)

Here, our approach is as follows:

  • If the list is empty, then simply return true.
  • If the list is not empty, then perform a logical AND on:
    • applying the function f on the head
    • applying conj on the current function and the tail

N1: You may be wondering why we want to return true if the list is empty. This is our base case, and the significance is that if we were to put #f instead, this function would never return #t. The final recursive call would return false, thus causing the parent function to evaluate the logical AND to always return false. In turn, every parent caller returns false.

This is why it must be true. If we wanted to do a disjunction, we would instead use the logical OR in our recursive call and let #f be in the base case.
