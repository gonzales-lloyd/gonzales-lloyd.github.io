Lecture 15

Data types are useful since it allows the compiler to know how to handle operations ahead of time, like memory allocation, calling constructors, and performing basic operations.

For example, in the following examples, types are helpful:

  • a+b:
  • var p ^integer;
  • new (p);

These examples encompass things like:

  • integer vs. floating-point addition
  • being able to allocate the "right size" in memory, based on the pointer type
  • calling the appropriate constructor in C++

A type system consists of:

  • a mechnaism for defining types (syntax, semantics)
  • rules for:
    • type equivalence (when are the types of two values the same?)
    • type compatibility (when can a value of type A be used in a context that expects type B?) - We can think of this as "relaxed" type equivalence.
    • type inference (what is the type of an expression, given the types of the operands?)

Type checking is a process through which we ensure that a program obeys the compatibility rules. A type clash is a violation of these compatibility rules.

There are several approaches to typing. Note that languages are usually not black and white with respect to typing, but some languages are clearly more strongly typed than others.

  • strong typing - the language prevents you from an applying an operation to data for which it is not appropriate. Weak typing is the opposite - such languages usually apply coercion to get types to work.
  • static typing - the language has strong typing (usually) and does type checking at compile time
  • dynamic typing - type checking is done at runtime

Notably, C is a static weak typed language. The type checking is done at compile time, but has a lot of coercions everywhere; you can use almost every fundamental type as a boolean, you can implicitly cast things like short and single upwards, and so on.

Java is both a dynamic and statically typed language, with strong typing.

Scheme has strong typing (no coercions, won't do things like implicitly making lists out of atoms), but is dynamically typed.

There are several ways to think about types:

  • Denotational - the definition of types as a collection of values over a certain domain
  • Constructive - often implementation-level, where you define the internal structure of data down to the level of fundamental types
    • built-ins like integers, characters, and reals are fundamentals; also called primitive of predefined types
    • these allow us to build composite types, like arrays, records, sets, and so on
  • Abstraction - interface specifying the operations that can be applied to objects

Denotational is a formal way of thinking; constructive is widely used; and abstraction is an object-oriented way of thinking.

Types can be classified into two categories:

  • discrete (ordinal) types - types that have a countable domain. These have well-defined notions of predecessors and successors (in terms of promotion),
  • non-discrete - types with a non-countable (or continuous) domain, like reals, rationals, and complex. This includes floating points.

Why are floating-points non-discrete, even though there are a finite number of possible floating point values (since they only take up a finite number of bits)? The reason is because if you increase the width of an integer, you're not going to be able to represent more numbers in between two specific values, like 3 and 5. But if you increase the width of your floating point type, you can represent far more values in between 3 and 5.

There are also scalar (simple) types, where each value conceptually corresponds to some specific number, whether discrete or non-discrete.

We also have composite types, like arrays, records, sets, and lists, which have an internal structure defined in temrs of simpler types.

Scalar types
...

Examples include:

Booleans are typically implemented over one bytes, where 1 is true and 0 is false.

Characters are one byte (for ASCII encoding) or two bytes ("wide characters"), often used in Java's Unicode character set.

Integers tend to vary in length by implementation and language. There are also signed and unsigned varieties.

Fixed-point representation is where the fractional point is at a fixed, known position. For example, if we have the format xxxx.xxxxx, we might say that the first four bits are always the non-fractional part, while the last five bits are always the fractional part (with negative powers of 2). Currency is a fixed-point value; there's only two fractional values ever to the right.

Floating-point representation, or reals, are usually split into a sign bit, the mantissa, and the exponent. Usually, we express this as
(assuming that excess-127 is not in use against the exponent).

Complex numbers are usually a pair of floating point numbers, composed of the real and imaginary parts.

Rational numbers are usually a pair of integers (since the mathematical definition is that they can be expressed as a ratio of two integers).

Enumeration types
...

These are not defined in all languages, but they do exist. Enumeration types were introduced in Pascal.

Suppose we wanted the type weekday to be an enumeration with the values {sunday, monday, tuesday, ..., saturday}.

Such types support things like:

  • ordered comparisons, like the idea that monday < tuesday
  • the idea of predecessors and successors, like the idea that tomorrow := successor(today)
  • enumerated loops, like for today := monday to friday {begin...end}
  • indexing in arrays, like var daily_attendance: array[weekday] of integer and then doing daily_attendance[mon] := 40
    • Worth noting is that in C, indexes are always integers. But strictly speaking, in enumerations, you can index into enumeration arrays using any ordinal/discrete type.
  • have ordinal values, such that ord(sun) == 1

Subrange types
...

Subranges are a contiguous subset of values from some discrete base type. These were also introduced in Pascal:

type test_score = 0..100;
	workday= mon..fri;

Often, you might use integers to represent this - that's four bytes. But you only need 7 bits to represent a test score; obviously most machines aren't happy with that, so you'll probably end up being able to use one byte instead.

Subranges are useful over full integers because:

  • they're easier to read and document programs
  • we can use semantic checks to ensure values are within range
  • they have efficient representations, where test_score can be represented on a single byte instead of four

Record (structures)

If you want to create a new type in C, you usually use typedef:

typedef struct{
	int i;
	float f;
} A;

Then, what is the type of A? This is the cartesian product of - that is, the set of all pairs containing an element from the set of all ints and an element from the set of all floats.

Variant records
...

Unions are very similar:

typedef union{
	int i;
	float f;
} B;

Just like in regular set notation, unions are helpful when you know that you need either an int or a float at a given moment, but never both - that is, they're mutually exclusive. The size of B is simply the size of the largest underlying type, and its type is .

Arrays
...

Suppose we have the following:

typedef 
	float C[10];

What is C's type? In C/C++, we usually use square brackets. But other languages use round parentheses. It turns out that at a high level, they're both the same - C is a function defined as the type of all possible index values being mapped to the elements contained in the array. In this case, the type of C is the mapping .

The actual types
...

There are quite a few composite types, including:

  • Records, introduced in Cobol, are a collection of fields with (potentially different) simpler types. Defined as the Cartesian product of field types.
  • Variant records, where only one field is valid at any moment. A variant record's type is a union of several field types.
  • Arrays, where components have the same type and are a function mapping an index type to a component type
  • Sets, a collection of distinct elements of a base type
  • Pointers, a reference to an object of the pointer's base type
  • Lists, a sequence of elements with no indexing; implemented as linked lists
  • Files, which have a notion of "current position" and are usually accessed sequentially

Type equivalence
...

Remember that type equivalence is how we answer the question "if i have two (different) types, when should i consider them the same thing?"

There are two ways to define type equivalence:

  • Structural equivalence - same components, put together in the same way
  • Name equivalence - each definition introduces a new type

First, let's look at this:

type T1 = record
	a, b : integer
	end;

type T2 = record
	a : integer
	b : integer
	end;

type T3 = record
	b : integer
	a : integer

Which ones are equivalent to each other? We would generally agree that T1 and T2 are always equivalent; they're only visually different because of how we've written it.

In most languages, all three types are equivalent - it's just a type of two ints. But in some languages like ML, T3 is not the same as T1/T2 because of how b comes before a, despite the fact we're still representing two integers in order.

Structural equivalence
...

This is an implementation-level way of thinking about types. Consider the following code:

type student = record
	name, address : string
	age : integer
type school = record
	name, address: string
	age : integer

x: student
y: school

x := y

Should this raise an error? Probably (in semantic terms), but strictly speaking, they have the same structure - two strings and an integer.

Name equivalence
...

To address the problem above, people invented name equivalence, where the idea is that the same name should imply the same type. But there exists a problem when handling type aliases:

TYPE celsius_temp = REAL;
	fahrenheit_temp = REAL;
VAR c : celsius_temp;
	f: farhenheit_temp;

f := c

Should you be allowed to do f := c? They're aliased for the same thing, yes, but you semantically shouldn't do that.

But if we have

type college = school

from the earlier example, we should probably allow that - it makes reasonable sense.

There are variants to name equivalence:

  • strict name equivalence, where aliased types are considered distinct
  • loose name equivalence, where aliased types are considered equivalent

Most languages use loose name equivalence, like Pascal. Ada allows the programmer to decide whether an alias is a derived type or a subtype, as in the case:


type celsius_temp is new real;
type fahrenheit_temp is new real;

where subtypes are equivalent to their parent/base type, and to its sibling types. Derived types are new types entirely.

Conversions
...

As a recap, consider that we can treat type equivalence in a few ways:

  • structural equivalence
  • name equivalence
    • strict name equivalence
    • loose name equivalence

and all of these can be expressed in terms of conversions, or casts.

For type compatibility, which is "weaker" type equivalence:

  • If two types are compatible, we do "nothing"; we do a silent conversion from one type to another, known as coercion as defined by the language.

So let's briefly look at type conversions and casts.

a := expr
a + b
f(arg1, arg2, ..., argN)

Assuming that the type provided and the type expected are required to be the same, we need to use casts, or explicit conversions. Does this involve additional code at runtime?

There are several cases:

  • The types would be structurally equivalent, but the language uses name equivalence
    • the types use the same representation - they have the same range and width
    • so we apply conceptual conversion, which involves no additional code
  • The types have different sets of values, but the same representation
    • like a one-byte int against a four-byte int; different range (and also different width), but same representation method for the same values
    • addresses subranges
    • If the provided type has some values that the expected type does not, then we only need to perform checks at runtime, but not any conversion of representatoin
  • The types have different representations entirely
    • conversion of representations are required
    • checks are sometimes required for things like overflows, etc.

Then consider the following example in Ada:

n: integer;
r: real;
t: test_score; -- 0..100
c: celsius_temp;  real

t := test_score(n)
n := integer(t)
r := real(n)
n := integer(r)
r := real(c)
c := celsius_temp(r)

What do we need to check at each assignment?

  • a runtime check; what if n is outside of the range of test_score?
  • nothing
  • a conversion (from integer to real)
  • a conversion and check (from real to integer - the range of a real may be larger than an integer)
  • nothing (real to real)
  • nothing (real to real)

Non-converting type cast
...

Finally, we have the non-converting type cast. What if we want to change the type without altering the underlying bits; that is, to interpret the bits of a value as if they were of another type?

Examples include:

  • memory allocation algorithms; the heap is a very large array of characters/bytes, so we might want to reinterpret portions of it as various data structures
  • high-performance numeric computations; we may want to reinterpret a float as an integer or record, extracting the sign, mantissa, and exponent

The code

int n = 5;
float f;

f = (float)n;

does not work, because the underlying bits will change as we cast it to a float. But if we instead do

f = *(*float)(&n)
then we:

  • take the address of n, yielding an *int
  • reinterpret *int to *float
  • Dereference the pointer to float
