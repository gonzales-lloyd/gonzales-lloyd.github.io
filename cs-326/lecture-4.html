Programming languages
...

In essence, we start with a design phase, in which we determine what the language will look like using only regular expressions (to describe tokens) and context-free grammars (to describe how those tokens should be put together). This is our generator.

Then, we need to implement the programming language - something that checks if a given input from the generator is valid or not. This is our acceptor, which performs the actual compilation. Then, we would want to implement a scanner that takes in our regular expressions, and a parser that takes in our context-free grammars.

Do note that scanners do exist in outside the scope of programming language implementation. Whenever we define a file format structure that we want to read in, like with freadf(), we've essentially defined a scanner with some specific structure that the file should adhere to.

Moving onto functionality; the scanner's main goal is to recognize tokens described by regular expressions. We also ignore some things, like whitespace and comments. This is implemented as a function that returns the next token every time it is called.

The parser is what calls the scanner to obtain tokens, building a parse tree from it.

But this actually implies we have two more languages, other than the programming language itself. This is the language taken in by the parser, and the language taken in by the scanner. The language accepted by the scanner and described by the regular expressions is the set of all valid tokens in the given programming language.

Then, the language accepted the parser is the set of all tokens following the syntax (or grammar) - that is, the set of all valid programs. So note the differences between the the programming language itself and the additional formal languages for the parser and the scanner.

Scanner
...

The scanner is implemented as one of three options:

  • an ad-hoc program
  • explicit finite automaton
  • table-driven automaton

An ad-hoc program is simply the minimum that works, often difficult to extend and maintain. It's cobbled together and works.

In finite automaton, we have a set of states , where is the initial state and some are the final state(s). Then, we have a set of inputs as well as transitions - the mapping of states and inputs to other states. Formally, that's . In other words, we read the first state and the first input, then the second state and the second input, and so on and so forth. A state diagram is sufficient to describe it. (Note that states and inputs are vectors, so represents the Cartesian product.)

Note that accepting states are not necessarily "ending" states in the sense that successive inputs are ignored; simply that they accept they input in some form. They may be ending states in which the parser calls the scanner again for more tokens, in which case we begin at the initial state again.

Let us consider a more complex example. Here, we'll look at Pascal:

Let's describe our state machine:

  • Start at some initial state A. If we observe irrelevant whitespace, return to state A.
  • If we see a colon, move to some other state. This is a final accepting state because the colon is a valid token in itself, but observe that there are two cases in which the colon has been used. One use is as a separator betweeen the variable name and its type; another use is as an assignment. Then, if we see an equal sign immediately after, we observe an assignment.
  • If we see a less than sign, move to a "less than" state. If an equal succeeds it, move to a "less than or equal to" state.
  • If we see a letter, immediately treat it as an identifier. Succeeding characters can be letters, digits, or underscores, in which case the identifier can extend indefinitely.

In short, you have a lex or flex that takes in a regular expression and spits out C code that implements the scanner. Knowing the regular expression is sufficient to know how to describe the scanner.

Parser
...

Given any context-free grammar, we can always create a parser that runs in time, where is the number of tokens. is unacceptable for most cases, but there are some classes of grammars where the parser for it runs in time instead. These are:

  • LL (left-to-right, leftmost derivation)
  • LR (left-to-right, rightmost derivation)

LL parsers are referred to as top-down/predictive parsers, while LR parsers are referred to as bottom-up/shift-reduce parsers.

Let's say I want to describe something that has inputs looking like A, B, C; - that is, a list terminated by a semicolon. We can define such a grammar as

  • list -> id list_tail
  • list_tail -> , id list_tail | ;

Let's think about how we would make a top-down parser with this grammar. It's referred to as predictive because we try to predict what we'll expand next based on the input. So we start with the list token. We can reliably predict that we'll expand it into id list_tail, so we perform such an expansion, immediately letting id be A, and then moving onto list_tail. We look at the next string, and we observe a rule that expands one-to-one with the following characters - B,.

So, we can expand this list_tail into ,, id (B), and then list_tail. Again, we see a substring that adheres to the start string of one of our rules, so we expand it based on that, and keep going.

Bottom-up parsers are different in that we keep reading from the input until we see a right side of a rule completely in place. Instead of expanding it, we contract it into what's on the left (the parent token).

So beginning with A, we see no rule that fits our description. It turns out that we need to read in the entire string - A, B, C; before we see an entire rule fit by the string, which is list_tail -> ;.

So we start with list_tail, which expands into ;. With the semicolon known as list_tail, we observe that a full rule is described by the current end of the string, which is effectively A, B, C list_tail; this fits the first rule of a list_tail, so we can add an id and a , to our list_tail to form another list_tail one level up. So this whole thing becomes its own list_tail, which continues until we exhaust the string.

As you can see, we start with some lowest-level terminal and gradually move upwards, looking for parts of the string that fit a rule (rather than starting with nonterminals).

It's referred to as shift-reduce, noting that we kept reading until we could recognize a rule in its entirety. This is referred to as a shift. Once we've identified a rule and replaced it with a token (like the combination of ,B list_tail), that's refererred to as reduction.

Warning

"Token" is used loosely here to refer to the actually-correct name "symbol", which is the left side of the entirety of a rule.

Likewise, "rule" refers to the thing to the right side of an arrow in these notes, but a "rule", formally speaking, is the entire thing - symbol and right side of arrow included.

Obviously, we would prefer to avoid reading the entire string just to start parsing. This is the fault of the grammar. So if we instead defined our grammar such that we see an id on its own in a rule, we would get better results since our recursion occurs on the left rather than the right:

  • list -> list_prefix ;
  • list_prefix -> list_prefix, id
  • list_prefix -> id

As with before, automatic parser generators exist. Similar to lex and flex, we have yacc ("yet another compiler" and bison. We provide a description of the language as a context-free grammar to yacc, and it spits out C code that implements the parser. This is integrated into lex/flex.
