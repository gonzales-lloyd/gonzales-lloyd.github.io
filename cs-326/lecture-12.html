Lecture 12

Recall sequencing is where you execute one instruction after another. But it may not be that simple:

(define (f x)
	(+ x 1)
	(+ x 2)  
)

Remember that in Scheme, most expressions are not imperative (have no side effects) - (+ x 1) returns the result of adding 1 to x, but does not affect x itself. But also, remember that only the last expression is returned in Scheme. Thus, removing (+ x 1) does not have change the behavior of f() at all; in the absence of side effects, sequencing doesn't matter at all.

But suppose we have instead

(define (f x)
	(display x)
	(+ x 2)
)

Then, (display x) has a side effect; if we were to remove it, the function would still return the same output, but it wouldn't behave the same.

That said, the lack of side effects in Scheme makes sequencing typically irrelevant in Scheme.

The lack of side effects ensures that functions are idempotent - that is, you can call a function repeatedly with the same parameters, and get the same result each time. The moment when a function is called does not affect the surrounding code; that is, a function is separate from its environment.

For example:

a = f(b);
c = d;

and

c = d;
a = f(b);

might leave us in the same place afterwards, if and only if:

  • f(b) does not depend on/access c
  • f(b) does not modify d

Expressions vs. statements
...

Expressions generate a value. Statements only have side effects, typically through assignments.

In turn, we have statement-oriented languages, which:

  • distinguish between statements and expressions
  • have statements as the building blocks of programs
  • have variables assigned values generated by expressions
  • are designed such that what a statement returns is not important or design

This is the case in most imperative languages. Expression-oriented languages tend to not have side effects.

Consider this in Algol-68:

begin
	a := if b < c then d else e;
	x := begin f(y); g(z) end;
	g(d);
	p := q := r
	2+3;
end

This illustrates a few things in each line:

  • The entirety of an if statement has been encased in an expression, returning either d or e and assigning it to a
  • As in Scheme, we return the last expression, but we evaluate the entirety of f(y) even though only the result of g(z) is returned. This is significant if f(y) has side effects.
  • g(d)'s return value is not used - it is (almost certainly) a statement with side effects.
  • In Algol, we have right-to-left associativity, so that q := r and then p is set to the result of that.
  • Clearly 2+3 has no side effects, but it is still evaluated nonetheless; and because it is the last expression in the function, 5 is returned by this function.

An expression is either a simple constant or variable, or an operateor applied on a collection of operands, each of which being an expression. This results in a value.

Operators can be the usual mathematical operators or function names; operands can be simple objects (constants or variables), or expressions.

Expressions in "pure" form, in "pure" functional languages, have no side effects. This is referential transparency.

Precedence and associativity
...

Some operators are grouped "more tightly" than others in the absence of parentheses, such that you evaluate them first. This is precendence.

The associativity of an operator is a property that determines how operators of the same precedence are grouped (in the absence of parentheses).

Suppose we have 1 + 2 + 3 in infix. This is + + 1 2 3 in prefix notation. Remember that in evaluating prefix notation, we don't attempt to evaluate anything until we get two operands in succession.

Now, suppose we have 9 - 3 * 2 + 1. How do we express this in infix? The answer is + - 9 * 3 2 1. How do we get there?

  • The first thing we notice is that 3 * 2 happens first, which gives us * 3 2.
  • Then, we do 9 - (3 * 2) - since * 3 2 gives us a result, we want - 9 * 3 2.
  • Finally, we want to add 1, and since the expression - 9 * 3 2 gives us something back, we place the operand 1 after it and the operator + before it to give us + - 9 * 3 2 1.

Suppose we have 4 * (1 + 2). Then we have * 4 + 1 2.

Then, suppose we have 4 * 1 + 2. Then we have + * 4 1 2.

Notice that there are no parentheses in our prefix expression, because "order of operations" is never ambiguous; you can just blindly apply operators to operands once you're able to. But in infix notation, we need parentheses.

But there is something significant to these expressions - each of these have a known number of operands (exactly 2) in this case. When we have * 4 + 1 2:

  • I see the *, so I want two operands.
  • I see a 4, so I still need another operand.
  • I see a +, which needs two operands in itself (before the *).
  • I see the 1 and the 2, so I can go ahead and add those together.
  • Now I have two operands for the multiplication, so I can go ahead.

The idea of precendence and associativity is irrelevant in the case of prefix (and postfix) notation - it only exists in infix. But we can only ignore these if we know how many operands we're looking for.

Assignments
...

A variable depends on the context in which it appears.

What is the meaning of a in the following?

d = a;
a = b + c;

Notice that on line 1, we care about the value of a so we can put it into d. But on line 2, we don't really care about what a currently is; we just need its address so we can put the result of b+c where it is.

l-values and r-values
...

Expressions that denote locations are often denoted as l-values, which are the things that tend to appear on the left sides of assignments. Expressions that denote values are often denoted r-values, which tend to appear on the right side of assignments.

  • a is both an r-value and an l-value
  • 5+1 is just an r-value; you can't say (in C) 5 + 1 = Z
  • NULL is just an r-value, because it's a constant
  • b[i] is both an r-value and l-value
  • p->s is both

f(a) is interesting. f(a) can be an r-value, but it may not return something; if f(a) is declared void, then you wouldn't do x = f(a). But if f(a) returns a valid value, it's an r-value.

But also, it generally doesn't have an l-value. But if f(a) returns an address (a pointer), then it is possible to have f(a) be an l-value, such that it can be the target of an assignment. So:

  • f(a) is an r-value except when declared void
  • f(a) is an l-value only when returning an address

The value model vs the reference model
...

Now, consider the code

b := 2;
c := b;
a := b + c

The value model for variables (on the left) is where we have a named container for a value. This is the case in C, where we have an independent 4 in a, an independent 2 in b, and an independent 2 in c.

But the reference model is where a variable is a named reference (pointer) to a value, as in the case of Scheme. This is where b and c are pointers to the same value 2, and a is a pointer to the value 4. It is important to distinguish between:

  • variables referring to the same object (eq?)
  • variables referring to different objects of identical values (equal?)

Initialization
...

Consider the difference between assignment and initialization. What is the difference between int a = 1; and int a; a = 1;?

Suppose that a is statically allocated, like a global variable - then there's a difference. In the first case, the value 1 is literally placed in some static location in the binary, so that it is assigned at compile time, so to speak.

In the second case, the value is uninitialized, so a is some garbage value until assigned by a = 1 at runtime.

But now suppose we put these into functions:

void f(int x){
	int a = 1;
}

void f (int x){
	int a;
	a = 1;
}

Then, both variables will be declared on the stack at runtime, and it just so happens that the effect is the same - the value 1 is placed at runtime for both.

Initialization is also useful to prevent the use of variables before they are assigned a value, so that we initialize them when they are first declared.

So the question is - can initialization be done automatially (with some default value) when the programmer doesn't do it? The answer is that it depends.

It's always possible to do, but it's not always done. If we have a static variable like a global, the compiler just has to stick a value at compile time. In C, they are guaranteed to be filled with zeros, unless otherwise initialized.

For dynamically allcoated variables on the stack or the heap, we can't place values ahead of time (at compile time), so we would have to do it at runtime; this is expensive, and not necessarily useful.

Another question: can the use of uninitialized variables be detected at runtime, a dynamic semantic error? In other words - is there a way to reliably determine if something is initialized or just garbage/

For uninitialized floating-points, we usually use NaN as defined by the standard (which then leads to error signaling if any operation is done on NaN); for all other types, we would instead need to have some sort of flag to note that a variable has not been initialized yet. That costs extra space, so it's not often done.

Combination assignment
...

Suppose we have

a = a + 1
b.c[3].d = b.c[3].d * e

This is hard to write, hard to read, and is prone to mistakes. Notice that you also have to write the whole thing twice.

But a more fundamental reason is that you have to access the structures (or the variables) twice and follow them around; you have to go to b, find c, compute the address 3 elements forward, and then find d. Then you do that again.

But now suppose you have a[f(x)] = a[f(x)] + 1. One issue is that you're unnecessarily computing f(x) twice, if it's idempotent; but if f(x) has side effects, then those side effects happen twice (and f(x) might not even return the same thing). Instead, we might use j = f(x) and index by j.

But rather, we have combination assignment oeprators in c, so that we instead have

a += 1
b.c[3].d *= e

and in addition, C also provides prefix and postfix increment and decrement operators:

  • ++i increments i, then returns the newly incremented value (i+1)
  • i++ returns i then increments it

Finally, we also have pointer arithmetic in C. If we have int* p and p += 3, then the pointer is moved three memory locations forward. More accurately, it is incremented by 3 * sizeof(int); if int is 4 bytes, then we move forward 12 bytes.

Ordering within expressions
...

Note that precedence and associativity defines the order in which operators are applied, but not the order of evaluating operands. Suppose you have

a - f(b) - c*d
g(a, f(b), c*d)

The most common form is to evaluate left-to-right. But this may not necessarily be true; is f(b) or c*d evaluated first? And what if c or d is changed by f(b)? This is why evaluation order is important. It's fine to leave the order unspecified if there are no side effects; else, you should explicitly state the order.

Short-circuit evaluation
...

Short-circuit evaluation is a generic way to say "evaluate only as much as needed if we need to compute the value of an expression". This is used in the evaluation of Boolean expressions.

Why do we do this? One reason might be to save time, like not bothering to evaluate a really large OR expression if we already know one is true.

But another is if we want to have logic in our Boolean expressions:

if (b != 0 && a/b == c){
	// Only try dividing if b != 0.
	// This avoids a division by zero error.
}

if(p && p->foo){
	// Only dereference p if p is not NULL.
}

if(i >= 0 && i < N_MAX && A[i] > x){
	// Only try to access A[i] if i is a valid index.
	// Avoids out-of-bounds memory accessing.
}

Note that languages may or may not have short-circuting. Some languages only have short-circuiting, while other languages don't, or offer both as an option.

Note that with or without short-circuiting, the result of your Boolean expression is always the same. But with short-circuiting, the behavior of your conditionals may or may not be the same, especially if you have side effects when calculating your conditionals.

Structured and unstructured flow
...

In assembly language, we manage control flow by jumping around with conditional and unconditional jumps.

Early imperative languages, like Fortran and Cobol, mimic this with labels and goto statements.

Then, in the late 1960s and 1970s, there was a debate on the merits and evils of goto. Today, we know that goto is gnerally considered harmful. Despite this, some languages still allow goto in some contexts; Ada allows goto in limited contexts, C++ still has goto for compatibility, and Java doesn't allow it but still has it as a keyword.

The removal of goto led to lexically nested blocks, selection through if statements, and iteration with for and while loops. But goto is still useful when we have mid-loop exits, like wanting to exit a loop early - most modern languages use break
