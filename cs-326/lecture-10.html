Lecture 10

Last time, we discussed dealing with fragmentation.

Specifically, we will deal with external fragmentation. External fragementation is more or less inevitable, because of how you will naturally allocate and deallcoate memory at various spots, and at some point you'll get Swiss-cheese looking allocated memory.

One way is to alleviate this is through storage compaction, which is what the disk defragmentation utility does. It rearranges allocated blocks of memory so that they are adjacent to each other, filling in the previously empty space.

The issue is that when we're doing this with internal memory (not external memory!), such as with the new keyword, we also need to update any pointers that refer to rearranged memory. Otherwise, the pointers will end up pointing to the wrong place. So you need to keep track of your pointers.

One way to implement this is to maintain a single linked list of free blocks. When you get a request for a block, you choose from one of those free blocks. There are several strategies for selection:

  • first fit, which is where you simply choose the first one that is large enough to fit the request. Of course, the issue with this is that it produces internal fragmentation - blocks tend to be larger than whatever you need.
  • best fit, where you choose the block that is large enough for the request and is the smallest that satisfies the request. But this can be inefficient, because it takes time to look for such a block.

A better solution is to have some sort of implementation that finds optimal sizes in time.

One option is the buddy system:

In the buddy system, you maintain pointers to linked lists with blocks of size . This means you can quickly calculate which linked list to find available blocks in.

If you find that there are no elements left in the best-fit linked list for your request, then you can go to the linked list that has blocks of size . Then, you can break it in half - one half of the list remains in place to fit the request, while the other half is added to the true subarray. Then, in the future, if we want to remerge these blocks together, we will try and find the block's buddy (highlighted in blue) in the lower-level linked list. If it is available, we recombine the block.

Deallocation
...

When we use explicit heap-based allocation, deallocation is also explicit. The main advantage is that it is very fast - you tell the computer exactly what you want to deallocate. But that's your problem as the programmer to figure out, and you can make mistakes.

In implicit heap-based allocation, a garbage collector is often used. This is done automatically, but tends to be slower and requires more overhead.

Scope rules
...

A summary

Dynamic scoping means that the scope of a binding is based on the most recently-active binding; static scoping is determined by simply inspecting the code at compile time.

Shallow binding is where we use the reference environment at a function call; deep binding is where we use the reference environment when a function is returned, stored, or passed. Binding type and scoping type are distinct and combineable.

We have static scoping, or statically (lexically) scoped bindings. This is where the scope for a binding is determined by examining program code at compile time.

The other is dynamic scoping. Here, scopes depend on the flow of control. The scopes will depend on the flow of control at runtime - that is, who calls what and in what order.

Static scoping
...

In static scoping, consider nested scopes. recover

There is also the closest nested scope rule. This determines what the scope is for a certain binding. For example:

  • Suppose that we know a name introduced in a declaration. This happens:
    - in the scope where it has been declared
    - in each internally nested scope, unless shadowed/hidden by another declaration of the same name

Another way is to find the object referenced by a name. Suppose you are the compiler, and you see y = x + 1. You will then want to find what the declaration for x is. So if you're in a certain scope and you want to find the declaration, what do you do? You look for an x in your current scope, and if you don't find it, you go to outer scopes.

  • Look for a declaration in the current scope.
  • If it's not there, look in the immediate outer scope. An important thing to note is that you don't go into scopes at all; you can't recurse into an inner scope different from your own.

If you have:

  • P1
    • P2
      • P3
    • P4
      • F1

P2 has no knowledge of F1, even though it has knowledge of P4.

Static chains
...

Suppose you have a something where your execution order is A - E - B - D - C, so that the stack contains frames for each function and C is on the top.

Suppose you have function A which has some local variable x. It is supposed to be available in C, which is nested in A. On the stack, x is located in A. So how does C access x?

Suppose (in the sense of nested scopes - not calling order!), we have:

  • A (which contains x)
    • B
      • C
      • D
    • E

One way to do this is to have pointers to the calling function on the stack.

Now, it's true that C could find x by following the parent pointers twice in a linear search. But the compiler is aware that C is nested two levels below A, so the compiler will generate code that automatically moves two pointers forward. The same would apply if our calling order was instead A - B - C.

Dynamic scoping
...

The rule for dynamic scoping is fairly simple: to choose a binding for a name, choose the most recent active binding.

What's the difference between this and static scoping? Consider the following code:

a: int

procedure first
	a = 1

procedure second
	a: int
	first()

// main (effectively)
a = 2
second()
write(a) // print the value of a

In static scoping, we can see that the a in the first procedure is the global a. Likewise, the references to a in the main function refer to the global a.

Under static scoping:

  • the global a becomes 2 in main
  • second is called
  • first is called
  • a in first refers to the global, so the global a becomes 1
  • write(a) refers to the global, so 1 is printed

Dynamic scoping is different in that we don't make any assumptions as to what a given reference to a refers to until the moment we need it. Instead, we have:

  • the global a becomes 2
  • second is called
  • a local a is declared and becomes the most recent a
  • first is called
    • when we look for a in first, we find that the local a in first has NOT yet been deallocated since second has not yet returned, so the local a is the most recent active scope
  • the local a becomes 1
  • second returns, and then the local a is deallocated
  • write(a) is called; since the local a is now inactive, the global a is now the most recent active binding, so 2 is printed

Binding rules
...

A referencing environment is the set of all active bindings at a given moment at runtime.

There exists a special issue whenever a function is:

  • passed as an argument
  • returned as a result
  • stored (somewhere)

and then is later called.

So pretend that throughout the course of your function, you pass/return/store it and then call it.

How do you figure out the scopes for the bound function? Which referencing environment should you use - the scopes available to you at the moment you call the bound function, or the scopes available when the function was passed/returned/stored in the past?

When we use the referencing environment from the moment of the function call, this is known as shallow binding.

When we use the referncing environment from the moment that the funciton was passed, returned, or stored, that's called deep binding.

Suppose we have the following example:

procedure C; begin end;

procedure A (P: procedure; i: integer);
	procedure B;
	begin B
	write(i);
	end B;
begin A
	if i = 1 then A(B, 2)
	else P;
end A;

begin main
	A(C, 1);
end main

Then, in terms of call order, we have:

  • A(C,1) in main
  • A(B, 2) in A(C, 1)
  • B in A(B, 2)
  • write(i) in B

If we have shallow binding, then we would use the referencing environment when B is called, so that write(i) gives us i = 2.

If we have deep binding, then we would use the reference environment when B was originally passed as an argument, which is in the body of the call to A(C, 1). At that moment, i=1, so write(i) gives us 1.

If you implement shallow binding, you don't need to do any extra work - just apply your rules for static/dynamic scoping as needed.

But if you implement deep binding, you need to somehow save the environment in the past so you can use it later at some unknown time. This is known as a closure, a saved reference moment together with the function being passed. These are placed together, and the closure is passed as an entire argument to be unpacked later on.
