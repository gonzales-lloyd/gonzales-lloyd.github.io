Lecture 8

Remember that we talked about how stop-and-go is bad, but pipelining allows us to send multiple "in-flight" packets at the same time that haven't been acknowledge yet.

There are two generic forms of pipelined protocols: Go-Back-N (GBN) and selective repeat.

Go-Back-N/GBN
...

Intro
...

Suppose we have nine packets that we want to send, but we can only send four packets at a given moment. Here's what GBN looks like in action:

Observe how the receiver completely ignores any packets (in order) past the packet that it didn't get, even if the sender continues sending packets in the four-packet pipeline whenever it gets the acknowledgments. However, for the ACKs it does get, it moves the sliding window over to the next packet, moving forward in the set of 9 packets we want to send.

The important thing here to note is that the sliding window represents all of the packets that have been sent, but haven't been acknowledged yet. Everything to the right hasn't been set yet; everything to the left has been acknowledged.

After enough time has passed without getting the ACK for a given packet that is, when the timeout has elapsed we resend everything from that missing ACK, which is everything in the sliding window at that moment (starting from packet 2). When the receiver gets those packets, it sends the corresponding ACKs, and this causes the sender to shift its receiver over.

Something worth noting is that even though the receiver is getting the packets it needs, it discards the ones that fall out of order and resends the highest ACK for the highest packet it's received so far.

Implementation: sender
...

To keep track of what we can and can't send, we have a k-bit sequence number in the packet header. The value of k depends on how large your window is; in the previous example, our window had a size of 4. To achieve this, our packet needs 2 bits to represent all four packets in the packet.

Warning

The explanation in class is that the window size was 8, where the blue ones are simply not usable and only the yellow ones are part of the window. In turn, your packet header would need a 3-bit sequence number.

This makes sense since the previous slide suggests that the blue packets would be sent immediately if the window size were indeed 14 like I describe below, but that's not the case; the blue packets are explicitly the ones we haven't sent. So the window size ought to be 8.

In this example, our window size is 14, so we need 4 bits:

You don't send the blue ones because (since the last timeout) you sent 14 packets starting from the leftmost packet, which is the maximum size of the window. You then receive six ACKs, so you move the sliding window forward 6 packets the remaining 8 packets are still "in transit", while you've opted to not send the next 6 packets. (Perhaps it's because you'd have to wrap the sequence number back around not sure.)

This represents a point in time where if the timeout were to happen right now, you would resend both the yellow and the blue packets.

In this system, we usually send cumulative ACKs, where a given ACK of sequence number means that this ACK represents an ACK for every packet up to and including .

Again, when we hit the timeout, we retransmit packet and all higher sequence packets in the window.

Receiver
...

In this case, the receiver doesn't buffer at all. This means that when we get out-of-order packets, we discard these packets and don't buffer them. We re-ACK the packet with the highest in-order sequence number again whenever we get these out-of-order packets.

As far as the receiver is concerned, it only sends ACKs for correctly received packets with the highest in-order sequence number. As described before, out-of-order packets may lead to the generation of duplicate ACKs. Notice how the receiver only needs to keep track of the expected next sequence number for this to work.

Selective repeat
...

Recall that GBN had cumulative ACKs. In contrast, selective repeat is where the receiver individually acknowledges all the correctly received packets.

And also unlike GBN, the receiver buffers packets as needed to eventually deliver packets in order to an upper layer (the application layer).

In selective repeat, the sender only resounds packets for which an ACK is not received, setting up a timer for each un-ACKd packet. The sender still has a window, where it sends N consecutive sequence numbers, which limits the number of un-ACK'd packets.

Here's what that looks like in action:

Notice how the receiver sends an acknowledgement for packet 3, even though it was out of order. The sender also sends packet 4 and 5 to keep the pipeline full, for which the receiver also sends corresponding ACKs. But since it still doesn't have packet 2, it buffers packets 3, 4, and, 5 so it can eventually send the packets in order to the upper layer.

When the timeout occurs, packet 2 is resent, the receiver buffers it and recognizes it can deliver packets 2-5 in order to the application layer, and an ACK for packet 2 is sent.

Now, notice how the window at the sender behaves. When ACK0 and ACK1 are received, the window moves forward. But when ACK3 to ACK5 are received, they're recorded but don't move the window forward. This means that until ACK2 is received, the window can't possibly move forward, and guarantees that the buffer on the receiver's side will never be "overfilled" or "overrun" with more packets than it can hold before it it can deliver packets to the application in order.

When ACK2 arrives, the window is advanced all the way to packet 6.

A recap of the pipeline protocols
...

In go-back-N:

  • The sender can have up to N un-ACK'd packets in the pipeline at a time.
  • The receiver only sends one cumulative ACK, which means that it doesn't ACK packets if a long gap or delay occurs.
  • The sender has a timer for the oldest un-ACK'd packet when the timer expires, we retransmit all un-ACK'd packets.

In selective repeat:

  • The sender can have up to N un-ACK'd packets in the pipeline at a time.
  • The receiver sends an individual ACK for each packet.
  • The sender maintains a timer for each un-ACK'd packet, such that when the timer for that packet expires, only that un-ACK'd packet is retransmitted.

A problem with selective repeat
...

Now, remember that we accept packets based on a "rolling" sequence number that wraps around whenever we hit the maximum.

But what if we encode two different packets in the exact same way? If the receiving window wraps around to accept another packet with sequence number 0, but the sender hasn't gotten an ACK for the original packet 0 with sequence number 0, what will happen is that the sender will resend packet 0, but the receiver interprets this as packet 5 (since we have four bits in our sequence numbers).

Notice how the duplicate data is accepted as new in this case, which is incorrect. So what relationship do we need between the sequence size and the window size to avoid this problem? It's that the window size must be less than or equal to half of the maximum sequence number; even in the worst case, the window will never accept duplicate packets, since the window won't have "wrapped around" to the smallest resendable sequence number.

A recap of reliable data transfer mechanisms
...

We have a few ways to set up reliable data transferring:

  • acknowledgments, which tell the sender that a packet or set of packets have been received correctly
  • timers, which enable the use of timeouts to retransmit packets if needed, possibly because the packet or its ACK was lost within the channel
  • sequence numbers, which allow for sequential numbering of packets of data flowing from the sender to the receiver
  • Pipelining, to improve sender utilization over a stop-and-wait mode of operation
  • Checksums, which detect bit errors in a transmitted packet
