Lecture 19

BGP
...

BGP, or the border gateway protocol, is the de facto routing protocol for the internet. It allows subnets to advertise its existence to the rest of the internet, essentially stating "I am here".

BGP provides each autonomous system a means to:

  • Obtain subnet reachability information from neighboring autonomous systems
  • Propagate reachability information to all autonomous systems and internal routers
  • Determine "good" routes to subnets based on reachability information and policy.

That is, it tells every AS how to reach a particular host.

At a high level, pairs of routers, known as BGP peers, exchange routing information over TCP connections in what are known as BGP sessions. These sessions do not need to correspond to physical links.

In the example below, when AS2 advertises a subnet to AS1, AS2 promises it will forward datagrams toward that subnet. (eBGP = external BGP; iBGP = internal BGP).

Then, using the EBGP session between 3a and 1c, AS3 can send reachability info to AS1. This means that:

  • 1c can then use IBGP to distribute this new reachability information to all routers in AS1
  • Because 1b is a border gateway, it can also then re-advertise new reachability information to AS2 over the 1b-to-2a eBGP session.

Whenever a router learns of new information, it creates an entry in its forwarding table.

BGP is specified in RFCs 4271 (the protocol itself) and 4274 (the analysis), of which version 4 is widely in use. BGP is very popular because of its simplicity despite no providing no performance or security guarantees.

The internet consists of thousands of autonomous systems today. Each AS uses BGP to exchange routing information. Something you might notice is that BGP is an incremental protocol – a BGP-speaking router sends an announcement message whenever a new route is available.

There are four key parts to BGP:

  • IP prefixes
  • AS-PATH (a path vector protocol) and AS numbers
  • Route selection
  • Import/export policy

IP prefixes
...

Every device on the internet is known by some IP. But obviously, it's very difficult to advertise every single IP address, and that would have too much overhead. So what do we instead? Remember that address are assigned to institutions in blocks of contiguous address.

BGP takes advantage of this by advertising using CIDR; for example, it might advertise 192.168.2.0/24 and 192.168.3.0/24. Using supernetting, we can simplify this to simply mean 192.168.2.0/23; notice that these two only differ by their lowest network bit.

IP prefixing and supernetting lead to smaller routing tables and fewer route advertisements.

AS-PATH
...

The BGP announcement carries another additional item, known as AS-PATH. The prefix plus the AS-PATH effectively comprises a "route".

AS-PATH is a path vector of AS numbers, or ASNs. ICANN, or the Internet Corporation for Assigned Names and Numbers, and the ultimate authority for delegating these ASNs.

ASNs are 16-bit numbers ranging from 1 to 65535. 1-64511 are public and are used by provider ASs; 64512 - 65534 are for consumer ASes. 0 and 65535 are reserved.

Import/export policy (routing policy and route attributes)
...

ASs are not just bound by physical relationships, but also by businesses and SLA; this creates provider/consumer relationships and stub/transit AS relationships.

The "best" path chosen by a BGP router may not necessarily be the shortest route in the physical topology. There are a variety of import and export policies that must be specified using route attributes.

Now, suppose that A advertises the path AW to B, and B advertises path BAW to X. Should B then advertise path BAW to C? The answer is no; B gets no "revenue" for routing CBAW since neither W nor C are B's customers. B wants to force C to route to W via A.

What does this mean? In short, B only wants to route to and from its customers. It's true that W-A-B-C-Y exists, but B doesn't get anything out of it.

BGP message types
...

There are four message types in BGP:

  • Open message: opens a BGP communications session between peers and is the first message sent by each side after a transport protocol connection is established.
  • Update message: used to provide routing updates to other BGP systems, allowing routers to construct a consistent view of the network topology.
  • Notification Message: used to close an active session or when an error condition is detected.
  • Keep-alive message: notifies BGP peers that a device is active.

The fields are:

Subnetting
...

Recall that IP addresses are a 32-bit identifier for a host/router interface. An interface is a connection between the host/routter and a physical link.

Routers typically have multiple interfaces. The host typically has one or two interfaces, often wired Ethernet and wireless 802.11. An IP address is associated with each interface.

Subnetting allows you to divide a network up. For example, if I have the address 172.19.0.0/16, then with a subnet mask of 255.255.255.0 or /24, I can divide up my network like this:

Why would we want to subnet? We can divide a larger network into smaller networks, but we can also limit the scope of layer 2 and layer 3 broadcasts to their subnet. This also allows for better traffic management.

Notice then, that we can have both a network and subnet mask:

That is, applying a mask that is larger than the default network mask will allow you to divide your network into subnets, where the bottom bits are your subnet hosts and the middle bits refer to the subnet within your fixed network bits.

In this specific scenario, using the middle 8 bits for the subnets gives us 255 subnets; the last subnet contains the broadcast address, so it can't be used.

And within each subnet, we can use 1 - 254 (since again, 0 is localhost and 255 is broadcast), which means that each subnet has 254 hosts, or 2^8 - 2.

So what are some important things to remember about subnetting?

  • You can only subnet the host portion; you don't have control of the network portion. That is, you're only dividing up the host bits.
  • Subnetting doesn't give you more hosts; it only allows you to divide your larger network into smaller networks. In fact, you'll actually lose host address, because in each subnet, you'll lose the address is broadcast address of that subnet.
    • Even more, if your subnets include reserved addresses (localhost/broadcast) you may or may not lose the first and last subnet based on your subnet mask.

Here's an example:

  • The major network address is 138.101.0.0 (so that in CIDR notation, you have 138.101.0.0/16 at the network level).
  • The major network broadcast address is 138.101.255.255.
  • The range of hosts, pretending it weren't subnetted, would be 138.101.0.1 to 138.101.255.254, eliminating the two reserved addresses in the network for broadcasting and localhost.

But what about the subnet information? To find the subnet address, perform a bit-wise AND between the give IP address and the subnet mask. The subnet address of this subnet, or "wire", is 138.101.114.192:

That is, it lives in the subnet represented by 138.101.114.192/26.

For the range of host addresses, use the network mask and divide up the network from the rest of the address. Then, use the subnet mask and divide the subnet form the hosts between the last 1 and the first 0 in the subnet mask.

That is, the lowest 6 bits identify the host within each subnet; the next 10 bits identify the subnet within the network. To figure out the first and last host, note that the subnet address itself is where the hostbits areis all 0s; the broadcast address is where the host bits are all 1s.

That means the first host is all 0s and a leading 1, and the last host is all 1s and a leading 0.

Accouting for the upper 2 bits representing the subnet address, this means that our host range goes from 193-254 with this specific IP address.

The total number of subnets is simply the number of subnet bits; in this case, it's 10, which gives us 2^10 = 1,024 total subnets.

Then, the total number of hosts per subnet is 2 to the number of host bits, minus one for the subnet address and one for the broadcast address. In this case, we have 6 host bits, which means we have 2^6 hosts per subnet, of which (2^6)-2 are useable (giving 62 hosts per subnet).
