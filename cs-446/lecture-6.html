Lecture 6

Remember that memory sharing involves quite a few individual topics:

  • Time
  • Static
  • Swap
  • Base
  • Segmentation

And every process has its own view of the address space, known as virtual memory. the allows for multiple programs to run at once, should be efficient and controlled, and has hardware support. The application can only access seeks own memory.

This is mapped to physical memory – but how? Address translation is where the hardware turns each memory access from a virtual address to the physical address; note that the OS manages memory, while the hardware actually performs the translation.

There are various possible memory sharing mechanisms:

  • Time sharing
  • Static relocation/allocation
  • Swapping
  • Base
  • Segmentation
  • Virtual memory

One option to run multiple programs at once is to simple hardcode all of the addresses. That's pretty bad. There's also static execution, where we also work from the same memory; but what if we execute the same task twice? They can't both use the same specific memory, even if it's not hardcoded.

We talked about time-sharing the CPU when we virtualize the CPU. We can do the same thing with memory, where we virtualize the memory by storing the memory state on the disk. The upside is that it's easy to understand; the downside is that it has very bad performance because of the IO stream that's involved with writing to disk.

Static relocation is an alternative method, where the OS relocates the code area whenever a new process starts. When we load the program, we relocate it to an assigned area. We adjust the pointers and globs, and set its stack pointer.

This creates a single address space without physical or virtual separation, however. This has various issues:

  • Processes aren't actually separated.
  • There's possible fragmentation, because the address space is of fixed size until the program exits.
  • Programs have to be adjusted when they're loaded, since previously allocated memory address will shift newly allocated memory addresses.

Alternatively, we can use dynamic allocation. Instead of allocating memory ahead of time, we can have the hardware relocate memory accesses as necessary. The hardware translates each memory address from the program's viewpoint into the hardware's point of view.

The term for this is interposition, which is where the hardware checks each memory access and dynamically/transparently translates the virtual address to physical address. Here, the OS handles the mapping of virtual to physical.

Swapping
...

Suppose that we have to add a process but there's not enough memory in RAM. Then, the operating memory will pick some specific process to take off based on a policy, save it to secondary storage (usually the SSD), and place the new process into RAM. This allows you to bring in information as needed at the possible expense of having to grab things from slower memory.

Memory management unit (MMU)
...

The MMU is physical hardware tasked with translating virtual memory to physical memory. It generates virtual addresses and is able to figure out where virtual addresses actually start.

How do we actually use the MMU? It's done using a base register. To translate virtual address to physical addresses, we use some offset. The offset is stored in a special base register that is controlled by the OS and is used by the MMU. Each process has a different offset in their base register depending on where processes earlier in memory stop.

The problem with this is that the processes aren't actually separated, since we don't know where they're supposed to end; we just have a starting access. If process 1 attempts to access 0x2000 in its virtual memory, it will actually access the start of process 2.

To prevent accidental overwriting, we have the base and bounds system. The base register is still the minimum address, but the bounds register is the virtual limit of the address space, expressed as either the size of the process's address space or the upper memory address of the process. (This is a design choice.)

The limit of a process address space, then, becomes the base + bounds; we use this to determine if a given memory access is valid or not. A simple implementation is:

size_t* checkBounds(base, bounds, addr){
	if(addr < bounds){
		return *(base+addr);
	}else{
		//raise segfault
		return NULL;
	}
}

This is better because it actually isolates processes (provides better security) and improves performance over just hardcoding everything (since the translation and if check are inexpensive).

But the problem is that process memory must be contiguous allocated, which can waste physical memory. Furthermore, it's difficult for processes to share information between each other because that would involve exceeding their base/bound.

Again, we can use something like a free list to help alleviate this, where we can quickly find unused regions of memory and get the size of these unused regions and where they're located.

What happens if we're out of space? One option is to simply just fail (and return NULL); the alternative is to create a small heap, having the OS increase the size allocated for that process.

But nowadays, we often use paging instead.

A recap of memory abstraction
...

We have a few options for managing memory:

  • No virtualization: the program runs on bare metal
  • Time sharing: one program at a time, for some period of time
  • Space sharing: relocate programs in one shared address space
  • Space sharing: virtual address through a segment (base only)
  • Space sharing: virtual address space through segment (base + bounds)
  • Space sharing: virtual address space through multiple segments
  • Space sharing: virtual address space through paging

We haven't talked about the last two so far, but they're actually used more often today.

Segmentation in the MMU
...

Instead of a base/bounds register pair, we can create a base/bounds memory area pair for each "segment". this allows a process to have many regions of contiguous memory mapped from virtual to physical, so that the overall process does not need to be contiguous itself. The hardware can override default segment registers, so that the programmer can specify which segment to use (which is useful when we want to load data from a code segment).

The MMU stores a segment table, which allows it to figure out what maps to where. This is great because we can now share physical address space between many processes.

However, the problem is that each segment has to have enough memory, the physical memory (overall) has to be contiguous (otherwise you'll get fragmentation), and you must explicitly or implicitly encode the target segment. This means you have to set up things ahead of time.

As another recap:

  • the base register shares the address space using a per-process offset
  • static relocation rewrites the code and data before running (as opposed to dynamic relocation)
  • segments are where we have several base + bounds pairs per process
  • base + bounds verify that an address is in the usable part of the address space, preventing overwriting
  • time sharing is where one process uses all the memory

Fragmentation
...

Internal fragmentation is where we have too much memory allocated to a process. This is often associated with paging.

External fragmentation is where the right amount of memory is provided for each process, but as each process terminates, we have an empty spot that must be field by the next process. This is often associated with segmentation. The next process might be smaller, result in chunks of unused memory at the end of a task's address space.

Paging
...

Paging allows the OS to alcoate memory in non-contiguous blocks. This limits external fragmentation, allowing the OS to grow segments as needed.

To do this, it breaks address spaces and physical memory into "pages." This assigns memory to a process based on pages – for example, process A might get 16 pages of size 20 megabytes to work with. Note that this can still cause internal fragmentation, because the page size is constant.

To do this, the OS manages two structures:

  • a free list of all free pages
  • a page table per process data structure to map virtual to physical memory

In the page table, the process virtual page number (VPN) and the offset are tracked. Then, the virtual page number can be used to index into the page table to find the physical frame where page 1 is stored.

Pages are the smallest unit we can break address spaces up into. The OS programmer decides what size that is, though it can also be allocated by the program creator, as well.

When a process is created in virtual address space, we call its pages a virtual page.

When we map the virtual address subspace to a physical address space, they are called page frames or physical pages. They both refer to pages in physical memory.

Now, remember that the memory management unit (MMU) needs to translate virtual addresses to physical addresses. Higher order bits correspond to the page number; lower order bits correspond to the page offset. The physical address and virtual address space size may be different.

For example, suppose we have 16 bits of virtual address space, and 20 bits of physical address space. Then, we might give 16 pages per process, with 256 physical pages available.

To manage this, we create a lookup table for each process, storing the pointer to the lookup table in a special register (CR3 on x86).

A page table is a data structure that maps virtual page numbers to physical frame numbers. It's a linear page table with an array of entires, and is indexed by the OS using virtual page numbers to lookup a page table entry that corresponds to a physical frame number.

However, page tables can become very large (if you have one entry per process each with a bunch of multiple pages, over many processes) – this is too large for hardware management, so it's stored in memory instead.

Here's the page table entry format:

The modified bit is also known as the dirty bit, indicating whether or not that area of memory has been modified (0/1).

Protection determines access rights (read/write/execute bitss).

"Referenced" determines if another page is using information from this page (0/1).

Here's an example of how a 64-bit address space for a process might be arranged:

If we need to translate virtual to physical addresses, it basically looks like this:

unsigned char v1[16];
void *toPhys(void* ptr){
	void *offset = ptr & (1<<12-1);
	// The page for the process is the virtual bits & (1 << (16-12) - 1)
	unsigned int idx = (unsigned int)((ptr>>12) & (1<<4-1));
	// Now get the physical offset
	return (void*)(((unsigned int)(v1[idx]<<12) | offset);
}

Paging is nice because it helps stops external fragmentation, since you can always fit a process by just taking all the next available pages. It's easy to allocate since you don't have to search for best/worst/smallest fit, and it's easy to free since when you free something, you don't need to rejoin blocks of segmented free code.

But there's a few problems:

  • We now need an additional memory reference to the page table, whence can be improved with a cache
  • It tends to cause internal fragmentation:
    • If we make the pages too small, we'll have less fragmentation but a huge page table that we have to search through really slowly (not to mention that page table takes up memory)
    • If we make pages too big, there will be far more internal fragmentation (but faster lookup times and a smaller page table)
  • Tables take up space – they can take up a lot of space, and they can also be slow to walk through.

Working set and locality
...

The program working set is the subset of code and data that is currently in use. This changes over time as the program progresses. Most memory access is confined to a few addresses, which makes the working set localized.

A process does not usually need all of the data and code to keep working. The OS might re-use memory that another process is not correlationally using; the hardware may optimize these memory accesses and translations to help seed up the process.

Note that memory accesses – which require a trap to the kernel – are expensive. The last accessed (or frequently accessed) pages go into a cache, called the translation lookaside buffer, or TLB. This reduces the number of memory accesses needed as we walk through the page table.

What is the TLB algorithm?

  • Get the virtual page number from the virtual address.
  • Check if the TLB has the virtual page number we need (TLB hit or miss).
    • If we find the virtual page, that's a hit – we can simply extract the physical frame number, concatenate the offset from the virtual address, and calculate the physical addresses we need to access.
    • If we don't find it, we take the virtual page number and look through the hardware access page table to find the corresponding translation using the MMU. This can involve the operating system and traps, which is slow. After this, we update the TLB with it; the hardware then checks the TLB again for the virtual page and proceeds with the "hit" case.

In general, TLB misses are expensive. They may be handled by either the hardware or the OS.

  • If handled by the hardware, this requires knowing where the page table is via a page table base register, and the format. The table is then searched to find the correct page table entry.
  • If handled by the OS, the OS takes over after the hardware raises an exception to pause instructions. It then jumps to the trap handler handling the TLB miss.

We can improve paging using multi-level page tables. Paging was originally introduced in a 32-bit system with a total of 4 mebibytes of memory. But with 4 MiB of metadata of the page table per process, you basically run out of memory for the OS and process itself. So how do we improve the page table access so that it can still run an OS and its processes?

Multi-level page tables
...

We can do this using page tables. First, consider the fact that most processes only need a small fraction of an allocated address space. So we can allocate data only for that small fraction.

The multi-level lookup table will improve space usage as we add levels - however, adding levels will result in more memory lookups, which are slow.

It's possible for the main memory to not be large enough to handle every processes. In turn, we can store unused pages of address space on the disk, so that the OS can reclaim memory as needed. The OS can also hand out extra memory than what is actually available, and can remove unused pages from disk as needed. However, this can result in deadlock and bad performance if we're not careful.

Page faults and swapping
...

The MMU is responsible for translating a virtual address to a physical address using the OS page tables. There is a "present" bit on each page table entry, indicating if that page is currently held in memory or not. During translations, the MMU will check this bit; if the bit isn't present, the MMU will cause a page fault.

When page faults happen, the OS must have some sort of policy in place to handle it. In other words, if a process needs access to a page, but that page isn't currently in main memory, we page fault. Then, we need some way to bring that inaccessible page into memory from either the SSD or the HDD.

When a page fault happens:

  • The MMU tells the CPU to trap to the OS
  • The page fault handler checks where (process and address) in memory it occurred, gathering the process data structures and page from the page table.
  • If the page is on disk, the OS will request to load the page and tell the scheduler to move onto another process so that it can be laodd. (If the page isn't on disk, there's not really much to do – this indicates a memory corruption, or something else irrecoverable)
  • If the page is still in memory or can be reproduced, the OS will create it and update data structures.
  • The OS re-executes the faulting instruction.

How do we bring a page from secondary memory? We have to first swap an old page out, so we reserve some space on our SSD or HDD to move pages back and forth.

But why does the MMU switch to the OS, instead of just handling it itself?

The reason is because we want to separate mechanism and policy. The OS is the policy handler for everything; the MMU is simply a mechanism for translation. The OS, then, is responsible for handling changes to the MMU.

  • This increases flexibility.
  • The OS decides what pages to allocate and replace.
  • The OS decides what information to pass onto a process.
  • This can be slow, because page faults don't occur often (since we usually reuse lots of the same pages from memory in the TLB).

Putting it all together
...

For example, on a read/write operation:

  • The CPU loads virtual addresses, which leads to an MMU operation
  • The MMU checks the TLB for this virtual address
    • If the TLB misses:
      • If we find the page table entry during the page walk, we update the TLB and keep walking
      • If we don't find the page table entry but we know it's valid (i.e. it's in secondary memory) (i.e. , we raise a page fault and swap to the OS. The OS then maps the page and returns to the process.
      • If we find out that it's invalid, we fault and swap to the OS. The OS segfaults and passes this to the program.
    • If the TLB hits:
      • The MMU is able to immediately obtain the physical address, fetch the memory location, and return to the CPU.
