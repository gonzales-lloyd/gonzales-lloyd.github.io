Lecture 7

Last time, we talked about various memory allocation methods.

One is through direct memory allocation or static relocation. This is usually reserved for hardware, and it must be contiguous.

Another way is through the use of the base register, which is slightly safer, but it doesn't ensure isolation. It also still has the requirement that process memory must be contiguous.

Then, we improved on this by implementing an additional bounds register, which improved isolation but still required contiguous process memory and made it more difficult for interprocess communication.

Segmentation, on the other hand, allows for non-contiguous allocation by splitting memory into variable size chunks. This is relatively slow because the (best/worst/first) fit algorithm requires a fair amount of searching, while chunks need to be merged into the segment list (external fragmentation). This is all handled by the compiler.

Paging is another approach, where memory is chunked into fixed size pages. This can lead to internal fragmentation. It's faster than segmentation since the free list does not need active maintaining, and there's no need to search for the "best" page they're all the same size. That said, there is some page table overhead (which takes us space), and instead of being handled by the compiler, paging is handled by the OS.

In paging, each process has its own page table for all of its associated pages. Virtual memory appears contiguous to the process, but corresponding physical frames are usually not contiguous.

The problem is that we have finite physical memory. You can only hold so many pages in memory, and it becomes harder and harder to keep an entire process in memory at once. So what we do instead is send certain pages off to secondary memory using a specific policy. If a given page is not in main memory, we raise a page fault.

Recall that memory management is comprised of the MMU (at the hardware), a page fault handler (kernel), and the external pager (the user).

Here's what that looks like:

Necessarily, all page tables must exist in main memory; otherwise, you wouldn't be able to know if something was in main memory or not.

Page tables have the following format:

  • Caching (0/1) signifies whether or not this page can be placed in the translation lookaside buffer.
  • Referenced (0/1) signifies whether or not another page is using information from this page.
  • Modified (0/1) signifies whether or not the page has been written to.
  • Protection (R/W/X) signifies access privileges.
  • Present/abseuse signifies whether or not the page is present in memory.

Paging algorithms
...

But what pages should be removed from working memory so that new pages can be brought in?

There are seven major algorithms we'll talk about.

Optimal
...

In the optimal algorithm, we label each page with the number of instructions to be executed before that page is referenced again. Then, when we need to remove a page, we remove the page with the largest label that is, the page least frequently used.

This is optimal, but it's not physically possible to implement until you've loaded the page into memory for the first time and run it. The OS does not know when the page will be referenced next, so there's no way to calculate the label number in advance.

That said, it is often used as a metric for other algorithms.

First in, first out
...

Another option is to simply swap the page at the front of the queue out that is, we throw out the oldest page all the time. It's very easy to implement and has no overhead; we don't need to keep track of the age of all the pages. We just assume that the page at the front of a queue is the oldest, and throw it out.

The problem is that this has relatively poor performance and doesn't consider frequency at all; if we have something used really frequently and it happens to be at the front of the line in not-ideal moments, it's just going to get kicked out all the time.

Not recently used (NRU)
...

Remember that we have the referenced (R, 0/1) and the modified (M, 0/1) bits. Periodically, by designer choice, we clear the referenced bit of every single page. This then indicates that the page wasn't recently referenced, which means that we can remove pages that weren't recently used.

Typically, we see pages broken into specific classes based on the states of their referenced and modified bits. Lower-class pages are removed first:

This makes sense if we have a page that isn't being written to and also isn't being read (referenced), then it's probably not too important and can probably be thrown out of memory. Note that "only referenced" is a higher class than "only modified" because being modified doesn't necessarily mean the page will be needed by something in the near future, but being referenced does.

Second chance (clock) algorithm
...

Here, we sort pages first-in-first-out, but we set the first page's referenced bit and move it to the end of the queue.

Clock page replacement
...

Clock page replacement is similar to the clock/second chance algorithm, but we add two hands. Here, we inspect the R bit of the page where the "hand" is pointing. If it's 0, we evict it; if it's 1, we set the R bit to 0 and move the hand forward.



(Note: stars/asterisks denote a set R bit for the given page table entry.)

With caching, this is indeed better than first-in-first-out.

Working set algorithm
...

The working set algorithm stores the current virtual time when a page fault occurs. Then, we go through all of the pages and check their referenced bit. If the R bit is:

  • 1, then set the page time to the current time.
  • 0, then compare the age of all the 0 R bit pages, where the smallest age means that it's the oldest; we remove the smallest age.

The MMU stores the times, in this case. Of course, the problem is that we have to check every single page, which is very expensive.

WSClock algorithm
...

The WSClock algorithm combines the working set and the clock algorithms. we start at the page currently being pointed to, checking the R bit. If R is set to:

  • 1, then we set it to 0 and advance the hand
  • 0, then we evict and replace it

Why not just use the clock algorithm, if we're not evein checking the times on the pages? It's because there's a chance we may not actually have any pages to evict on an entire pass of the clock; in that case, we have to resort to evicting the oldest page instead.

Recap
...

Design: local versus global page allocation
...

In general, we need to allocate a memory amount between competing processes. Should we use an intraprocess (local) or an interprocess (global) algorithm? Either way, if we don't allocate memory very well, we'll end up thrashing, which is where we page fault every few instructions (which is bad).

If we do local allocation, then:

  • if the working set grows, we'll have thrashing
  • If the working set shrinks, memory is wasted

Global allocation is usually better, especially if you know the stack and heap might grow, but we now have to decide how many page frames each process gets.

Remember that the older the page, the smaller its number is (since it uses the clock, and we store the virtual clock number). If we're using local allocation, then A5 is the oldest page for process A if we need to add page A6, we kick out A5.

If we're using global allocation, then we kick out B3, which is globally the oldest page, when we need to add A6 in. That is to say - local allocation means that we kick out pages looking only at a process's current pages, while global allocation involves looking at all of memory.

When dealing with allocation, we have to consider the following properties:

  • Does it evenly give equal shares to both small and large processes?
  • What happens if an instruction needs more pages than it's been allocation?
  • Are we able to dynamically update what pages are in memory?

Load control
...

Even with a good page replacement algorithm and global allocation, it's still possible for thrashing to occur. Suppose that every single process is waiting on I/O. Now what?

Suppose that a process needs more memory, but no processes have any available memory to relinquish. We could use swapping, but if the CPU-bound process is swapped, then the CPU will idle.

So the important thing here now is considering page size:

  • Small page sizes
    • Avoid space waste
  • Large page sizes
    • Smaller page table
      • Shorter load times
    • Longer to transfer equivalent sizes (you can't transfer half of a page, even if you need a tiny bit of data from the page)
    • Less TLB space (you can only fit so many large pages into a buffer, which could lead to repeated page faulting)

Instruction space and data
...

One more question is whether or not we're going to break out our address space into a separate instruction and data space:

If we keep them in the same space, it's possible for data to take pages that could have been used by instructions; the same holds the other way around.

Shared pages
...

What happens if we need to share pages between several processes? What if several users run the same program, or a signleuser runs several programs that use the same library?

If we need to ever share, it's a good idea to make pages read-only; this prevents a change by one process affecting all the other processes.

Static libraries
...

When we do static linking, object files and the instructions (and usually libraries) are named. The .o files are linked, and then the libraries are scanned. The function is not defined in the object file; it is considered an undefined external function until it is looked for in the library, then included if it is found.

Shared libraries
...

When we have shared libraries, the linker includes a stub function instead of the actual function. This is either loaded with the program or is already loaded.

Cleaning policy
...

What if we just have "dead" pages that aren't being used? What if we want to make sure we always have pages available?

The paging daemon is a background process that ensures free page frames are (always) available using a specific algorithm. By working ahead, this helps prevent thrashing.

The paging daemon is usually asleep, but occasionally wakes to inspect memory based on an OS timer. If too page frames available, it starts selecting pages to evict and writes the evicted page to disk if necessary.

The OS's role in paging with processes
...

But how does the OS get involved when we create a process, execute a process, get a page fault, or terminate a process with respect to paging?

On process creation, the OS determines how large (of a memory area it needs), creates a page table for that process, and then defines a swap area. The swap area is space allocated for pages to swap. Information about the swap area and the page table are saved to the process table.

On process execution, the MMU is reset for a new process and flushes the TLB (since the TLB entries no longer apply to the current process, so we clear it so we can quickly add new entries). The new process's page table is unadopted, and all or some pages are brought into memory to reduce immediate faulting.

When handling a page fault, we follow a fairly specific process:

  • Hardware traps to kernel, saves the program counter to stack
  • Assembly started and saves general registers and volatile information
  • The system finds the page fault and tries to discover the needed virtual page
  • Once the page is found, we check if the address is valid and allow the access
  • If the selected frame is dirty, schedule for transfer to disk, context switch, and temporary suspend faulting
  • After the frame is clean, the OS looks up the secondary memory address (disk address) where the page is and schedules it to be brought in
  • When the disk interrupt indicates that the page has arrived, we update the tables and frames
  • We back up the faulting instruction to where it began, resting the program counter
  • We schedule the faulting process to run again, returning to its calling routine
  • The routine reloads registers and state info, returning to the user space.

On process termination, we release the page table, pages, and any disk space used. If shared pages are involved, we release them after (all) shared processes terminate.

...

One option to deal with this is by pinning pages, which allows pages used in I/O to stop faulting.
