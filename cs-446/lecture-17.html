Lecture 17

So why are we talking about filesystems?

Storage devices have persistent blocks on them that need access, initialization, and organization to be managed. Who's going to do that managing? The operating system.

Persistent memory is usually managed by the file system. The file system allows the user to name and manipulate files, providing users with an organizational structure and file metadata.

If for each block, we need to manage access, initialization, and organization, this means we need to be able to:

  • Map bytes from the disk to a file
  • Share files between users and processes while also managing locking
  • Allow file caching of the metadata and contents of files

For the purposes of this class, we consider a file as a linear, persistent array of bytes. You can read and write to them (those are your two data operations), and creation, deletion, modification, permissions, and users are all related to metadata.

Each directory contains subdirectories. (It used to be the case there was "one" directory everything fell under, usually your desktop). In Unix, directories are treated as files.

Files can be identified in one of the following ways:

  • A file name (what humans use)
  • An inode and device number location (a persistent ID)
  • A file descriptor (what processes see and use)

Now, you can see that we multiple layers for managing I/O to files:

Naturally, this builds complexity that we need to manage. Each layer of the filesystem has its own propose, of which the three we'll talk about are:

  • File naming, which gives the file name and encoding
  • Directory access, which maps the name of a file to a file object and resolves names (strings) to objects
  • File access, which provides available file operations (create, delete, read, write)

Back then, we had typed files, which directly provided a structure to a file. This defined all the possible file types, and the OS would stop you from changing file extensions. Certain machines would let you change file extensions, but it would actually cause the underlying bytes to change. (For example, with a filesystem that uses typed files and supports this, you might be able to switch between a .jpg and a .png such that the underlying bytes change.)

Untyped files don't provide any inherent structure through their file extension; files are treated as a sequential array of bytes. This is the modern approach. The file system doesn't care what's in the file or if it's typed "correctly"; file operations can be performed regardless of file type.

Common file operations include:

  • create (create a new, empty file)
  • open (map a file to a file identifier)
  • close (close a file identifier, usually by name)
  • read (read from the current position in the file, often from the start)
  • write (write to the current position in the file, often from the start)
  • seek (modify the file position)
  • control (allows metadata to be changed, like permissions)
  • createdir (creates a new directory)
  • rmdir (removes a directory)
  • readdir (read directory, returns all files in a directory)

What's the cost of a file operation? Sequentially reading or writing is low cost and common. But the problem with this is that you can get internal fragmentation from this, and it takes a long time to defragment the disk.

Random accesses require seeking, and tend to be more expensive due to the overhead of managing pointers. It's less common.

Many files are very small, while a few are much larger.

Inodes
...

The OS uses inodes and device IDs to manipulate files. IDs are unique, while inodes contain the metadata of a file.

Each file has one inode, which is unique to that file system. Inodes should be recycled after use so that you don't run out of them. Hard links can map multiple file names to the same inode.

You can see that we store data and inodes separately. Files are allocated statically. You must have the inode number to access file contents, with most OSs having an area to store file names to inode maps.

So why do we have filenames at all, if we have to map from filenames to inodes to data? Humans are typically better at remembering strings than memory addresses and huge values.

A special file stores the name to inode mappings. This allows hierarchies to be created; for example, if we have /etc/newfile.txt, /etc points to newfile.txt which points to some actual data.

Special directories (again, in Unix) are mapped by inodes in each directory. . is "this" directory, while .. is one directory up. ls hides this by default without the required flags. This allows us to track the hierarchy of files.

But how do files actually access files? Processes often store file names along with the inode and device ID, but this can lead to constant lookups of filenames to inodes. We can limit this constant lookup by storing the file inode of the file in a per-process table.

The file API
...

The basic API we're going to be using is:

// why return an int? it helps with error checking (besides returning the file descriptor)
int open (char* path, int flag, mode_t mode);
size_t read(int fd, char* buffer, size_t nbyte);
size_t write(int fd, char* buffer, size_t nbyte);
int close(int fd);

You might notice that there's no delete call or anything. This is because we simply mark an inode free if it's no longer referenced (its reference count is zero). We can use unlink() to remove a file from a directory, thus decreasing the reference count.

In the case of file descriptors, they're freed at close or on process exit. It's worth noting that some files are temporary and are deleted automatically when a process ends.

There are a few things we need to think about with respect to sharing files and handling concurrent communication. After a file is opened, we have to deal with:

  • Permissions
  • Moving
  • Deleting
  • Changing owners
  • Process forking
  • IPC (interprocess communication)

What might a temporary file look like?

int main(int argc, char* argv[]){
	int fd = open("test", O_CREAT | O_RDWR, 0600);
	//unlike("test"); // what would happen to fd?
	char *data = "test";
	char rdata[64];

	write(fd, data, strlen(data));
	sleep(10);
	lseek(fd, 0, SEEK_SET);
	read(fd, &rdata, 64);
	rdata[63] = 0;
	printf("We read %s\n", rdata);
	close(fd);
	return 0;
}

Now, it can be the case that you might have one OS and many filesystems. this might be the case if you have several disk partitions (Linux + Windows), multiple disks, or network storages. So how should we organize, manage, and display each filesystem in a standard way?

On Windows, each filesystem is assigned a letter. C is the main hard drive, with new disks named in alphabetic order of addition. All filesystems are mapped to the same root, which is "This PC" (analogous to / on Unix).

On Unix, file systems are mapped into a single tree. Each directory is a mount point, such that mounting a filesystem hides the files in the original directory. Then, home/* might live on a different file system. You can mount anywhere you want, except for some protected directories; if you were to mount your USB drive to home, you could.

Linking
...

Links are file pointers; they don't store any data, but just serve for referencing. You can have soft and hard links:

  • soft: A directory entry, or dentry, points to a file. That file contains the actual filename, which is resolved by the OS when accessed.
  • hard: the directory points to an actual existing file, which increases the number of references of that file.

Hard links are why inodes don't bother with names.

Soft links are slightly safer, as you're not directly overwriting the actual file.

Virtual filesystems
...

Often, you can model an OS's filesystem in a virtual setup. Different filesystems will use different data structures. Unix uses EXT4 and Windows uses NTFS; there's also FAT, which can be used on booth Unix and Windows.

Virtual filesystems abstract the real filesystem so that the filesystem can be treated as an object with operations. Some terms are:

  • Superblock (the mount point) - represents the live filesystem
  • File (open) - a file opened by a process
  • Directory entry(dentry) Cache - speed up path to inode translation
    • Things like frequently used documents, a path to home, etc.
  • Inode - a filesystem object (file or dir)

Most virtual file system operations map to a system call, like open, seek, close, and so on. Whenever you want to implement a new file system, you must use the VFS API, allowing you to use different filesystems without needing to use different code.

This also ensures that system calls are separate from the file system, allowing the kernel to do all the work.

Renaming
...

Renaming only changes the filename. What do you actually do on a rename? You just update the directory entry (dentry); the inode stays the same, since the underlying data hasn't changed. But if the data is on another disk or partition, it may need to be moved.

Filesystem implementation
...

Filesystems manage data given a large set of N blocks.

Their goal is to construct data structures to encode the file hierarchy and metadata while avoiding internal fragmentation and maintaining low overhead. You need to provide and define file API operations, ensuring efficient file access.

That said, there are many different ways you can save a file to a hard drive. When constructing a file system, there are many different possibilities, just like there are many different possibilities for virtual memory design (paging, segmentation, dynamic relocation, etc.). These methods include:

  • contiguous
  • Linked blocks - each block has a pointer to the next
  • File allocation tables (FAT) - a table with a list of block references
  • Indexed - pointers in inodes
  • Multi-Level indexed - a tree of pointers, similar to a multi-level page table

Each option comes with tradeoffs in:

  • Fragmentation
  • File size
  • Performance (in terms of random versus sequential access)
  • Metadata overhead
