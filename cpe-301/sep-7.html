Addresses and decoding
...

Data is stored in memory at an address, where an address is the location in memory. That could be RAM, ROM, etc.

A CPU communicates with memory over buses to get to these addresses. In its simplest form, a bus is a conductive set of wires - though it gets more complicated than that in terms of interfacing with the CPU.

There are three main types of buses:

  • the address bus, which carries the address. This is how the CPU knows where we are going in memory. Often used in read/write operations.
  • the control bus, containing the CPU control signals (as described in CS 219), including read/write enable signals, clock signals. etc.
  • the data bus, which carries arbitrary data to and from the address that is currently defined on the address bus. In other words, it carries data to and from external systems.

At a high level, you have the CPU, RAM, ROM, and some arbitrary set of peripherals (DACs and other exxternal components). Then, the CPU has three buses coming out fo it, which are connected to both RAM and ROM. Not all CPUs have their buses connected to all external components; for example, peripherals often don't control the CPU, and therefore don't need a control bus connection.

Then, we need to talk about bus activity. Opcodes are instructions executed by the CPU. These are loaded into memory through buses. Such instructions are then loaded into the instruction register, or IR.

The program counter, or PC, tells us the location for the next instruction.

So, the arrangement might look something like this:

The PC uses the address bus to retrive from memory for the CPU to execute. That said, the control bus tells the CPU to actually perform the read (when the read state is active) or write, synchronizing with the clock.

The simplified view of a CPU contains the IR after a fetch operation. The PC contains the next (address of) the next instruction, which is then passed into the instruction decoder. It analyzes the instruction, decodes it, and then executes it.

The arithmetic logical unit, or ALU for short, performs mathematical and logical operations.

Then, we can say that our slightly more complex representation of the CPU contains:

  • the IR (instruction register)
  • the PC (program counter)
  • the general purpose registers, acting as internal memory for the entire CPU, including the ALU and other components
  • the instruction decoding unit
  • the ALU

Now, everything has an address as a result of needing to be addressable by the CPU. Every physical device is mapped to a physical address. You can have virtual addresses, which are pieces of physical addresses (though they're beyond the scope of this class).

We call such devices serial devices. The term serial means "one bit at a time" (being read or written), as opposed to parallel. There exists a clock across the buses that ensures that writes don't occur at unexpected times (in general), namely by the serial device.

There are several steps to writing to a serial device:

  • set the control lines on the control bus, setting the internal state of the buses
  • set the address on the address bus
  • load data onto the data bus, whether that means writing to or reading from the serial device.

The issue with this is that if we share the buses across all the internal components, and we have two serial devices, we're going to end up writing to (or reading from) two devices at the same time.

One way to solve this is to make it so that every peripheral has its own address bus, which can be expensive.

We can solve this issue with decoding insterad.
