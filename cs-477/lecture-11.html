Radix sort
...

Radix sort consider keys as numbers in a base-k number. (There are multiple versions of radix sort.) Radix sort operates in linear time.

If keys are numbers, then a -digit number occupies a field of columns. Sorting looks at once column at a time:

  • For a digit number, sort the least significant digit first (LSD radix sort).
  • Continue sorting on the next least significant digit, until all digits have been sorted
  • Requires only passes through the list

Something significant about LSD radix sort is that it allows (but not necessarily enforces) a stable implementation; MSD radix sort does not, because equal-valued keys may fall out of order since those depend on the least significant digit.

Radix sort basically looks like this:

algorithm RADIX-SORT(A, d)
	for i = 1 to d
		stable sort array A on digit i

Bucket sort
...

Bucket sort is another non-comparison sort that assumes that the input is generated by some random process distributing elements randomly over the interval . We take our space over , chop it into equal parts, and then place elements into their final bucket based on their value. If the elements were perfectly distributed, all the buckets would have exactly one element.

But this is unlikely, and we're more likely to have a few keys per bucket.

So what we do is that:

  • divide into equal-sized buckets
  • distribute the input values into the buckets
  • sort each bucket (using any sort)
  • go through the buckets in order, reassembling the full array
algorithm BUCKET-SORT(A, n)
	// allocate space for the sublists in B
	for i = 1 to n
		insert A[i] into list B[floor(n*A[i])]
	for i = 0 to n - 1
		sort list B[i] with insertion sort
	concatenate lists B[0] ... B[n-1] together
	return concatenated list

What's happening here? B is an array of buckets; the calculation floor(n*A[i]) determines what bucket the current element should go into.

For example, suppose we have ten inputs evenly distributed over . Then, when we look at the element with a value of 0.78, we multiply it by , giving us 7.8; taking its floor tells us that we should put 0.78 into bucket 7.

We then do this for every element; where buckets contain more than one element, we can apply nearly any sort (because of how small the buckets should be) such that it wouldn't matter overall. After this, we concatenate all the elements together to form the sorted output. Notice that this does require auxillary arrays to complete.

Priority queue implementations and cost

We're now going to move towards analyzing different structures and how they solve certain problems, and then how we can augment these structures so they're much better for specific problems. The fundamental graphs, trees, heaps, etc. can all be modified in some way to solve certain problems more efficiently.

How do we implement priority queues efficiently? The worst-case asymptotic costs for a priority queue with items is
| | Insert item | Remove max item |
| --------------- | ----------- | --------------- |
| Ordered array | N | 1 |
| Ordered list | N | 1 |
| Unordered array | 1 | N |
| Unordered list | 1 | N |

Can we implement both of these efficiently? We can, using heaps, which are tree-like structures.

Background on heaps
...

Let's revisit binary trees. Binary trees are structures composed of a finite set of nodes that either contain no nodes or are compoesd of three disjoint sets of nodes: a root node, a left subtree, and a right subtree.

A heap is a nearly complete binary tree with the following two properties:

  • structural property: all the levels are full, except possibly the last, which is filled from left to right
  • order property: for any node x, the value of the parent of x is greater than or equal to x itself

Some definitions:

  • the height of a node is the number of edges on a longest simple path from the node down to a leaf. (That is, the height of a leaf is 0; a leaf's parent has a height of 1; and so on.)
  • the depth of a node is the length of a path from the root to the node. The depth of the root is 0.
  • The height of the entire tree is the height of the root node, and because heaps are (approximately) balanced binary tree, the tree's height should be for a heap of elements.

How do we store heaps? Visually, heaps are trees; but remember that we actually store them in arrays, because we can perform index calculations that allow us to readily perform heap operations and find children/parents.

It also holds that the elements in the subarray are leaves, representing all the elements in the last level. This is significant in that anything going past the halfway point must be a leaf; multiplying by 2 past the halfway point will bring you past the bounds of the array.

We just looked at maxheaps, where the largest element is at the heap and all parent nodes are greater than or equal to the value of their child nodes. There are also minheaps, where parents are less than or equal to their children instead.

There are a few operations on heaps:

  • max-heapify, which maintains the maxheap propert(ies)
  • build-max-heap, which creates a maxheap from an unsorted array
  • resolve

Max-priority queues support the following operations:

  • insertion
  • extraction
  • getting the maximum value
  • increasing the value of a key (usually the highest key)

Maintaining the heap property
...

Let us demonstrate how the heap property is maintained.

Suppose that a node is smaller than a child. Then, the left and right subtrees of that node at are maxheaps. Then, the invariant is that the heap condition is violated only at that node. We can fix this by taking the maximum of the two children, and then swapping with the root. We then check if this violates the child node being a maxheap, and continue until we're done.

We must ensure that the following assumptions hold while maintaining the heap property:

  • the left and right subtrees of i are maxheaps
  • A[i] may be smaller its children
algorithm MAX-HEAPIFY(A, i, n)
	l = LEFT(i)
	r = RIGHT(i)
	if l <= n and A[l] > A[i]
		largest = l
	else
		largest = i
	if r <= n and A[r] > A[largest]
		largest = r
	if largest != i
		exchange A[i] with A[largest]
		MAX-HEAPIFY(A, largest, n)
