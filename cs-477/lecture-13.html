Lecture 13

Remember that binary search trees have pretty efficient fundamental operations. But the problem is that these trees can get really unbalanced, in which case the operations stop being efficient. So what if we have a self-balancing tree?

Red-black trees is one such binary tree, guaranteeing running time on its basic dynamic-set operations.

The non-modifying BST operations MAXIMUM, MINIMUM, SUCCESSOR, PREDECESSOR, and SEARCH run in time, taking (since ) in an RBT.

But what about TREE-INSERT and TREE-DELETE? It turns out that they will still run in , though we also have to guarantee that the modified tree is also a red-black tree.

Insertion
...

There are a few problems we face when adding new elements:

  • When we color things red, we can't have two red nodes in a row.
  • When we color things black, we might break the property that says all paths from a node to its leaves contain the same number of black nodes.

In general, it's easier to insert red nodes; we don't have to immediately fix the black heights everywhere, and we just have to start recoloring consecutive red nodes if it turns out to be bad.

Deletion
...

What happens if the node is red when we delete it? Let's analyze the properties of an RBT:

  • Every node is still red or black.
  • The root is still black, since we haven't deleted the root since it cannot be red
  • Every leaf is still black.
  • If a node is red, both its children are black - this is still true, since the result is that we have several black nodes in a row, which is still fine.
  • The black-heights have still not changed, since the number of black nodes remains the same.

But what if it was black?

  • Every node is still red or black.
  • The root might be black after the deletion, but what if the child that replaces it is red?
  • Every leaf (NIL) is still black.
  • If a node is red, both its children are black - this might be true, but removing black nodes could lead to consecutive reds.
  • Necessarily, the black-heights of some nodes could change.

So on black deletion, we might violate three properties.

Rotations
...

Rotations are operations for restructing the tree after an insert or delete operation. They take a node within the tree and then move the pointers around.

In a left rotation, we assume that the right child of a node x, which is y, is not NIL. The idea is that we want to pivot around the link from x to y - the properties of a BST and RBT, notably, are still held after the rotation.

The idea is as follows:

  • Make y the new root of the subtree
  • x becomes y's left child
  • y's left child becomes x's right child

Here's an example of a left-rotate:

The general process is as follows:

  • Get y, the right child of x that is to be made the new root
  • Turn y's left subtree to become x's right subtree
  • If the left subtree of y is NOT nil,
    • then let x become the parent of y's left subtree
    • otherwise, do nothing (you may not "attach" NILs directly to real nodes) - this causes y's left subtree (NIL) to just be hanging, as intended
  • Let the parent of x be the parent of y
  • If the parent of x is NIL (x was the root),
    • reassign the root of the whole RBT to be y
    • otherwise, if x is the left child of its parent
      • then let the left child of the parent of x be y
      • otherwise, let the right child of the parent of x be y
        • the two conditions above amount to "replace x with y relative to its parent"

Right rotations are simply mirrored versions of a left rotate, except that the assumption is now that the left child of y, which is x, may not be NIL. Again, the properties still hold.

Insertion, in-depth
...

So our goal is to insert a new node z into an RBT. To do so:

  • Insert z as for an ordinary BST
  • Color the node red (which may lead to two red nodes in a row)
  • Fix violated properties

Here, we will keep a "trailing" pointer in addition to our "seeking" pointer. The "seeking" pointer determines which branch z should go down to; the "trailing" pointer keeps track of the parent of the current node we're inspecting. The trailing pointer is necessary because when the "seeking" pointer hits a leaf, we know that we need to insert z there, but we still need the trailing pointer to know what to attach z to.

Then:

  • if the tree was previously empty, make this the root
  • else, let z be the left or right child
  • color z red
  • fix the tree

Again, our concerns lie in properties 2 and 4 - the root might not be black (in which case we recolor it, if it was the first node inserted). The real concern, then, lies in property 4: what if we have two red nodes in a row?

So how do we fix up our tree? (Call such an algorithm RB-INSERT-FIXUP.) There are three cases.

We will actually take the third option. But the problem, then, becomes that we have changed the black-heights. So we can take it back to the original black-height by recoloring C to be red. The problem, then, is that C's parent might be red, in which case we are creating consecutive red nodes somewhere else. (If it's black, we're done.)

That's fine - we have removed the problem locally, but we can continue fixing any successive issues until we're done. In essence, we keep pushing the red node up until we hit the root, which is guaranteed to be black.

Augmenting data structures
...

It is unusual to have to design all-new data structures from scratch. Usually, the idea is to store additional information in existing data structures, such that they can support new operations. One problem we might have is finding the th order statistic.

We begin with the order-statistic tree, a red-black tree with additional information stored in each node. It still has the usual fields of an RBT: key, color, parent, left, and right. But what additional information might we want to find the th order statistic?

Our additional field will be "size", representing the number of internal nodes (including itself) in the subtree rooted at x
