Counting sort
...

When we know certain things about the data, we can possibly beat . How is it possible to put things in order without comparing?

Consider the fact that when you are given the array [0, 5, 6, 1, 2, 4, 3], you know that 5 should belong in index 5, 6 belongs in index 6, and so on. This makes "sorting" in this manner a linear-time operation.

So let's suppose that the elements to be sorted are integers in the range 0 to . What does this tell us? It means that for every possible key, we can keep a spot in an auxillary array and count the occurrences for each key. For example:

[2 5 3 0 2 3 0 3]

If we know that there are two 0s, then we know for a fact that since 0 is the smallest possible value, index 0-1 shall be composed of 0s. If we know there are two 2s and are the next-largest array, index 2-3 shall be composed of 1s.

So in our auxillary array, we count the number of elements less than or equal to the index itself. For the example above, we would have [2 2 4 7 7 8].

Let us implement this:

// A = input array
// B = output array
// n = num elements
// k = largest-valued key
algorithm COUNTING-SORT(A, B, n, k)
	// populate the external array C
	for i = 0 to k
		do C[i] = 0

	// have C[i] contain the number of elements equal to i
	for j = 1 to n
		do C[A[j]] = C[A[j]] + 1

	// adjust the counts to eliminate duplicate keys
	// C[i] then contains the number of elements less than or equal to i
	for i = 1 to k
		do C[i] = C[i] + C[i-1]
	
	// NOTE: why do we go in reverse?
	for j = n downto 1
		do B[C[A[j]]] = A[j]
			C[A[j]] = C[A[j]] - 1

Let's actually apply a full example. Note that A and B are 1-indexed in the slides, but C is 0-indexed.

  • A = [2 5 3 0 2 3 0 3]
  • After the first running pass, C = [2 0 2 3 0 1] representing the number of times each "index" appears as a value in A
  • After the second running pass, we take the running sum so that C = [2 2 4 7 7 8]

Then, we know that the next time we see a 3, it belongs at index 7. We now decrement C[3] by 1, so that the next time we see a 3, it belongs at index 6. You can see how this addresses the issue of duplicates.

At this point, we simply apply the algorithm and place elements into the output array B based on what we see in C as we walk through A.

The end result is a running time of . You can see that if is sufficiently large, then we might end up with very bad running times; but otherwise, it's very close to linear.
